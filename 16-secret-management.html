Why Secret Management Matters in Banking

In a bank, a leaked secret is not a technical incident — it is a regulatory breach. APRA CPS 234 mandates that Westpac maintain information security controls commensurate with the threats it faces. Hardcoded or poorly managed credentials violate this directly.

What counts as a secret in banking:
Database passwords, API keys, TLS private keys, OAuth client secrets, JWT signing keys, SWIFT credentials, payment gateway tokens, encryption keys (DEK/KEK), service account passwords, SSH private keys, SAML certificates.
APRA CPS 234
Requires information asset classification, access controls, and incident notification within 72 hours of a material breach
PCI-DSS
Payment card data environments must rotate credentials, enforce least privilege, and maintain audit logs for all secret access
Cost of a Breach
Average financial breach costs $5.9M AUD. Beyond fines, APRA can impose licence conditions or require independent audits
Audit Trail Requirement
Every secret access must be logged: who, what, when. This eliminates plaintext env vars — you can't audit those
The fundamental rule: A secret that lives in code, a Dockerfile ENV, or an unencrypted ConfigMap is not a secret — it is a liability. Westpac interviewers will probe whether you understand this distinction.

AWS Secrets Manager Deep Dive

AWS Secrets Manager is a fully managed service for storing, rotating, and retrieving secrets. Unlike Parameter Store, it is purpose-built for secrets with native rotation support.

Storage
Secrets stored as JSON key-value pairs. Encrypted with KMS (default aws/secretsmanager or customer-managed CMK)
Versioning
AWSCURRENT (active), AWSPENDING (during rotation), AWSPREVIOUS (last version). Apps can pin to a version label
Rotation
Lambda-based. AWS provides templates for RDS, Redshift, DocumentDB. Custom Lambda for everything else
Pricing
$0.40/secret/month + $0.05 per 10,000 API calls. Cross-account replication costs extra. Plan for this at scale

Automatic Rotation — How it Works

sequenceDiagram participant SM as Secrets Manager participant L as Rotation Lambda participant DB as RDS Database participant App as Application SM->>L: createSecret (generate new password) L->>SM: putSecretValue (AWSPENDING stage) SM->>L: setSecret (apply to DB) L->>DB: ALTER USER SET PASSWORD = new_pass SM->>L: testSecret (verify new works) L->>DB: connect with new_pass L-->>SM: success SM->>SM: finishSecret (AWSPENDING → AWSCURRENT) SM->>SM: old AWSCURRENT → AWSPREVIOUS App->>SM: GetSecretValue (gets new AWSCURRENT) 
# Retrieve a secret in Python (application code)
import boto3
import json

def get_db_credentials(secret_name: str, region: str = "ap-southeast-2"):
    client = boto3.client("secretsmanager", region_name=region)
    response = client.get_secret_value(SecretId=secret_name)
    secret = json.loads(response["SecretString"])
    return secret["username"], secret["password"]

# Usage
username, password = get_db_credentials("prod/confirmation/db")
# Cross-account secret access — resource policy on the secret
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::WORKLOAD-ACCOUNT-ID:role/app-role"
    },
    "Action": ["secretsmanager:GetSecretValue", "secretsmanager:DescribeSecret"],
    "Resource": "*",
    "Condition": {
      "StringEquals": {
        "aws:PrincipalOrgID": "o-xxxxxxxxxx"
      }
    }
  }]
}

SSM Parameter Store

Parameter Store is often confused with Secrets Manager. They serve different purposes. Parameter Store is AWS Systems Manager's configuration store — it handles both config and secrets, but with fewer secret-specific features.

FeatureSecrets ManagerParameter Store
Primary useSecrets (passwords, keys)Config + lightweight secrets
Cost$0.40/secret/monthFree (standard), $0.05/advanced/month
Automatic rotationYes (Lambda-based)No (manual or custom)
VersioningYes (AWSCURRENT/PENDING/PREVIOUS)Yes (version numbers)
EncryptionKMS (mandatory)KMS (SecureString only)
Max size65KB4KB (standard), 8KB (advanced)
Cross-accountYes (resource policy)Limited
Audit via CloudTrailYesYes
When to use Parameter Store: Non-sensitive config values (feature flags, URLs, environment names), or when cost is a concern and you have hundreds of parameters. Use Secrets Manager when you need automatic rotation or cross-account sharing. 
# Parameter Store hierarchy for banking
/westpac/prod/confirmation/
  db/host              → String (not sensitive)
  db/port              → String
  db/credentials       → SecureString (encrypted with KMS)
  rabbitmq/url         → SecureString
  jwt/public-key       → SecureString

# Fetch with AWS CLI
aws ssm get-parameter \
  --name "/westpac/prod/confirmation/db/credentials" \
  --with-decryption \
  --region ap-southeast-2

HashiCorp Vault

Vault is the enterprise standard for secrets management in complex, multi-cloud, or on-prem environments. Westpac may use Vault in hybrid scenarios where secrets span on-prem and AWS.

Vault HA Architecture

graph TB subgraph vault_cluster["Vault Cluster (Raft/Consul)"] V1["Vault Active\n(Leader)"] V2["Vault Standby"] V3["Vault Standby"] V1 <-->|Raft Consensus| V2 V1 <-->|Raft Consensus| V3 end subgraph backends["Secret Backends"] KV["KV v2\n(Static Secrets)"] PKI["PKI Engine\n(Cert Authority)"] DB["Database Engine\n(Dynamic Creds)"] AWS2["AWS Engine\n(IAM Dynamic Creds)"] end V1 --> KV V1 --> PKI V1 --> DB V1 --> AWS2 App["K8s Pod"] -->|Auth: JWT/SA Token| V1 V1 -->|Token + Secret| App
Auth Methods
Kubernetes (ServiceAccount JWT), AWS IAM, LDAP, AppRole, GitHub. Pods use K8s auth — no static credentials needed
KV v2 Engine
Key-value store with versioning and soft-delete. Each write creates a new version. Can restore to any previous version
Dynamic Secrets
Vault generates short-lived DB credentials on-demand. Leases expire automatically. No long-lived passwords in existence
PKI Engine
Vault as an internal CA. Issues short-lived TLS certs (24h). No cert sprawl. Used by Istio for mTLS in some setups

Dynamic Database Credentials — The Gold Standard

# Vault policy for a microservice
path "secret/data/confirmation/*" {
  capabilities = ["read"]
}

path "database/creds/confirmation-readonly" {
  capabilities = ["read"]
}

# The app gets credentials that expire in 1 hour
# Vault auto-revokes them when the lease expires
# No humans ever know the actual password
# Vault Agent Injector — inject secrets into pod without app changes
# Annotations on the pod:
metadata:
  annotations:
    vault.hashicorp.com/agent-inject: "true"
    vault.hashicorp.com/role: "confirmation-app"
    vault.hashicorp.com/agent-inject-secret-db: "database/creds/confirmation-readonly"
    vault.hashicorp.com/agent-inject-template-db: |
      {{- with secret "database/creds/confirmation-readonly" -}}
      postgresql://{{ .Data.username }}:{{ .Data.password }}@db:5432/confirmation
      {{- end -}}
# File appears at /vault/secrets/db inside the container

External Secrets Operator (ESO)

ESO is the Kubernetes-native way to sync secrets from AWS Secrets Manager, Parameter Store, or Vault into Kubernetes Secrets automatically. This is the preferred pattern for EKS workloads.

sequenceDiagram participant ESO as ESO Controller participant SM as AWS Secrets Manager participant K8s as Kubernetes API participant Pod as Application Pod ESO->>SM: GetSecretValue (using IRSA) SM-->>ESO: secret value ESO->>K8s: Create/Update Secret K8s-->>Pod: Mount secret as env/volume Note over ESO: Refresh every refreshInterval (e.g. 1h) ESO->>SM: GetSecretValue (check for rotation) SM-->>ESO: new rotated value ESO->>K8s: Update Secret 
# ClusterSecretStore — one per cluster, references AWS
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: aws-secrets-manager
spec:
  provider:
    aws:
      service: SecretsManager
      region: ap-southeast-2
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
            namespace: external-secrets

---
# ExternalSecret — sync one secret from AWS SM to K8s
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: confirmation
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: db-credentials          # name of K8s Secret created
    creationPolicy: Owner
  data:
    - secretKey: DB_PASSWORD       # key in K8s Secret
      remoteRef:
        key: prod/confirmation/db  # name in Secrets Manager
        property: password         # JSON field
Key insight for interviews: ESO keeps the K8s Secret in sync automatically. If Secrets Manager rotates the password, ESO picks it up within refreshInterval and updates the K8s Secret. The pod gets the new value on next restart, or if you mount via volume with optional: true, it updates live.

Kubernetes Native Secrets — What to Know

Critical misconception to avoid: Kubernetes Secrets are NOT encrypted by default. They are base64-encoded, which is trivially reversible. echo "cGFzc3dvcmQ=" | base64 -d = password. Saying "secrets are encrypted in Kubernetes" in an interview at a bank will damage your credibility.

What you should say instead: Kubernetes Secrets are stored in etcd. Encryption at rest requires explicitly enabling EncryptionConfiguration on the API server. In EKS, AWS enables etcd encryption automatically with a KMS key — but self-managed clusters need this configured manually.

Sealed Secrets (Bitnami)
Encrypt secrets with a cluster public key so they're safe in Git. Controller decrypts with private key. One-way: you can't extract the value back
SOPS (Mozilla)
Encrypt YAML/JSON files with KMS, PGP, or age. Works with any git workflow. Age-encrypted secrets are portable across clouds
Avoid in Git
Never commit plain Kubernetes Secret manifests to Git. Use Sealed Secrets, SOPS, or ESO to pull from a secrets store
RBAC on Secrets
Restrict get on secrets via RBAC. Even within a namespace, not every service account should read all secrets

Secret Rotation Patterns

Rotation is where theory meets production reality. The naive approach (change password, update app) causes downtime. Banking systems require zero-downtime rotation.

Dual-Active Pattern (Zero Downtime)

sequenceDiagram participant DB as Database participant SM as Secrets Manager participant App as Running App (old creds) Note over DB,App: Phase 1 — Create new credentials SM->>DB: CREATE USER app_v2 WITH PASSWORD 'new' SM->>SM: Store new creds as AWSPENDING Note over DB,App: Phase 2 — Both old + new work simultaneously App->>DB: Connect with old creds (still works) SM->>SM: Promote AWSPENDING → AWSCURRENT Note over DB,App: Phase 3 — App picks up new creds App->>SM: GetSecretValue (gets new AWSCURRENT) App->>DB: Connect with new creds Note over DB,App: Phase 4 — Remove old credentials SM->>DB: DROP USER app_v1
The key insight: There is a window where BOTH old and new credentials are valid. This eliminates downtime. The app rotates at its own pace (next request/restart), not in sync with the DB change.

IRSA — IAM Roles for Service Accounts

IRSA is how EKS pods get AWS permissions without any static credentials. This is the gold standard for banking workloads — no AWS access keys stored anywhere.

sequenceDiagram participant Pod as K8s Pod participant SA as Service Account participant OIDC as EKS OIDC Provider participant STS as AWS STS participant SM as Secrets Manager Note over Pod,SA: Pod starts, mounts SA JWT token Pod->>STS: AssumeRoleWithWebIdentity(JWT token, RoleARN) STS->>OIDC: Validate JWT signature OIDC-->>STS: Valid (issued by this cluster) STS-->>Pod: Temporary credentials (15min-12h TTL) Pod->>SM: GetSecretValue (using temp creds) SM-->>Pod: Secret value Note over Pod: SDK auto-refreshes before expiry 
# Step 1 — Create IAM role with trust policy for the service account
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::ACCOUNT:oidc-provider/oidc.eks.ap-southeast-2.amazonaws.com/id/XXXX"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "oidc.eks.ap-southeast-2.amazonaws.com/id/XXXX:sub":
          "system:serviceaccount:confirmation:confirmation-app",
        "oidc.eks.ap-southeast-2.amazonaws.com/id/XXXX:aud": "sts.amazonaws.com"
      }
    }
  }]
}

# Step 2 — Annotate the Kubernetes ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: confirmation-app
  namespace: confirmation
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::ACCOUNT:role/confirmation-app-role

# Step 3 — Pod uses this ServiceAccount (no credentials anywhere)
spec:
  serviceAccountName: confirmation-app
  containers:
    - name: app
      # AWS SDK automatically uses IRSA — no env vars needed
Interview answer: "We use IRSA for all EKS workloads. The pod's ServiceAccount JWT is exchanged with AWS STS via the cluster's OIDC provider for temporary credentials. Zero static secrets. Each service account maps to a least-privilege IAM role. If a pod is compromised, the credentials expire in under an hour and can be revoked by deleting the IAM role trust."

Audit & Compliance

For APRA CPS 234, every secret access must be auditable. AWS CloudTrail captures all Secrets Manager API calls automatically.

# CloudTrail query — who accessed prod DB password in the last 7 days?
# Run in CloudTrail Lake or Athena
SELECT
  eventTime, userIdentity.arn, sourceIPAddress,
  requestParameters.secretId, responseElements.name
FROM cloudtrail_logs
WHERE eventName = 'GetSecretValue'
  AND requestParameters.secretId LIKE '%prod/confirmation/db%'
  AND eventTime > NOW() - INTERVAL '7' DAY
ORDER BY eventTime DESC;
Secret Scanning in CI/CD
Run git-secrets or truffleHog in GitHub Actions on every PR. Block merge if AWS keys, passwords, or private keys detected in code
APRA Incident Notification
If a secret is leaked, APRA CPS 234 requires notification within 72 hours of identifying it as a material incident
Secret Sprawl Detection
Regularly audit: grep for hardcoded keys in repos, scan ECR images, check ConfigMaps for base64 secrets, review K8s secrets without ESO ownership
Vault Audit Log
Vault writes every request/response to an audit log. Ship to Splunk or Datadog. Alert on unexpected secret access patterns

Interview Scenarios

Q: We have 200 microservices with hardcoded DB passwords in environment variables. How do you migrate them to a secrets manager without downtime?

This is a phased migration, not a big bang. I'd break it into four phases:

Phase 1 — Inventory: Run a script to grep all repos and Helm values for patterns like password=, DB_PASS, base64-encoded strings. Classify by sensitivity and service owner.

Phase 2 — Infrastructure: Deploy External Secrets Operator on the cluster. Create a ClusterSecretStore pointing at AWS Secrets Manager with IRSA. Now the plumbing exists.

Phase 3 — Service-by-service migration: For each service: (1) Store the current password in Secrets Manager, (2) Create an ExternalSecret that syncs it to a K8s Secret with the same env var name the app already uses, (3) Remove the hardcoded env var from the Helm chart, (4) Deploy — the app gets the same value, zero behaviour change. (5) Enable rotation in Secrets Manager.

Phase 4 — Rotate: Now that all services read from Secrets Manager, trigger rotation. The dual-active pattern ensures no downtime.

Timeline: ~2-3 months for 200 services with a dedicated team. Prioritise payment-critical services first for APRA compliance.

Q: A developer accidentally committed an AWS access key to a public GitHub repo. Walk me through your incident response.

First 5 minutes — contain: Immediately revoke the key in IAM console. Don't wait to investigate first — revoke now. Then check if GitGuardian or GitHub Secret Scanning already alerted (it likely did within seconds of the push).

Next 30 minutes — assess blast radius: Pull CloudTrail logs for that access key ID. Check every API call made since the commit time. Look for: new IAM users created, S3 data exfiltrated, EC2 instances launched, cross-region activity. Run aws iam get-access-key-last-used — was it used externally?

Remediation: Purge the key from git history using git filter-repo. Force push (coordinate with team). Create a new key, store in Secrets Manager, rotate all systems using it.

APRA notification: If customer data was accessed, notify APRA within 72 hours. Document timeline, root cause (no pre-commit hook blocking secrets), and preventive controls added.

Prevention: Add git-secrets as a pre-commit hook. Add truffleHog to CI/CD pipeline. Enable GitHub Advanced Security secret scanning with push protection.

Q: What is the difference between IRSA and using an EC2 instance profile for EKS pods?

An EC2 instance profile gives AWS permissions to the entire EC2 node. Every pod on that node — regardless of what service it is — inherits those permissions. If one pod is compromised, it can use the node's role to access anything the node can.

IRSA gives permissions at the pod level via a ServiceAccount annotation. Each microservice gets its own IAM role with only the permissions it needs. If the confirmation-app pod is compromised, it can only access confirmation secrets — not payment secrets, not other services' databases.

For a bank, this isn't optional. APRA's least-privilege principle means you cannot share an instance profile across services. IRSA is the correct architecture. We use it for every EKS workload at Thomson Reuters and I'd implement the same at Westpac.

Q: Secrets Manager or Vault — which would you recommend for Westpac?

It depends on the deployment model. If Westpac is fully AWS-native with EKS, AWS Secrets Manager + External Secrets Operator is simpler, fully managed, and integrates natively with IAM and CloudTrail — less operational overhead.

If Westpac has on-premises systems or a multi-cloud footprint (AWS + Azure + on-prem), HashiCorp Vault is the better choice because it is cloud-agnostic. Vault's dynamic secrets (short-lived DB credentials) are also more powerful than Secrets Manager's rotation.

In practice, large banks often run both: Vault for on-prem and hybrid, Secrets Manager for AWS-native workloads, with ESO bridging K8s to both. I'd assess Westpac's hybrid footprint first, then recommend.

Q: How do you handle secret rotation for a database used by 50 microservices?

The naive approach — one shared DB user for all 50 services — is an anti-pattern. Each service should have its own DB user with only the tables it needs. That limits blast radius if one service is compromised.

For rotation: use the dual-active pattern. Secrets Manager creates a new password, updates the DB user, promotes it to AWSCURRENT while keeping the old AWSPREVIOUS alive for a short window. ESO on the cluster picks up the new value within the refreshInterval (e.g., 1 hour). Pods get new credentials on their next restart or via a rolling update triggered by the secret change.

If all 50 services must restart simultaneously, stagger the rollout — use ArgoCD sync waves or Helm hooks to rolling-restart in batches. The dual-active window (both passwords valid) gives you enough time.
