Why This Companion Senior

Section 08 covers PostgreSQL HA topology — Patroni clusters, pgBackRest backups, PGO operator, primary/replica services, the 24h/6h/2min RPO ladder. That's the operational view: what the system looks like from the outside.

This companion (8b) goes inside a single PostgreSQL instance and explains the mechanics that show up in interview questions about "why is the database slow?", "what does VACUUM actually do?", "what's a checkpoint?". Banking interviewers (Cognizant, Westpac) drill these because losing one transaction in audit/payments is a regulatory event — they want engineers who understand the durability boundary, not just the cluster boundary.

Q: "What's the difference between PostgreSQL HA and PostgreSQL internals from your platform's perspective?"
"HA is the cluster topology: two Patroni nodes (primary + replica) with a repo-host for backups, fronted by a -primary service. Patroni handles leader election via the DCS, pgBackRest handles full/diff backups plus continuous WAL archive, and PGO is the Kubernetes operator that wires it all together. RPO is bounded by archive_timeout=120s for WAL-archive recovery, sub-second for streaming-replica failover.

Internals is what happens inside one of those nodes: a query arrives, postmaster forks a backend, the backend reads pages from shared_buffers (with OS page cache as a second layer), modifies tuples in MVCC fashion, writes WAL on commit and waits for fsync before returning to the client, then later checkpoint flushes dirty pages to data files in spread I/O. When the platform has a slow query or a connection storm or bloat, that's where you debug — not at the Patroni layer."

The 4 Containers

PostgreSQL 16 + Patroni in one image. Holds postmaster, checkpointer, bgwriter, walwriter, autovacuum launcher, archiver, walsender, plus one backend process per client connection. Patroni runs alongside as the HA controller.

Watches the pgBackRest ConfigMap and reloads the sidecar when config changes. Negligible memory.

Why pgBackRest Is a Sidecar, Not a CronJob

WAL archival is event-driven, not scheduled. Every time PostgreSQL fills a 16 MB WAL segment (or hits archive_timeout=120s with partial WAL), it invokes archive_command. That command pushes the segment to the pgBackRest sidecar over a UNIX socket / TLS — instantly, in the same pod. A CronJob would have to start a fresh container, mount the data PVC, run the archive, and exit — multiple times per minute on a busy primary. Sidecar pattern keeps the connection warm and the latency bounded.
Common confusion: The full and diff backups are CronJobs (created by PGO, scheduled via Helm). Those CronJobs talk to the sidecar to read data files. The sidecar handles both jobs: continuous WAL archival (push from PG) and on-demand backups (pull by CronJob).
Q: "Why does kubectl top show the pgBackRest sidecar using 30 GB of memory when its actual process RSS is ~8 MB?"
"That's OS page cache, not application memory. kubectl top shows working set memory, which on Linux includes file-backed page cache that the kernel attributes to the cgroup. When pgBackRest reads the PostgreSQL data files for backups, the kernel caches those pages — and because the sidecar is in the same pod (same cgroup) as the database, the cache shows up under either container depending on which process touched the page first.

The actual pgBackRest server process is tiny (~8 MB RSS). The 30 GB is reclaimable — if another process needs RAM, the kernel drops these pages. So memory.working_set_bytes looks scary, but memory.usage_bytes - memory.cache is what matters for OOM risk. We learnt this when an alert fired on the sidecar at 31 GB and the pod was perfectly healthy."
8b.3

Shared Buffers, Clock-Sweep, work_mem Trap Critical

PostgreSQL has two memory regions you must understand: shared_buffers (one big pool allocated at startup, shared by all backends) and work_mem (per-operation, per-backend, allocated on demand). Mixing them up is the most common interview blunder — and it's also how production OOMs happen.

Two-Layer Cache Architecture

LayerOwnerAlgorithmSize hint
shared_buffers PostgreSQL (allocated once at startup) Clock-sweep with usage_count 0-5 ~25% of RAM (e.g. 16 GB on a 64 GB pod)
OS page cache Linux kernel (uses any free RAM) LRU with active/inactive lists Whatever's left after PG + work_mem + other processes

Every page read by PostgreSQL is checked in shared_buffers first (~0.001 ms hit). Miss falls through to the OS page cache (~0.01 ms hit). Miss again hits the actual disk (~0.1-1 ms on SSD). PostgreSQL's docs explicitly say "a reasonable starting value for shared_buffers is 25% of the memory in your system" — beyond that, you starve the OS cache, which is actually a smarter LRU than PG's clock-sweep.

Clock-Sweep Eviction

How a Page Gets Evicted 
shared_buffers is a circular array. There's a "clock hand" pointer.

  [Page A · usage=3] [Page B · usage=0] [Page C · usage=1] [Page D · usage=5]
                       ^ hand here

When a backend needs to load a new page and the buffer is full:
  1. Hand moves clockwise.
  2. At each page, decrement usage_count by 1.
  3. If usage_count reaches 0 → EVICT this page, write back if dirty,
     load the new page in its slot, set usage_count=1.
  4. If usage_count > 0 → skip, move on.

Pages touched often (hot working set) keep their counts high → survive.
Pages touched once (one-off scans) drop to 0 quickly → evicted fast.
effective_cache_size is a hint, not an allocation. It tells the planner how much OS page cache it can assume exists, so it can pick index scans over seq scans on data that's likely cached. Setting it to 0.5 * RAM doesn't reserve memory — it just changes plan choices. People conflate this with shared_buffers all the time.

The work_mem Multiplication Trap

work_mem is the limit for each sort, hash, or merge operation, in each backend, separately. It's not a per-query budget. It's not a global limit. A single query with 4 join nodes plus a sort runs 5 separate work_mem allocations. Multiply by concurrent connections and you get terrifying numbers.

Real Math From Our QA Cluster 
work_mem = 256 MB

Query plan: Hash Join → Hash Join → Sort  (3 operators that allocate work_mem)
  One query                          → 3 × 256 MB = 768 MB

Concurrent connections running similar plans:
  5 connections × 768 MB             = 3.8 GB
  146 peak connections × 3 ops × 256 = 112 GB ← exceeds total RAM

Reduced to work_mem = 128 MB:
  146 × 3 × 128 MB                   = 56 GB (still alarming, halved)
Spill behaviour: When an operator exceeds work_mem, it spills to $PGDATA/base/pgsql_tmp/ as temporary files. The query still completes, but ~10-100x slower because it's now disk-bound. Watch for log_temp_files = 0 in postgresql.conf — anything logged there is a sort or hash that didn't fit. If you see lots of spill, either lower work_mem (cheaper queries, fewer OOM risk) or raise it for specific roles only (ALTER ROLE reporting SET work_mem = '512MB').
Q: "How would you size shared_buffers and work_mem for a 64 GB Postgres pod that handles ~150 connections?"
"Starting baseline:
  • shared_buffers = 16 GB (25% of pod RAM). Holds the hot working set. Going higher than ~40% of RAM hurts because OS page cache is a smarter LRU and we'd be starving it.
  • effective_cache_size = 32 GB (~50% of RAM). Hint to the planner that ~half of RAM is available as OS page cache. Doesn't allocate anything.
  • work_mem = 32 MB as the safe default. A bad query with 4 hash/sort operators × 150 connections × 32 MB = 19 GB worst case — still survivable.
  • maintenance_work_mem = 1 GB for VACUUM, CREATE INDEX, etc. Only 3 autovacuum workers run by default, so worst case is 3 GB.
Then I'd profile: turn on log_temp_files = 0, watch for spill, and bump work_mem selectively per role for known reporting workloads. The mistake we hit on QA was setting 256 MB globally — looked fine in dev (5 connections), exploded under load (140+ connections × multi-operator plans)."
Q: "What's the difference between shared_buffers and effective_cache_size?"
"shared_buffers is real memory PostgreSQL allocates at startup for its own page cache. It's a fixed allocation — the OS sees the full block as used by the postmaster. Backends read and write into it; the clock-sweep evicts pages from it.

effective_cache_size is not memory at all. It's a number you tell the planner so it can guess how much of your data is likely cached either in shared_buffers or the OS page cache. The planner uses it to decide between an index scan (cheap if cached) and a seq scan (cheap if you already need lots of pages). Setting it too low makes the planner pessimistic and pick seq scans; too high makes it optimistic and pick indexes that hit cold disk. Common rule: set it to ~50-75% of total RAM."
8b.4

MVCC, VACUUM, Autovacuum Circular Deadlock Critical

PostgreSQL never deletes rows in place. Every UPDATE creates a new tuple version; every DELETE marks the old one dead. Dead tuples accumulate until VACUUM reclaims them. This is MVCC — and understanding the cleanup loop is what separates a junior who knows VACUUM exists from a senior who can debug why the database is slowly suffocating.

Tuple Visibility — xmin / xmax

Every row in a heap page has a header containing two transaction IDs:

FieldMeaningSet when
xminThe XID of the transaction that created this versionINSERT or UPDATE
xmaxThe XID of the transaction that deleted/superseded this version (0 if still live)DELETE or UPDATE (writes new row, marks old)
ctidPhysical location: (page, offset)Set on insert

A reading transaction with snapshot XID 1000 only "sees" tuples where xmin < 1000 AND (xmax = 0 OR xmax > 1000). That's how concurrent readers and writers don't block each other — readers walk the heap and skip versions that aren't visible to them.

Bloat = Dead Tuples That Nobody Cleaned Up

Why a 1 GB Table Becomes a 10 GB Table 
UPDATE users SET last_login = now() WHERE id = 42;

What actually happens on disk:
  1. Old tuple (xmin=500) gets xmax=1234 → still on page, marked superseded
  2. New tuple (xmin=1234, xmax=0) inserted, possibly on a different page
  3. Both index entries now exist (HOT update may avoid this; assume not)

After 10 million updates with no VACUUM:
  Heap pages full of dead tuples (90% dead, 10% live)
  Index entries pointing to dead tuple ctids
  Sequential scans read 10x more pages than needed
  Random reads waste cache slots on dead data → cache miss rate climbs

VACUUM reclaims the space inside the page (returns it to FSM)
VACUUM FULL rewrites the table file entirely (locks table, dangerous)

Autovacuum Trigger Math

The autovacuum daemon checks pg_stat_user_tables stats and fires per-table when this is true:

Default Autovacuum Trigger Formula 
n_dead_tup > autovacuum_vacuum_threshold
            + autovacuum_vacuum_scale_factor × n_live_tup

Defaults:
  autovacuum_vacuum_threshold     = 50      (rows)
  autovacuum_vacuum_scale_factor  = 0.2     (20% of live rows)

So a table with 1,000,000 live rows triggers VACUUM at:
  50 + 0.2 × 1,000,000 = 200,050 dead rows

A table with 100 live rows triggers at:
  50 + 0.2 × 100 = 70 dead rows

The Circular Deadlock

This is the senior-level scenario interviewers love. A long-running query holds an MVCC snapshot. While that snapshot is open, autovacuum cannot reclaim dead tuples that are still visible to the snapshot — even if those rows are dead to every other transaction. Bloat grows. The query gets slower (more pages to scan). The query takes longer to finish. The snapshot stays open longer. More bloat. The DB is now slowly suffocating itself, and a restart-the-pod fix doesn't help because the bloat is on disk.
The Loop 
1. Reporting query starts → opens snapshot at XID 5000
2. UPDATE traffic creates dead tuples with xmax > 5000 → autovacuum can't touch them
3. After 30 min, dead_tup count = millions
4. Query at step 1 still scanning → now reads bloated pages → 10x slower
5. Query takes 4 hours instead of 30 min
6. Even more dead tuples accumulate → next query slower still
7. idle_in_transaction connections from app pool extend this further

Mitigations:
  • idle_in_transaction_session_timeout = 5min → kill stuck transactions
  • statement_timeout per role for reporting queries
  • Move heavy reads to a replica (logical separation)
  • For one-off cleanup: VACUUM (not VACUUM FULL — too disruptive)

Stale Stats Trap (135 Tables in QA)

A subtler failure: autovacuum's analyze phase updates pg_statistic with row counts and column histograms. If those stats are stale, the planner makes wrong choices — and if the stats say "0 rows" the trigger formula above never fires, locking in the staleness.

Diagnosis Queries

Find Bloated Tables and Stale Stats 
-- Tables with most dead tuples (live bloat)
SELECT schemaname, relname,
       n_live_tup, n_dead_tup,
       round(100.0 * n_dead_tup / nullif(n_live_tup,0), 1) AS pct_dead,
       last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC
LIMIT 20;

-- Tables that have NEVER been analyzed (stale stats trap)
SELECT schemaname, relname, n_live_tup, last_autoanalyze
FROM pg_stat_user_tables
WHERE last_autoanalyze IS NULL
  AND last_analyze IS NULL
ORDER BY relname;

Transaction ID Wraparound — The Doomsday Scenario

XIDs are 32-bit. After ~4 billion transactions they wrap around. If VACUUM never freezes old tuples, an old XID 1000 starts looking like a future XID after wraparound — those rows become invisible (effectively data loss). PostgreSQL prevents this by FORCING an emergency shutdown when a database is within 1 million XIDs of wraparound. We have monitoring on pg_database.datfrozenxid for exactly this. "VACUUM never optional" is the lesson.
Q: "A table is 90% dead tuples. Should you run VACUUM or VACUUM FULL?"
"Default to VACUUM. It scans the table, marks dead tuples free in the Free Space Map (FSM), and updates visibility maps — all online, with only a brief SHARE UPDATE EXCLUSIVE lock that doesn't block reads or writes. The table file size doesn't shrink, but new writes will reuse the freed space.

VACUUM FULL rewrites the entire table into a new file, releases the old file, and shrinks disk usage. But it takes an ACCESS EXCLUSIVE lock — the table is unreadable and unwritable for the duration. On a 100 GB table that's potentially hours of downtime. Banking-grade ops never run VACUUM FULL during business hours.

The right pattern for serious bloat: pg_repack (extension) — rewrites the table online, no exclusive lock, copy-and-swap. We'd schedule that during a maintenance window with a CHG ticket."
Q: "Why might autovacuum not be cleaning a table even though it has millions of dead rows?"
"Five common causes, in order of likelihood:
  1. A long-running transaction is holding a snapshot — autovacuum can't reclaim tuples still visible to that snapshot. Check pg_stat_activity for state='idle in transaction' with old xact_start.
  2. Stale stats say the table has 0 live rows so the threshold never trips. Run a manual ANALYZE.
  3. Table is locked by a long DDL or a bulk operation — autovacuum politely backs off when contended.
  4. maintenance_work_mem is too small, vacuum needs many passes, takes too long, doesn't complete before the next workload spike.
  5. autovacuum is disabled per-table via ALTER TABLE ... SET (autovacuum_enabled = false) — sometimes left over from a debugging session.
The diagnostic flow: SELECT * FROM pg_stat_progress_vacuum to see if it's running, SELECT * FROM pg_stat_activity WHERE state LIKE 'idle%' for blockers, SELECT relname, last_autovacuum FROM pg_stat_user_tables WHERE n_dead_tup > 100000 to see when it last ran."
8b.5

WAL Mechanics + Checkpoint I/O Spreading Critical

The Write-Ahead Log is the durability boundary. "COMMIT" doesn't mean the data file is updated — it means the WAL record is on disk and fsync returned. Everything else (data file flush, replication, backup) happens later, asynchronously.

WAL Segments and LSN

WAL is stored as 16 MB segment files in $PGDATA/pg_wal/. Each file has a 24-character hex name encoding timeline + log file + segment number:

WAL Segment Naming 
/pgdata/pg16_wal/
├── 000000010000502C00000001   ← 16 MB segment
├── 000000010000502C00000002
├── 000000010000502C00000003

Naming: TTTTTTTTSSSSSSSSXXXXXXXX
        │       │       │
        │       │       └── Segment number within log (hex)
        │       └── Log file number (hex)
        └── Timeline ID (hex, increments after promote/failover)

LSN format: XXXXXXXX/YYYYYYYY  e.g.  502C/DFF50F18
            │        │
            │        └── Byte offset within segment
            └── Segment number

Bytes from start = (segment × 4 GB) + offset

An LSN is a monotonically increasing byte position across the entire WAL stream. The replica is "X bytes behind" means primary_LSN - replica_LSN. "Slot has 6 GB of WAL retained" means current_LSN - slot.restart_lsn.

fsync — The Durability Boundary

What COMMIT Actually Does 
1. Backend writes WAL record to wal_buffers (16 MB shared memory)
2. On COMMIT:
     a. WAL buffer flushed to /pgdata/pg16_wal/<segment>
     b. fdatasync() syscall → kernel flushes from page cache to physical disk
     c. Disk controller confirms persistent write (~0.5-2 ms on SSD)
3. ONLY after fdatasync returns success:
     → Backend sends "INSERT 0 1" / commit ACK to the client
     → Transaction is durable: survives power loss, OS crash, container kill

At this point the actual data file STILL has the old data.
The dirty page sits in shared_buffers until the next checkpoint.
But the WAL has every change needed to rebuild it — that's the contract.

WAL Retention Parameters

Three settings control WAL recycling — and they're often misunderstood. min_wal_size is a floor on retained WAL even after archival, not a checkpoint trigger.

ParameterProd valueQA valueWhat it actually controls
min_wal_size 1 GB 32 MB Floor: PG keeps at least this much WAL on disk by recycling filenames rather than deleting. Why QA shows 5 GB of WAL despite max=256MB — replication slots override.
max_wal_size 16 GB 256 MB (1) Trigger checkpoint when WAL since last checkpoint hits this. (2) Try to keep total WAL on disk below this by aggressive recycling after checkpoint.
wal_keep_size 4 GB 16 MB Minimum WAL retained for physical streaming replication to a replica that fell behind. Independent of archive.
max_slot_wal_keep_size 15 GB (CI) unset Cap on WAL retained for logical replication slots (Fivetran). If a slot lags past this, slot is invalidated → consumer must re-sync.
The order of overrides matters. Replication slots (logical) > wal_keep_size (physical replica) > max_wal_size > min_wal_size. If any consumer needs a WAL segment, it stays — even past max_wal_size. That's why QA shows 5+ GB on disk: 5 Fivetran slots each pinning ~6 GB of WAL behind them. PG would happily delete the WAL but a slot says "I haven't read past LSN X yet".

Checkpoint = Random I/O Across Many Files

WAL writes are sequential (append to one file, ~500 MB/s). Checkpoint writes are random — every dirty page in shared_buffers goes to its home position in its data file, scattered across hundreds of files (~50 MB/s on the same disk). If you flushed all dirty pages instantly, every commit would suffer a tail-latency spike.

Why Spread Over 90% of the Interval 
checkpoint_timeout              = 30 min
checkpoint_completion_target    = 0.9
max_wal_size                    = 16 GB

Checkpoint triggers:
  • timer: every 30 minutes, OR
  • WAL volume: when WAL since last checkpoint reaches 16 GB
  whichever comes first.

Once triggered, PG identifies all dirty pages in shared_buffers.
It then SPREADS the writes over (30 min × 0.9) = 27 minutes.

  Time:  0min      9min      18min     27min     30min
         │          │          │         │         │
         ▼          ▼          ▼         ▼         ▼
  I/O:  ████████████████████████████████      [idle]
         [── spread over 90% of interval ──]
                                          └─ checkpoint complete
                                             buffer for next checkpoint

Without spreading (immediate flush):
  I/O:  ████████████████████████████████████████████
         all at once — 100% disk → query latency spikes

Timed vs Requested Checkpoints

pg_stat_bgwriter — The Health Indicator 
SELECT checkpoints_timed, checkpoints_req,
       round(100.0 * checkpoints_req / nullif(checkpoints_timed + checkpoints_req, 0), 1)
         AS pct_requested
FROM pg_stat_bgwriter;

Healthy:   95%+ timed, <5% requested
Unhealthy: 30%+ requested → max_wal_size too small, checkpoints firing
           on WAL volume, not on timer. Bursty I/O, not spread.

Fix: raise max_wal_size, raise checkpoint_timeout (with corresponding
recovery-time tradeoff — more WAL to replay on crash).
Q: "What's the relationship between WAL, fsync, and a successful COMMIT?"
"COMMIT in PostgreSQL means: the WAL record describing this transaction has been flushed to physical disk via fdatasync, and the syscall returned success. Until that returns, the client is blocked. After it returns, the transaction is durable — the database can crash, the pod can be killed, the disk can be unmounted-and-remounted, and on restart PostgreSQL will replay the WAL from the last checkpoint and the row will exist.

What COMMIT explicitly does not guarantee: that the data file (table heap, index pages) has been updated on disk. Those dirty pages sit in shared_buffers potentially for the full checkpoint_timeout (30 minutes) before being flushed — but this is fine, because the WAL has every byte needed to rebuild them on recovery.

This is also why fsync = off is dangerous (and never used in prod): it lets COMMIT return without forcing the WAL to disk. Massive throughput improvement, zero durability — a power loss can lose the last several seconds of committed transactions."
Q: "Your pg_stat_bgwriter shows 70% requested checkpoints. What does that mean and how do you fix it?"
"It means checkpoints are firing because max_wal_size is too small for the write rate, not because the timer expired. Symptoms:
  • I/O is bursty, not spread — checkpoint storms during peak.
  • Query p99 latency spikes during those storms.
  • WAL is being recycled aggressively — replication slots and wal_keep_size get squeezed.
Fix sequence:
  1. Raise max_wal_size (e.g. 1 GB → 16 GB). Costs disk space but pushes timer-driven checkpoints back.
  2. Optionally raise checkpoint_timeout from 5 min to 30 min — same idea: longer interval, more spread, less frequent storms. Tradeoff: longer crash recovery (more WAL to replay).
  3. Confirm checkpoint_completion_target = 0.9 (the modern default). Lower values mean checkpoint finishes early and idle disk afterwards — wasted spreading window.
  4. Re-check pg_stat_bgwriter after a few hours. Goal: 95%+ timed."

postmaster accepts the TCP connection, forks a dedicated backend process for the session (~5-10 MB RSS, more under load). Each Npgsql pool slot maps to one backend.
2
Parse, analyze, rewrite, plan
SQL text → tokens → AST → semantic check (table OIDs, column types, permissions) → plan tree. For a simple INSERT, planner cost is trivial.
3
Heap + index modifications
Backend loads the target page into shared_buffers (cache hit or disk read), inserts the new tuple with xmin=current XID, xmax=0. Marks page dirty. Updates every index. Constraints (UNIQUE, FK, CHECK) verified.
4
WAL write → fdatasync → ACK
A WAL record describing the change goes into wal_buffers. On COMMIT, the buffer is flushed to the active pg_wal/ segment file, then fdatasync forces the kernel page cache to physical disk. Only when fsync returns success does the backend send the ACK back to the client.
5
Three parallel post-WAL paths
(A) Checkpoint — checkpointer wakes every 30 min (or on WAL volume), flushes the still-dirty page in shared_buffers to the actual data file, spreads over 27 min. (B) Archive — pgBackRest sidecar receives the 16 MB segment via archive_command when full or after archive_timeout=120s. (C) Replication — walsender streams WAL bytes to the replica's walreceiver (async by default; sync mode possible for stricter RPO).
Q: "A client commits an INSERT and gets ACK at t=0. The pod is killed at t=1ms. The new primary is promoted at t=30s. Did the row survive?"
"It depends on whether the WAL bytes had streamed to the replica before t=1ms.
  • The WAL record was definitely on the primary's disk at t=0 (that's what ACK means). When the primary comes back, it would replay from its WAL — but in this scenario the primary is gone and the replica is being promoted.
  • With asynchronous replication (default), the WAL bytes may or may not have reached the replica. Typical lag is sub-millisecond on a healthy LAN, but it's not guaranteed at the moment of ACK. If the WAL wasn't streamed yet, the row is gone after promotion — small data loss, bounded by the streaming lag (typically <1s).
  • With synchronous replication (synchronous_commit = on, synchronous_standby_names set), COMMIT doesn't return until the replica has the WAL too. Zero loss at the cost of latency proportional to network round-trip.
  • For longer outages (both nodes gone), recovery falls back to the WAL archive on the repo-host. Bounded by archive_timeout=120s — anything written in the last 2 minutes that wasn't archived yet is lost.
For our platform: async replication for performance + accepting sub-second loss tolerance for non-financial paths. Banking/payments would push for sync replication on the critical tables, accepting the latency tax."

Connection Pool Exhaustion + Patroni State Machine Critical Senior

Every Connection Costs Memory

PostgreSQL is a process-per-connection system, not thread-per-connection. Each backend is a full Unix process with its own memory pages.

Per-Connection Memory Footprint 
Idle backend:
  Process stack + heap         ~5-10 MB
  Catalog cache                ~2-5 MB     (table/column metadata)
  Relcache                     ~5-20 MB    (relation metadata)
  ───────────────────────────────────
  Per idle connection:         ~12-35 MB

Active backend running a sort/hash query:
  Above + work_mem allocations ~128 MB per sort/hash node
  ───────────────────────────────────
  Per active connection:       ~150-500 MB

200 idle connections × 25 MB avg  = 5 GB (just sitting there)
200 connections, 20 active query  = 5 GB idle + 4 GB active = 9-10 GB
idle_in_transaction is the worst kind of idle. A connection that issued BEGIN and never committed holds (1) a snapshot — blocks autovacuum from reclaiming visible-to-it dead tuples, and (2) any locks acquired during the transaction — blocks DDL and other writers. Always set idle_in_transaction_session_timeout = 5min to kill these. Saw this exact pattern in INC-010 (ReportPortal exhausted PG connections) — a runaway service held connections open in idle-in-transaction state, autovacuum got blocked, pg_stat_activity filled up to max_connections, every other service got "FATAL: too many clients".

PgBouncer — The Pooler Pattern

Why PgBouncer Saves Real Money 
Without PgBouncer:
  100 .NET pods → 100 PG connections → 100 × 25 MB = 2.5 GB idle
                                       100 backend processes context-switching

With PgBouncer (transaction mode):
  100 .NET pods → PgBouncer → 15 PG connections → 15 × 25 MB = 375 MB idle
                              15 backends that the pooler shares

  Memory saved:                ~2.1 GB
  Connection overhead saved:   85 fewer fork()s
  Connection-establishment latency: amortised across many app requests

PgBouncer modes: session (one PG conn per client, useful for long-lived sessions but offers no pooling benefit), transaction (PG conn returned to pool at each COMMIT — most common), statement (returned at each statement, breaks transactions — rarely used). For Confirmation, transaction-mode pooling is the right default; the only gotcha is that prepared statements and SET statements need careful handling.

Failover Step by Step

1
Healthy state
Primary holds the DCS leader key with a TTL (default 30s). Patroni renews the key every loop_wait (default 10s) — the key never expires while the primary is healthy. Replica streams WAL.
2
Primary fails
Pod crashes, network partition, or kubelet kills it. No more heartbeats to DCS.
3
Leader key expires
After TTL (30s) the key disappears from the DCS. Surviving Patronis see "no leader" — they're allowed to race for the key.
4
Race + LSN check
Each candidate replica checks its own LSN versus what's recorded in DCS. The candidate with the highest LSN (closest to what the primary had) wins. It writes its name as the new leader key.
5
Promotion
The winner calls pg_promote() on its local PostgreSQL — the replica becomes a primary, starts a new timeline, opens for writes.
6
Service repoint
PGO watches Patroni state via the Kubernetes API and updates the Service endpoints to point to the new primary. Apps reconnect (Npgsql retries).
7
Old primary returns
When the failed pod restarts, Patroni notices someone else is leader. It runs pg_rewind against the new primary to discard any divergent WAL it had buffered, then rejoins as a streaming replica.

What If the DCS Itself Fails?

If the DCS (etcd / Kubernetes API) becomes unreachable, Patroni demotes the primary to read-only as a safety measure. The reasoning: if the primary can't reach the DCS, it can't tell whether it's still leader — and if a replica somewhere has assumed leadership in a partition, both writing would be split-brain. The conservative choice is "stop accepting writes until I can confirm I'm still leader". This is why DCS availability is treated as critical infrastructure — losing the K8s API server (our DCS backend) means writes pause everywhere on the cluster, not just for one app.
Q: "Your application logs show 'FATAL: too many clients already'. How do you respond?"
"Triage in this order — INC-010 was exactly this pattern:
  1. Confirm: SELECT count(*), state FROM pg_stat_activity GROUP BY state; — is it really at max_connections? What's the breakdown of active vs idle vs idle_in_transaction?
  2. Find the offender: SELECT application_name, count(*) FROM pg_stat_activity GROUP BY application_name ORDER BY 2 DESC; — usually one runaway service.
  3. Look for idle_in_transaction: SELECT pid, application_name, state, xact_start, query FROM pg_stat_activity WHERE state = 'idle in transaction' ORDER BY xact_start; — these are the silent killers. Old xact_start = stuck transaction.
  4. Immediate mitigation: SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state='idle in transaction' AND xact_start < now() - interval '5 minutes'; — kill the stuck ones to free slots.
  5. Tactical fix: restart the offending service pod (its connection pool is probably leaking). Confirm count drops.
  6. Strategic fix: introduce PgBouncer in front of Postgres for that service. Set idle_in_transaction_session_timeout = 5min in postgresql.conf so this can never fully exhaust again. Add a Datadog monitor on pg_stat_activity count > 70% of max_connections.
Q: "Walk me through what happens during a Patroni failover from the application's perspective."
"From the .NET service's view:
  1. t=0: Primary dies. Active queries hang (TCP FIN never arrives, connections in CLOSE_WAIT or just stuck).
  2. t=10-30s: TCP keepalive on existing connections starts seeing failures. Npgsql's connection-validation logic kicks in — the next time the pool tries to lease a connection, the validate fails and the pool starts opening new ones.
  3. t=10-30s in parallel: Patroni observes leader-key expiry, replica wins the race, calls pg_promote(), writes itself as new leader. PGO updates the K8s Service endpoints.
  4. t=30-60s: New connections from Npgsql resolve confirmation-pgdatabase-qa-primary.207804-postgres-qa.svc via the now-updated Service endpoints — they land on the new primary. Reads and writes resume.
  5. What the user sees: a 30-60s window of errors (or retries with exponential backoff if the app has them). Confirmation services use Polly retry policies for this — most of the user-visible impact is masked.
  6. Old primary returns: pg_rewind runs against the new primary to align any WAL it had locally that didn't make it across. Rejoins as replica. The cluster is back to 1+1.
The critical detail: the Service repoint happens at the K8s networking layer — DNS resolution doesn't change, but the endpoint backing that DNS does. Apps that use long-lived connections without validation logic won't notice the failover until they try to use a stale connection."

Fivetran Logical Slots + PITR Recovery Critical

Logical Replication Slots — The Anchor That Pins WAL

Fivetran uses PostgreSQL logical replication slots to stream change events to its data lake. A slot is a server-side bookmark with two LSN positions:

Slot fieldWhat it tracksEffect on WAL retention
restart_lsn Earliest LSN the consumer might still need (e.g. start of an open transaction) PostgreSQL cannot recycle WAL beyond this LSN. WAL files older than restart_lsn stay on disk forever, regardless of max_wal_size.
confirmed_flush_lsn LSN that the consumer has confirmed it processed Reporting only — used to compute lag.

The Failure Mode

If a Fivetran consumer falls behind, WAL piles up on disk. The chain: producer (PG) writes WAL faster than consumer (Fivetran agent) reads → restart_lsn doesn't advance → WAL files accumulate → $PGDATA/pg_wal/ fills up → at some point disk runs out and PostgreSQL stops accepting writes (read-only mode). The defence is max_slot_wal_keep_size: when a slot retains more than this, PG marks the slot invalidated and starts recycling WAL again — but the cost is that Fivetran has to do a full re-sync from the source tables (slow, expensive on large datasets).

Slot Manager CronJob — Heals After Failover

Logical slots only exist on the primary (they're not replicated to the replica). When Patroni failover happens, the new primary has zero slots — Fivetran can't reconnect, and re-sync would be required.

Slot Manager (runs every 5 min) 
schedule: "*/5 * * * *"

What it does:
  1. Connects to current primary (via -primary service)
  2. Checks if all 5 expected logical slots exist
  3. If any missing → creates them (CREATE_REPLICATION_SLOT logical)
  4. If Fivetran connectors need re-syncing → calls Fivetran API to reset

Net effect: Fivetran experiences a small lag but no full re-sync

Point-in-Time Recovery (PITR)

PITR combines a base backup with a WAL replay to restore the database to any second within the retention window.

PITR Recovery Sequence 
Goal: restore to "Apr 14 14:23:17 UTC" (pre-incident timestamp)

Step 1: Restore base backup
  pgbackrest restore --type=time --target='2026-04-14 14:23:17 UTC'
   → reads latest full + diff (e.g. full from 14 Apr 21:00 prior day)
   → writes data files to /pgdata
   → leaves recovery.conf with target_time set

Step 2: Replay WAL from archive
  PostgreSQL starts in recovery mode
   → reads WAL segments from pgBackRest archive
   → replays each record: BEGIN, INSERT, COMMIT, INSERT, ...
   → STOPS at the first transaction whose commit_time > target_time

Step 3: Promote
  pg_promote() (or recovery_target_action = 'promote')
   → DB becomes a normal primary on a new timeline

Recovery time = (data restore I/O) + (WAL replay rate × WAL volume since base)
RPO = bounded by archive_timeout = 120 seconds

March 2026 DR Data Gap — The Lesson

What happened: An incident in DR required PITR to a specific second. The team kicked off the restore — base backup applied cleanly, WAL replay started — and replay halted at a missing WAL segment. Some segments in the band of interest had simply never been archived. The archive_command had been silently failing for hours; pg_stat_archiver.last_failed_wal was set but no monitor was firing on it. PG itself was healthy (writes succeeded locally), but the archive was incomplete.

The lesson:
  • A successful local fsync does not mean the WAL has been archived. The archive is an asynchronous, separate-failure-mode pipeline.
  • Always alert on pg_stat_archiver.last_failed_wal IS NOT NULL — and on the archived_count rate going to zero.
  • Always alert on pg_stat_archiver.last_archived_time growing stale (e.g. > 10 min behind now()).
  • Periodically test a PITR restore end-to-end — a backup you've never restored is not a backup.
Q: "How does Fivetran avoid losing data when Patroni fails the primary over?"
"Logical replication slots only exist on the primary — when the replica gets promoted, the slots don't come along. Fivetran would normally need to do a full re-sync from the source tables (potentially hours, depending on size).

Our platform mitigates this with a Slot Manager CronJob that runs every 5 minutes. It connects to whoever is currently the primary, checks for the 5 expected logical slots, and recreates any that are missing. After a failover, within 5 minutes the slots are back. Fivetran reconnects and resumes from the slot's confirmed_flush_lsn — which is at most 5 minutes of lag, not a full re-sync.

The tradeoff: any WAL events generated between failover and slot recreation are lost from Fivetran's perspective. For our analytics use case that's acceptable. If we needed strict zero-loss CDC, we'd run synchronous replication and use a slot-replication extension like pg_failover_slots — but that's not free, especially on Fivetran which doesn't natively integrate with it (this came up in INC-012)."
Q: "What's the difference between RPO from a streaming replica vs from the WAL archive?"
"Streaming replica RPO is sub-second — the replica receives WAL bytes via a TCP stream as soon as the primary writes them. Lag is typically <100ms on a healthy LAN. If the primary dies and Patroni promotes the replica, you lose at most the in-flight bytes between primary's last fsync and the replica's last applied LSN.

WAL archive RPO is bounded by archive_timeout — in our case 120 seconds. The archive pipeline ships completed 16 MB segments to the repo-host. If a segment hasn't filled and 120 seconds have elapsed, PG forces a partial switch and archives what's there. Worst case: the primary dies just before that switch — you lose up to 120 seconds of transactions.

The recovery hierarchy:
  1. Streaming replica (sub-second RPO) — Patroni promotes automatically.
  2. WAL archive (120s RPO) — used when both nodes are gone.
  3. Last diff backup (6h RPO) — used if WAL archive is also gone or corrupt.
  4. Last full backup (24h RPO) — last-ditch option.
The March 2026 DR gap was specifically a tier-2 failure: streaming replica was unavailable (DR scenario), and the WAL archive had silent archive_command failures, leaving us with sub-tier-3 recovery only. Now we monitor archive health as a top-tier alert."

Diagnostic Cheat Sheet Senior

Memorise these seven queries. Every PostgreSQL incident on a managed cluster starts with one of them.

1. pg_stat_activity — Who's Connected and What Are They Doing 
SELECT pid, usename, application_name, client_addr,
       state, xact_start, query_start, wait_event_type, wait_event,
       left(query, 80) AS query
FROM pg_stat_activity
WHERE state <> 'idle'
ORDER BY xact_start NULLS LAST;
-- First check during connection storms, slow queries, idle_in_transaction.
2. pg_stat_user_tables — Bloat and Vacuum Health 
SELECT schemaname, relname,
       n_live_tup, n_dead_tup,
       round(100.0 * n_dead_tup / nullif(n_live_tup,0), 1) AS pct_dead,
       last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC LIMIT 20;
-- Find tables with most bloat, when autovacuum last ran.
3. pg_stat_bgwriter — Checkpoint Health 
SELECT checkpoints_timed, checkpoints_req,
       round(100.0 * checkpoints_req /
             nullif(checkpoints_timed + checkpoints_req, 0), 1) AS pct_requested,
       buffers_checkpoint, buffers_clean, buffers_backend,
       maxwritten_clean
FROM pg_stat_bgwriter;
-- 95%+ timed = healthy. High pct_requested → max_wal_size too small.
4. pg_replication_slots — Logical Slot Lag and WAL Pinning 
SELECT slot_name, slot_type, active, wal_status,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn))
         AS retained_wal,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn))
         AS lag
FROM pg_replication_slots
ORDER BY retained_wal DESC;
-- Watch for wal_status = 'extended' or 'lost' — slots about to be invalidated.
5. pg_stat_archiver — WAL Archive Health (DR-Critical) 
SELECT archived_count, last_archived_wal, last_archived_time,
       failed_count, last_failed_wal, last_failed_time,
       now() - last_archived_time AS staleness
FROM pg_stat_archiver;
-- last_failed_wal NOT NULL = silent archive failure. Alert on this.
-- staleness > 10 min on a busy DB = pipeline broken.
6. pg_stats Freshness — Stale Statistics Trap 
SELECT n.nspname AS schema, c.relname AS table,
       c.reltuples::bigint AS planner_estimate,
       s.n_live_tup AS actual_live,
       s.last_analyze, s.last_autoanalyze
FROM pg_class c
JOIN pg_namespace n ON n.oid = c.relnamespace
JOIN pg_stat_user_tables s ON s.relname = c.relname AND s.schemaname = n.nspname
WHERE c.relkind = 'r'
  AND (s.last_analyze IS NULL AND s.last_autoanalyze IS NULL
       OR abs(c.reltuples - s.n_live_tup) > 0.5 * greatest(c.reltuples, 1))
ORDER BY abs(c.reltuples - s.n_live_tup) DESC LIMIT 20;
-- Tables where planner estimate diverges from actual → bad query plans.
7. pg_locks — Blocking and Deadlock Investigation 
SELECT blocked.pid AS blocked_pid,
       blocked.query AS blocked_query,
       blocking.pid AS blocking_pid,
       blocking.query AS blocking_query,
       blocked.wait_event_type, blocked.wait_event
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking
  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE blocked.wait_event_type = 'Lock';
-- Identifies who's blocking whom. Use pg_terminate_backend(blocking_pid) carefully.
Memorise the order of triage: #1 to see who's there, #7 to find blockers, #2 to check vacuum health, #4 if logical replication or Fivetran involved, #5 if PITR/DR involved, #3 if checkpoint storms suspected, #6 if planner is making bad choices on a known-good schema.
