🌍 Mental Model Two buildings connected by a tunnel: The cloud VPC is your headquarters in Manhattan (lots of services, multi-AZ, internet-connected). The Outpost VPC is a satellite office in a regulated facility on-premises (single-AZ, fixed capacity, BGP peering with corporate routers). The Service Link is a private dedicated tunnel between them so HQ knows the satellite exists and can manage it. The Local Gateway is the satellite office's loading dock — that's where customer traffic enters and leaves through the corp network. Everything in this section is "which door does the packet take, and why?"
15b.2

Two VPCs — Outpost VPC vs Cloud VPC Differentiator

An Outpost is an extension of an AWS region — but the way it actually works architecturally is that you run one logical VPC that spans both the rack and the region. In TR's preprod account (833618288067), this is vpc-0d4351e45bfd185a1 ("tr-vpc-1"). It carries three CIDR blocks: 10.137.124.0/22 (cloud subnets), 10.154.16.0/21 (Outpost rack node subnet), and 100.81.64.0/19 (CGNAT pod IP subnet on the rack).

A second VPC — vpc-0c3dc6b4bdf952c96 ("depr-tr-vpc-1", 172.28.32.0/19) — is a deprecated legacy VPC for CodeBuild and pen-testing, peered to the active VPC via pcx-03c96331909a36681. The Outpost route table sends 172.28.32.0/19 traffic to that peering connection.

Why Two Halves?

The rack is single-AZ by definition (us-east-1a) — there's one physical Outpost in TR's datacenter. But you still want HA for managed AWS services (NAT, VPC endpoints, EKS control plane ENIs). You can't put those in the rack — they're region-only services — so the cloud anchor exists to host them across three AZs (1a, 1b, 1c). The Outpost subnet talks to those cloud-anchor resources over the Service Link as if they were just another subnet in the same VPC.

15b.4

Why RFC 6598 (100.64.0.0/10) for Pod IPs? Differentiator

Pod IPs in the Confirmation platform look like 100.81.66.91 or 100.81.86.203. That's not RFC 1918 (the familiar 10.x, 172.16/12, 192.168/16 private ranges) — it's RFC 6598, the "shared address space" allocation reserved for Carrier-Grade NAT (CGNAT) operators. The full block is 100.64.0.0/10, which gives carriers a private-ish range that won't collide with any customer's home 192.168.1.x network.

Why Did TR Pick CGNAT for Pod IPs?

ReasonWhy It Matters
1. Corporate IPv4 scarcity TR's enterprise 10.x allocation is rationed across thousands of internal teams. Asking for an 10.x /19 just for pod IPs (8K+ addresses on a single Outpost rack) burns through a corp allocation that takes months to get approved. RFC 6598 sidesteps that — it's not part of the corp routable space.
2. Avoid collisions with peered VPCs The deprecated VPC uses 172.28.32.0/19. CurrentGen has its own 10.x ranges. NAT mappings on the TechM extranet firewall translate legacy 192.168.x.x → 10.131.x.x. Pod IPs in 100.81.x can't possibly collide with any of those.
3. VPC CNI custom networking The AWS VPC CNI plugin supports a "custom networking" mode where node primary ENIs use one subnet (10.154.x) and secondary ENIs / pod IPs come from a different subnet (100.81.x). That decouples node count scaling from pod density scaling.
4. Pod IPs never need to be routable on the public internet Pods reach external endpoints either via VPC endpoints (which SNAT internally), via the corp proxy (which SNATs at the proxy), or via the LGW with explicit BGP advertisement. In none of those cases does the pod IP itself need to be globally unique.
📞 Analogy Why RFC 6598 was created: when ISPs ran out of IPv4 and started double-NATing customers, they needed a private range that wouldn't collide with the customer's home network (192.168.x.x) OR the ISP's existing internal infrastructure (10.x). IETF carved out 100.64.0.0/10 for that purpose. TR is essentially using the same idea: pod IPs need to be private and large-scale, but mustn't collide with corp IPs.

Q: "Why doesn't pod traffic to the corp network conflict with someone else's 100.64 range? Plenty of carriers use that block."
"Because the LGW only announces TR's specific /19 — not the whole /10. The Local Gateway BGP-advertises 100.81.64.0/19 (8K addresses) to TR's datacenter top-of-rack switches. The TR network learns that this specific /19 is reachable through the Outpost LGW. If some other team or carrier were using a different RFC 6598 slice — say 100.64.0.0/19 — that wouldn't conflict because BGP would route it to a different next-hop.

The conflict you'd worry about is if TR ever directly peered with an external network that also used 100.81.x. But that's not the threat model here — RFC 6598 is non-routable on the public internet by design, and within TR's BGP table the network team manages allocation just like they do for 10.x ranges.

The real win of RFC 6598 over 10.x isn't 'no collisions ever' — it's 'no need to ask the corp network team for a giant 10.x allocation that they'd refuse'."

What "direct-vpc-routing" Means

direct-vpc-routing mode: the Outpost subnet IPs (10.154.16.0/21 and 100.81.64.0/19) are advertised by the LGW directly into the datacenter BGP fabric. From a TR datacenter switch's perspective, those addresses are just another routable corp prefix. No NAT happens at the LGW. A pod with IP 100.81.66.91 sending a packet to 10.131.5.10 (an on-prem PostgreSQL) keeps that source IP all the way through the LGW, through the ToR switch, into the DC core network, until either a firewall NATs it or the destination receives it directly.
Alternative mode — CoIP (Customer-Owned IP): the second LGW route table (lgw-rtb-0de7ed9502631db6f) is configured for CoIP mode but is inactive (no virtual interface group association). CoIP would let Outpost workloads use elastic IPs in a customer-owned IP pool — useful if you need stable, customer-routable IPs for inbound access. We don't use it; ingress comes through F5 / Istio ingress gateway instead.

Why a Private 32-bit ASN?

The LGW uses ASN 4200003112. That's in the 4-byte private ASN range (4200000000–4294967294). The 2-byte private range (64512–65534) only has ~1000 ASNs, and a large enterprise like TR runs out fast across hundreds of internal sites and Outposts. 4-byte private ASNs (RFC 6996) give effectively unlimited supply, and modern BGP implementations (RFC 4893) carry them transparently.

Q: "VIF 1 (VLAN 2511) goes down. Walk through what happens."
"BGP on VIF 1 keepalive timer (typically 10s, hold timer 30s default for AWS LGW BGP) expires. The LGW marks VIF 1's BGP session as Down. Routes learned via VIF 1 are withdrawn from the LGW's RIB. But VIF 2 (VLAN 2513) is still up and was advertising the same prefixes — so the LGW route table still has paths for 10.154.16.0/21 and 100.81.64.0/19 via VIF 2's peer (10.154.15.6/31). Datacenter ToR switch B has the same routes in its RIB.

Failover time: BGP convergence with default timers is 30-90 seconds — for sub-second failover you'd configure BFD (Bidirectional Forwarding Detection) which AWS LGW now supports. During that 30-90s window, in-flight TCP connections may break. New connections automatically use VIF 2.

What we'd see on the DevOps side: brief spike in 5xx as TCP retries fail, Datadog connection-drop metric on the istio sidecars, then recovery. If both VIFs went down simultaneously, the LGW is effectively offline — pods can't reach on-prem PostgreSQL, Orleans grain calls fail, MassTransit RabbitMQ traffic via on-prem broker dies. That's a P1 incident, not a brownout."

Service Link — The Hidden Tunnel Your Platform

The Service Link is an encrypted WAN tunnel between the Outpost rack and the AWS region. It's the lifeline that lets the rack pretend it's a normal AWS subnet. AWS owns it end-to-end — you can't tune it, audit it, or replace it. You only choose its bandwidth at order time.

The Service Link Subnet Is AWS-Managed

The 10.137.127.192/26 subnet (64 IPs) holds AWS-managed ENIs that terminate the Service Link tunnel on the cloud side. You cannot put workloads in this subnet. You cannot modify its route table. AWS Landing Zone team owns it. If you see this subnet referenced in your VPC inventory, leave it alone — modifying it can sever the rack from AWS.
Q: "What's the difference between Service Link and LGW, and which one can you replace if you needed to?"
"Service Link is AWS-managed; LGW is customer-managed. You can't replace, tune, or troubleshoot the Service Link — if it's slow, you open a support ticket and AWS investigates. You only control its bandwidth tier at rack order time (1, 10, or 100 Gbps), and resizing requires a support case.

The LGW, on the other hand, you fully own. You picked the BGP ASN (4200003112), you picked the VLAN IDs (2511, 2513), you ran the cabling from the rack to the ToR switches, and you maintain the BGP peering on the datacenter side. If the LGW has issues, your network team troubleshoots it — not AWS.

Practical example: if pods can't reach Secrets Manager, that's almost always a Service Link or VPC endpoint issue — AWS-side. If pods can't reach on-prem PostgreSQL, that's an LGW or BGP issue — customer-side. Knowing which side owns which path is the first triage step."

Route Tables — Where Does Each Packet Go? Differentiator

Route tables are where the Outpost vs cloud distinction becomes concrete. Same VPC, but the Outpost subnet's route table sends 0.0.0.0/0 to the LGW; the cloud subnet's table sends it to the TGW. Longest prefix match wins — specific routes (S3 prefix list, peered VPC CIDR, VPC endpoint) override the default.

CHG0130049 — The Proxy Routing Change Differentiator

This is the single most operationally consequential network change to the Outpost since it was provisioned. Date: February 26, 2026. The default route on the Outpost subnet route table flipped from TGW to LGW — meaning every outbound packet from a pod now exits via the corp proxy unless explicitly excepted. The CHG was driven by TR Security: they wanted egress visibility, DLP scanning, and threat inspection on all internet-bound traffic from the platform.

Why The Proxy Blocks Private IPs

Corp proxy blocks CONNECT to RFC 1918 (10.x, 172.16/12, 192.168/16) destinations. The proxy is designed for internet egress, not internal routing — sending a private IP to it is an indicator of misconfiguration or attempted exfiltration. Result: any internal endpoint that resolves to a 10.x IP must be in NO_PROXY, or pods will fail with 403 Forbidden or 502 Bad Gateway from the proxy.

VPC endpoint ENIs (10.137.124.x) hit this trap. CurrentGen NAT-translated IPs (10.131.x) hit this trap. Internal CG hostnames that resolve to 10.x hit this trap (this is INC-004).

Q: "Walk me through what changed in CHG0130049 and why."
"Before the change, the Outpost subnet's default route went to a Transit Gateway, which routed straight to corp / internet without inspection. Why TGW Is On The Cloud Side Only

TGW is a region-only managed service — you can't deploy a TGW on an Outpost rack. So the TGW attachment is to the cloud subnets of the active VPC. The Outpost subnet does not directly attach to TGW. Pre-CHG0130049, the Outpost route table's default route went via TGW (because traffic hopped: Outpost subnet → service link → cloud subnet → TGW). Post-CHG, the Outpost subnet's default goes via LGW instead, sidestepping the TGW entirely for egress.

Cloud subnets still use TGW for their default route. So if you have an EC2 instance in a cloud private subnet (e.g., the NAT GW or an endpoint ENI test instance), its egress flows differently than a pod's egress.

Interview Q&A Bank Differentiator

Q1: "What's an Outpost and how does it differ from a Local Zone?"
"AWS Outposts is a physical rack of AWS-owned hardware that sits in your datacenter. You lease it monthly. It runs EC2, EBS, EKS, ECS, RDS (limited), and S3 on Outposts — all locally on the rack. Connectivity to the AWS region is via the Service Link (encrypted, AWS-managed) for control plane and regional services; connectivity to your DC LAN is via the Local Gateway (BGP-peered to your switches).

A Local Zone is different — it's an AWS-owned facility in a metro near you (e.g., AWS Local Zone in Los Angeles). You don't own the building, AWS does. Local Zones give you sub-10ms latency to the nearest population centre and run a subset of AWS services. They don't help with on-prem data residency — the data is still in an AWS facility, just a closer one.

The use case split is: Outposts for data residency + on-prem latency (TR audit confirmation runs on Outposts because data must be in TR's DC). Local Zones for edge latency to consumers (gaming, video streaming, real-time analytics)."
Q2: "Why does an Outpost setup use two VPCs (or two halves of one VPC)?"
"It's actually one logical VPC with two physical realms — sometimes people call them two VPCs but technically it's a single VPC with multiple subnets, some on the rack, some in the cloud region.

You need both halves because:
  1. Outpost is single-AZ. The rack lives in one physical location. You can't get cross-AZ HA out of it.
  2. Some AWS services are region-only. NAT Gateway, VPC Interface endpoints, EKS control plane ENIs — you can't put any of these on the rack. They have to live in cloud subnets.
  3. HA for those region-only services. You put one cloud subnet per AZ (1a, 1b, 1c) so endpoint ENIs replicate. If 1a goes down, ENIs in 1b/1c still serve the Outpost.
The Outpost subnet talks to the cloud subnets transparently over the Service Link — from a pod's perspective, they're just neighbours in the same VPC. The route tables differ though: Outpost subnet sends 0.0.0.0/0 to LGW, cloud subnets send it to TGW.

A separate (deprecated) VPC also exists in our account — 172.28.32.0/19, peered — for legacy CodeBuild and pen-testing workloads. That's a real second VPC, but it's not part of the Outpost story."
Q3: "What runs on the rack vs in the region?"
"Cleanly split:

On the rack (Outpost): EKS worker nodes, pods (Confirmation services), local EBS gp2 volumes, Fivetran agent EC2 instance, dbt runner EC2, S3 on Outposts (PG backups, OpenSearch snapshots). Single AZ. Capacity is fixed by the rack model.

In the region (cloud anchor + AWS-managed): EKS control plane (managed by AWS), VPC Interface endpoint ENIs (ECR, Secrets Manager, STS, SSM, CloudWatch), NAT Gateway, IGW, Service Link tunnel termination ENIs, the regional services those endpoints proxy to (ECR registry, SM, STS, etc.), TGW, and any cloud-native databases (RDS, DynamoDB regional).

Key consequence: a Service Link outage doesn't kill the cluster — pods on the rack keep running with cached secrets and existing connections. But it kills new activity: ECR pulls fail, ExternalSecrets refresh fails, kubelet can't talk to API server, scheduling stops. Existing in-flight work survives, anything new dies."
Q4: "How does a pod on the Outpost reach S3?"
"Depends on which S3:

Regional S3 (my-bucket.s3.us-east-1.amazonaws.com): the Outpost route table has an entry for the AWS-managed S3 prefix list (pl-63a5400a) pointing at the S3 Gateway endpoint vpce-01efb7c5f93543f68. The pod sends the request, route table matches the prefix list, packet goes to the Gateway endpoint via Service Link, and reaches regional S3. Free, no proxy, no LGW.

S3 on Outposts (s3-outposts.us-east-1.amazonaws.com — different service, different endpoint): there's an Interface endpoint vpce-0d73aae700160bffb. Pod traffic goes to the Interface endpoint ENI, which is in the cloud anchor subnet, then back into the rack to the local S3 service. Slightly more circuitous — the endpoint ENI lives in cloud, but the actual S3 storage is on the rack.

For both, s3.us-east-1.amazonaws.com and s3-outposts.us-east-1.amazonaws.com need to be in NO_PROXY — otherwise the corp proxy intercepts and the request fails."
Q5: "Walk me through what changed in CHG0130049 and why."
"Date was Feb 26, 2026. The Outpost subnet's default route flipped from TGW to LGW. Before, 0.0.0.0/0 went to tgw-0d3aebf5bb40f0224 → corp / internet directly. After, it goes to lgw-07a5d519d3cb15dcf → DC LAN → corp proxy at webproxy.df1.corp.services:80. Same change applied to 10.0.0.0/8 — all private IP traffic also via LGW.

Why: TR Security wanted egress visibility, DLP scanning, and threat inspection on every outbound HTTP request from the platform. The corp proxy already did all that for the rest of TR — the change brought the Outpost in line.

Operational impact: every microservice now needs HTTPS_PROXY env, NO_PROXY includes .cluster.local, .svc, internal CIDRs, all VPC endpoint hostnames, and any service-specific internal endpoints. The corp proxy blocks CONNECT to RFC 1918 IPs, so any internal endpoint that resolves to 10.x must be in NO_PROXY or it gets blocked.

The first big lesson was INC-004: reports-be was hitting Commercial Gateway hostnames that weren't in NO_PROXY. The .NET HTTP client tried the proxy, proxy returned 502 because the destination was a private 10.x IP, .NET retried with exponential backoff for 53-57 seconds before timing out. Fix was adding the CG hostnames + their NAT IPs to the reports-be values.yaml NO_PROXY."
Q6: "Could you put a NAT Gateway on the Outpost itself?"
"No. NAT Gateway is an AWS-managed service that's only available in regional VPCs — not on Outposts. The supported-services matrix is explicit about this. The closest you can do is host an EC2-based NAT instance on the rack, but that's an old pattern with a single point of failure and no managed lifecycle.

What we do instead: outbound internet from pods follows the LGW path — LGW → DC LAN → corp proxy → TR's internet egress. Effectively, the corp network is our NAT — the proxy SNATs traffic to TR's internet IP block and inspects it. That gives us inspection / DLP that a NAT Gateway wouldn't.

For traffic that needs cloud-side egress (some legacy paths), we route it via TGW → cloud anchor private subnet → NAT Gateway in cloud subnet → IGW. That's what the cloud subnets' default route does. But for Outpost workloads, post-CHG0130049, the LGW + corp proxy path is the standard.

The architectural insight: on a regulated on-prem deployment, you usually want egress to flow through corp inspection rather than a regional NAT — otherwise you give up the visibility that drove putting workloads on-prem in the first place."
Q7 (bonus): "How would you debug a pod that suddenly can't reach Secrets Manager?"
"Step-by-step:
  1. kubectl exec -it pod -- env | grep -i proxy — verify HTTPS_PROXY is set, NO_PROXY includes secretsmanager.us-east-1.amazonaws.com.
  2. kubectl exec -it pod -- nslookup secretsmanager.us-east-1.amazonaws.com — should resolve to a private 10.137.124.x IP (endpoint ENI). If it resolves to a public IP, the VPC private DNS isn't being used — check VPC enableDnsSupport and enableDnsHostnames.
  3. kubectl exec -it pod -- curl -v https://secretsmanager.us-east-1.amazonaws.com — should connect with TLS handshake. If "connection refused" check the endpoint ENI status; if "403 from proxy" check NO_PROXY.
  4. aws ec2 describe-vpc-endpoints --vpc-endpoint-ids vpce-002fbb41a128322fa — verify endpoint state is "available" and the security group allows :443 from the Outpost CIDR.
  5. kubectl get externalsecret -n <ns> — check Status conditions; "SecretSyncedError" means the operator itself is failing.
  6. If everything looks fine on Outpost side, check Service Link health in CloudWatch — OutpostsLinkLatency metric. Sustained >100ms p95 = degradation; AWS support case territory."

Lead with the concrete details — the subnet IDs, ASNs, VLANs, CHG number — and the conversation will go in your direction.
