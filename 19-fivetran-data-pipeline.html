19.1

Why Fivetran? The Schema Remapping Problem Differentiator

The first question any data architect will ask is "why pay for Fivetran when PostgreSQL has native logical replication built in?" The answer is a single, hard PostgreSQL limitation: native logical replication cannot remap schema names from publisher to subscriber. That one line is why Fivetran exists in our architecture.

The Goal

Replicate data from 5 source databases (cfm_core_{env}, cfm_core_rel_{env}, cfm_forms_{env}, cfm_pricing_{env}, cfm_record_{env}) into 1 destination database (cfm_datalake_{env}) on the data warehouse. dbt then transforms this raw data into analytics tables that Power BI reads.

The PostgreSQL Restriction (verbatim from the docs)

From postgresql.org/docs/current/logical-replication-restrictions.html: “The tables are matched between the publisher and the subscriber using the fully qualified table name. Replication to differently-named tables on the subscriber is not supported.” Unchanged from PG 10 through PG 18. There is no MAP SCHEMA parameter, no configuration option, no extension hook in core PG.

The Collision

Multiple source databases share the same table names under the public schema. With native PG replication into one destination:

Source DBSource tableDestination (forced by PG)Result
cfm_core_cipublic.orleansstoragepublic.orleansstorageFirst write OK
cfm_record_cipublic.orleansstoragepublic.orleansstoragePK collision
cfm_forms_cipublic.orleansstoragepublic.orleansstoragePK collision

Even if you pre-create schemas like _raw_cfm_core_ci on the destination, native replication won't use them. When a WAL record arrives tagged public.orleansstorage, PG looks for exactly that — not your custom schema.

How Fivetran Solves It

Fivetran does not use PG's native subscriber. Instead it:

1
Reads the WAL independently via pgoutput
Same protocol PG uses, but Fivetran is the consumer, not another PG instance. It owns the replication slot, holds the LSN bookmark, and decodes row-level changes itself.
2
Adds a configurable schema prefix per connector
Each connector has a destination schema prefix setting in the Fivetran UI. Source public.X from cfm_core_ci gets rewritten to _raw_cfm_core_public.X before write.
3
Writes with its own INSERT/UPDATE statements
Not WAL replay — regular SQL DML against the destination. Fivetran chooses the schema name. No collision. Auto-creates schema and tables if they don't exist.

The 5 Connector Schema Mapping

ConnectorSource DBPrefixDestination schema
1cfm_core_{env}_raw_cfm_core_raw_cfm_core_public
2cfm_core_rel_{env}_raw_cfm_core_rel_raw_cfm_core_rel_public
3cfm_forms_{env}_raw_cfm_forms_raw_cfm_forms_public
4cfm_pricing_{env}_raw_cfm_pricing_raw_cfm_pricing_public
5cfm_record_{env}_raw_cfm_record_raw_cfm_record_public
Naming formula: {prefix}_{source_schema}. Fivetran appends _public because the source schema is public. Result: 5 collision-free Bronze schemas in cfm_datalake_{env}, dbt knows exactly which source produced which row.
Q: "Can't you just use native PostgreSQL logical replication and skip the Fivetran cost?"
"No, and the reason is specific. Native PG replication matches publisher tables to subscriber tables by fully qualified name — schema.table. The PG docs spell it out: replication to differently-named tables on the subscriber is not supported, unchanged from PG 10 through PG 18. We have 5 source databases, each with a public.orleansstorage table. They all need to land in one destination database for dbt to do cross-source joins. Without remapping, three of those five writes collide on the same public.orleansstorage primary key.

We considered alternatives: separate destination DB per source breaks cross-DB joins in PG, trigger-based forwarding doubles write load and is fragile, pglogical isn't supported by our PGO/Crunchy operator, and Debezium+Kafka is an entire CDC platform to maintain for 5 connectors. Fivetran's schema prefix feature solves the only blocker for ~$1,500/mo plus an EC2 agent. Cheap relative to what we'd build otherwise.

The honest framing: we're not paying Fivetran for its 600+ connector ecosystem — we use exactly one connector type (PostgreSQL). We're paying it to do {prefix}_{source_schema} remapping that core PG cannot do."

19.3

Replication Slots — 3-Layer Protection Critical

A logical replication slot is a server-side bookmark on the WAL — PG won't recycle WAL the slot still needs. Lose a slot and Fivetran has to do a full historical re-sync, which is hours of analytics downtime per environment. We have 5 slots × 3 envs = 15 critical bookmarks. Three independent layers protect them.

Why Slot Loss Hurts So Much

Lost slot ⇒ full re-sync. When Fivetran's bookmark disappears, the agent has no LSN to resume from. The connector flips to historical=true and re-reads every row in the source table. For a busy cfm_core_{env} with hundreds of millions of rows, that is 4-6 hours of catch-up per connector. Power BI dashboards stay stale until the connector returns to incremental mode. This is the most expensive failure mode in the pipeline.

The Slot States You Need to Recognise

wal_statusBound byMeaningRisk
reservedmax_wal_sizeHealthy. WAL within normal recycle window.None
extendedmax_slot_wal_keep_sizeWAL grew past max_wal_size to keep slot alive.Disk pressure. Consumer is lagging.
unreserved(about to disappear)WAL exceeds even max_slot_wal_keep_size.Critical — slot about to be dropped.
lost(gone)WAL deleted, slot dead.Fatal — full historical re-sync needed.

Layer 1 — PostgreSQL WAL Parameters

postgres-cluster Patroni config 
parameters:
  wal_level: logical                  # enables WAL decoding (Fivetran reads row-level changes)
  max_replication_slots: 20           # capacity (we use 5 of 20)
  max_wal_senders: 20                 # concurrent WAL streaming connections
  hot_standby_feedback: on            # replica reports needed LSN -> primary won't prematurely cleanup
  max_slot_wal_keep_size: 15GB        # safety cap: if Fivetran lags >15GB, slot is dropped to protect disk

hot_standby_feedback is the subtle one. Without it, the primary doesn't know which LSN the standbys still need; it can vacuum tuples that the standby's slot copy still references, causing query conflicts on the standby. With it on, the standby's slot LSNs flow upstream and the primary respects them.

Layer 2 — Patroni useSlots: true

Patroni-managed slot copy 
patroni:
  useSlots: true            # slot LSN state mirrored to standbys before promotion
  slots:
    fivetran_core_slot:
      type: logical
      database: cfm_core_ci
      plugin: pgoutput
    fivetran_forms_slot:
      type: logical
      database: cfm_forms_ci
      plugin: pgoutput
    # ... 3 more slots

Without useSlots, a Patroni failover would promote a standby that has no slot at all — Fivetran's next call to read from fivetran_core_slot on the new primary would fail with replication slot does not exist, triggering a full re-sync. With it, the standby holds a copy of the slot's LSN at all times; on promotion, the slot is already there.

Layer 3 — Slot Manager CronJob (every 5 min)

A K8s CronJob in each 207804-postgres-{env} namespace. Schedule: */5 * * * *. Logic:

1
Connect to the postgres-cluster primary
Reads PGO secret for auditdevops superuser.
2
Query pg_replication_slots for each expected slot
Compares against the 5-slot manifest baked into the job ConfigMap.
3
If missing or wal_status=lost → drop, recreate, re-sync
Calls Fivetran API POST /v1/connectors/{id}/resync using credentials from a207804-ue1-op-fivetran-resync-api-preprod-sm.
4
If healthy → log OK, no action
Idempotent — running it does no harm if everything is fine.
Q: "How do you protect against slot loss when Patroni fails over?"
"Three independent layers, designed so any one of them surviving keeps the connector running.

Layer 1 is the PG parameters themselves — wal_level=logical obviously, but the load-bearing one is hot_standby_feedback=on so the standby's slot LSN flows back to the primary, plus max_slot_wal_keep_size=15GB as the disk safety cap.

Layer 2 is Patroni useSlots: true. Without that flag, only the primary holds the slot. When Patroni promotes a standby, the new primary has no slot — Fivetran's next read fails. With useSlots, Patroni continuously mirrors slot state to standbys, so on promotion the slot is already there at the right LSN. This is the layer most people miss.

Layer 3 is a Slot Manager CronJob every 5 minutes. It runs in the 207804-postgres-{env} namespace, queries pg_replication_slots, and if it sees a slot missing or wal_status=lost, it recreates the slot AND calls Fivetran API POST /v1/connectors/{id}/resync. So even if both PG params and Patroni fail us, we self-heal within 5 minutes plus the cost of a re-sync — versus indefinite downtime if nobody noticed."
Q: "What happens if all three layers fail?"
"The connector enters a broken state. The slot is gone, the WAL is gone, Fivetran has no LSN to resume from. Recovery path:
  1. Drop and recreate the slot manually: SELECT pg_drop_replication_slot('fivetran_core_slot'); SELECT pg_create_logical_replication_slot('fivetran_core_slot', 'pgoutput');
  2. Trigger a Fivetran historical re-sync via API or UI. The connector flips to historical=true and re-reads every row in the source tables.
  3. Wait 4-6 hours per connector. During this time the Bronze schema in the DWH is incomplete, dbt downstream is wrong, Power BI shows stale data.
  4. Confirm return to incremental mode once historical phase finishes. Slot lag should drop back to near-zero.
The mitigation is exactly what the three layers prevent. We saw layer-3 in action during the Apr 26 incident: QA hit 46 GB lag (extended), one more day would have been unreserved → lost. The CronJob caught it in time."

How Connector Sees It

From Fivetran's connector config in the cloud UI, it always uses port 5432. The F5 LB accepts on 5432 from the agent and internally translates to the appropriate Istio port (5433 for Demo, 5434 for CI, 5435 for QA). Connector creators don't think about the multiplexing — Fivetran just sees one PG host per env.

Why auditdevops, not fivetran: All connectors authenticate as the auditdevops superuser, with REPLICATION privilege. Password lives in AWS Secrets Manager (a207804-ue1-op-auditdevops-{env}-db-sm) and the PGO-generated K8s secret confirmation-pgdatabase-{env}-pguser-auditdevops. Any future identity hardening would create a dedicated fivetran role per env with the minimal grants — currently we accept the simpler superuser model because the agent is on a private network.
Q: "Why not just use one LoadBalancer Service per environment instead of one shared Istio gateway with port multiplexing?"
"Two reasons.

(1) F5 LB IPs are scarce. Each type: LoadBalancer service consumes a TR-allocated F5 frontend IP, and on Outposts those IPs come from a bounded on-prem CIDR. Provisioning a new one is a network team ticket — Balaji and Ambigi sign off, takes a week or two. Port multiplexing on the existing private-ingressgateway means the network team allocates one IP per env once and we can add more services behind it without new tickets.

(2) Istio AuthorizationPolicy works uniformly. Because all source PG traffic flows through the Istio ingress gateway, we get one place to apply mTLS, IP allow-lists, and per-port AuthorizationPolicy. With separate LoadBalancer services per env, each one would need its own NetworkPolicy and AWS SG rules — duplicated config that drifts.

The trade-off is one shared gateway = a blast radius if it breaks. Mitigated by 3 pod replicas behind an internal NLB and Istio's standard health checks. We'd revisit this if the gateway became a hot-spot."
19.5

Apr 26-27 2026 Incident — 21h Outage (STAR) War Story

This is the war story for behavioural rounds. Use it for "tell me about a time monitoring failed you" or "walk me through a time you hardened a fragile system." It is recent (April 2026), measurable (21 hours, 15 connectors, 46 GB slot lag), and the fix is architecturally interesting (3 layers of restart durability).

Situation

Apr 26, 06:18 UTC. AWS issued an Outpost host-level event. The Fivetran NEW agent EC2 (i-0fc07642a57b87826, r5.4xlarge) rebooted. All 6 Fivetran containers (controller + 5 donkeys) exited with Exit(255) and never auto-recovered. All 15 connectors (CI/Demo/QA) flipped to broken state. Slot lag started accumulating. Datadog alerting was silently broken (expired API key). Nobody noticed for 21 hours.

Task

Two things needed to happen, fast:

  1. Restore replication before slots reach lost state. QA was already at 46 GB lag — one more day and the slot would be deleted, forcing a 4-6 hour full historical re-sync per connector.
  2. Eliminate the failure mode permanently. Why did a routine EC2 reboot kill replication for 21 hours? Why didn't Datadog alert? Find every gap and close it.

Action — What I Did

Phase 1 — Discovery (05:00 UTC, Apr 27)

1
Confirm containers dead via SSM
docker ps -a → 6 containers in Exited (255) for 21 hours. The bare docker run install had no --restart policy, no systemd unit, no anything that would re-spawn them.
2
Start the controller manually
sudo docker start controller-73b24d3a-... → controller up in seconds, 15 donkey containers spawned automatically by the controller via /var/run/docker.sock. PING(READY) heartbeats appeared in logs at ~40ms RTT to Fivetran cloud. Slot lag started draining within minutes.
3
Verify slot lag drainage
Demo: 15 GB → 800 MB. QA: 46 GB → 176 MB. CI: 1.2 GB → 1.3 GB (steady-state, normal). All slots back to wal_status=reserved.

Phase 2a — Belt-and-Braces Docker Restart Policy

Set restart policy on the running container — no restart needed 
sudo docker update --restart unless-stopped controller-73b24d3a-2f22-4f98-b989-b2763f4c7698

This already covers Docker daemon restarts and OS reboots — but it was just a band-aid. The real fix was Phase 2b.

Phase 2b — systemd + hdagent.sh (mirror the OLD agent)

I discovered we had two Fivetran EC2 instances: the OLD agent (i-05da72dff91d23fac) which had survived the reboot fine, and the NEW agent (i-0fc07642a57b87826) which hadn't. The OLD one used the standard Fivetran install with a systemd unit and the official hdagent.sh wrapper. I copied that pattern verbatim onto the NEW agent.

/etc/systemd/system/fivetran-agent.service 
[Unit]
Description=Fivetran Hybrid Agent Service
After=docker.service                # don't start until docker is up
Requires=docker.service             # if docker stops, stop me too

[Service]
Type=simple
User=ec2-user
WorkingDirectory=/home/ec2-user/fivetran
RemainAfterExit=yes                 # script forks into background; stay "active"
ExecStartPre=-/usr/bin/docker stop controller    # idempotent cleanup
ExecStartPre=-/usr/bin/docker rm controller
ExecStart=/home/ec2-user/fivetran/hdagent.sh start
ExecStop=/home/ec2-user/fivetran/hdagent.sh stop
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
drop-in /etc/systemd/system/fivetran-agent.service.d/proxy.conf 
[Service]
Environment="HTTPS_PROXY=http://webproxy.df1.corp.services:80"
Environment="NO_PROXY=169.254.169.254,localhost,127.0.0.1,10.0.0.0/8"

Then: systemctl daemon-reload && systemctl enable --now fivetran-agent.service. Verified is-active, is-enabled, controller logs showing PING(READY) heartbeats.

Result — 3 Layers of Restart Durability

LayerMechanismCatches
Layer 1 Container --restart on-failure:3 (set by hdagent.sh) Controller process crashes (exit != 0). Up to 3 retries, Docker daemon handles it.
Layer 2 systemd Restart=on-failure RestartSec=10 hdagent.sh script itself crashes. systemd retries the unit.
Layer 3 systemd enable + Docker daemon enabled at boot OS reboot (the Apr 26 case). Docker comes up first, systemd starts fivetran-agent.service, ExecStartPre cleans stale state, ExecStart spawns fresh controller.

Compounding Failure — Why Datadog Was Silent

Two-fault chain. Fault 1: agent had no restart policy. Fault 2: Datadog API key on the NEW agent was expired (returning 403), and the OLD agent had its DD API key hardcoded in plaintext at /etc/datadog-agent/datadog.yaml. Neither agent was actually delivering metrics. We had no monitor on slot lag, no monitor on connector state, no monitor on the agent EC2 itself. The Outpost reboot was invisible until somebody noticed Power BI dashboards were stale.

Follow-up Work Items (Filed as ADO P1/P2)

  • P1: Bake Phase 2b artifacts (hdagent.sh, systemd unit, proxy drop-in) into Susham's monthly Golden AMI rotation. Future EC2 launches come up Fivetran-ready with one boot.
  • P1: Migrate Datadog API key from plaintext datadog.yaml to AWS Secrets Manager injection at launch via user-data. Rotate the existing key.
  • P1: Add 4 Datadog monitors — agent heartbeat, slot wal_status, connector broken state, EC2 host check. The slot wal_status monitor is the highest leverage one.
  • P2: Move Fivetran agent token from disk (conf/config.json) to AWS SM. Inject at launch.
  • P2: Decommission OLD agent EC2 once new pattern stable.
Q: "Tell me about a time when monitoring failed you. What did you learn?"
"April 26 this year. AWS Outposts host event rebooted the Fivetran agent EC2. All 15 connectors went broken. Datadog stayed silent for 21 hours.

What failed: two faults stacked. The agent was installed with bare docker run and no restart policy, so containers stayed dead after the reboot. And the Datadog API key on that EC2 had expired — agent metrics weren't getting through. Either fault alone would have been recoverable. Together, we had a silent 21-hour outage where slot lag climbed to 46 GB on QA, one day from lost state and a 4-6 hour historical re-sync per connector.

What I did the next morning: restored the agent first (docker start + --restart unless-stopped as a band-aid), drained the slots, then mirrored the OLD agent's standard install — Fivetran's hdagent.sh wrapper plus a fivetran-agent.service systemd unit with Restart=on-failure. Three independent restart layers now: container --restart on-failure:3, systemd Restart=on-failure, and systemd enable at boot. Apr 26 mode is now caught by layer 3.

What I learned: two things. First, monitoring you don't verify isn't monitoring — we now have a synthetic monitor that pings the agent heartbeat from outside Datadog so an expired DD key can't hide the silence. Second, every install pattern needs to survive a reboot from day one. The OLD agent was installed correctly years ago and just kept working. The NEW agent was installed quickly during a migration and the systemd step got skipped. Now we bake the artifacts into the Golden AMI so the install can't drift again.

Honest reflection: the deeper failure was process — we never ran a 'pull-the-power-cord' test on this agent before declaring it production-ready. If we had, we'd have caught both faults in five minutes."

Why Citus Doesn't Help OLTP

Citus shards a table by a distribution column across worker nodes. A query then goes to the coordinator, which parses it, picks the relevant shards, dispatches sub-queries to workers, and merges results. For a query like SELECT * FROM users WHERE id = 42, that coordinator round-trip adds latency — the lookup hits one row and you're paying for parallelism you don't use. OLTP queries hit 1-10 rows by primary key. Sharding makes them slower, not faster. Even at 100 GB, the OLTP cluster is best served by vertical scaling and good indexes.

Q: "Why is Citus on the DWH but not on the OLTP cluster?"
"Query pattern, not data volume.

OLTP queries are point lookups — SELECT * FROM users WHERE id=42 or a single-row INSERT. Citus would route through a coordinator, parse the query, dispatch to the relevant shard, and merge — for a query that hits one row by primary key. That coordinator round-trip is pure overhead. We'd be slower than a vanilla PG instance with a good index.

DWH queries are aggregations — SELECT region, SUM(amount) FROM billing_facts WHERE date BETWEEN ... GROUP BY region over 100M+ rows. That's exactly the workload Citus parallelises: each worker scans its shards, returns partial sums, coordinator merges. We get 2-3x speedup on the queries that actually matter, and columnar storage on top of that compresses analytics tables and reads only the columns the query touches.

The other reason is scaling strategy. OLTP scales vertically (bigger instance, NVMe, more RAM for buffer cache). Analytics scales horizontally (more workers, more shards). Citus only matches the second pattern. Putting it on OLTP would be solving a problem we don't have while introducing one we do."

Q: "Why webhook-triggered dbt instead of just cron'ing it every 30 minutes?"
"Three reasons.

(1) Avoid wasted work. Cron'd dbt runs even when Fivetran hasn't synced anything new — that's 107 model executions and a stack of dbt test assertions burning compute and DWH connections for no data change. Event-driven only fires when there's actual new Bronze data to transform.

(2) Avoid race conditions. If cron fires at minute 30 but Fivetran sync started at minute 28 and finishes at minute 32, dbt reads partial data. Webhook on sync complete guarantees Bronze is consistent before dbt reads it.

(3) Per-environment fan-out. Five connectors per env, each fires its own webhook. HookDeck routes fivetran_sync_end_ci to the CI workflow and fivetran_sync_end_qa to QA. Cron would either run all envs together (waste) or need separate cron schedules per env (drift).

The HookDeck layer matters too — it's a durable buffer. If GitHub is unavailable for 30 minutes, HookDeck queues the events and replays them. A simple direct webhook from Fivetran to GitHub would lose events during GitHub outages."

Cleaner isolation, no shared docker.sock privilege concern.

  • Trade-off: EFS data goes to Region over service link, not local Outpost storage. Fivetran syncs are throughput-oriented not latency-sensitive, so acceptable.
  • Risk: service link outage → EFS unavailable → syncs stop. Mitigation: keep EC2 agent as fallback during a 1-2 week parallel run.
Stable egress IP for Fivetran cloud allowlist. Pod IPs are ephemeral. State on PVC, not pod. Replication checkpoints live in /data. Q: "What are the operational risks of moving Fivetran from EC2 to K8s?"
"Three real ones, plus the wishful-thinking ones to watch out for.

Risk 1: RWX storage. Fivetran K8s mode requires ReadWriteMany. Our default StorageClass is EBS gp2 — RWO only. If the controller pod and a donkey Job land on different nodes, EBS can't mount on both. Plan is EFS via service link, but that adds a service link dependency to the data pipeline. If service link degrades, Fivetran syncs stop.

Risk 2: Egress IP not stable. The Fivetran cloud-side may have an IP allowlist for the agent. Pod IPs are ephemeral; on restart you get a different IP. We'd need an egress NAT gateway with a static IP attached to the namespace, or accept potential auth churn. Verify with Fivetran admin first — this is a hard blocker if true.

Risk 3: Pod restart != container restart. On EC2, the controller container has restart policy on-failure:3 and persists state on EBS. On K8s, a pod restart is more aggressive — you lose ephemeral process state, the donkey Jobs in flight may need to be re-spawned, and if the PVC re-attach is slow you're sitting in ContainerCreating while slot lag accumulates.

The wishful-thinking risks: 'we'll just deploy it and trust the Helm chart' — Fivetran's K8s mode is ~1 year old (March 2025), so it's not battle-hardened. Mitigation is the parallel run. Run K8s agent in CI for 1-2 weeks before touching Demo/QA, keep the EC2 fallback ready. Don't decommission the EC2 until we've seen a Patroni failover, a service link blip, and a Datadog incident on the new path.

Honest framing for the interviewer: $700/mo is real, but it's not life-changing. The bigger driver is Datadog observability and ArgoCD GitOps consistency — that's why I'm pushing this. The cost saving is the headline; the real value is consistency with the rest of the platform."

Mock Interview Bank

Six high-probability questions for a Senior DevOps round at Cognizant/Westpac. Each answer is what you should aim to say in 90-120 seconds.

Q1: "Walk me through your data pipeline end-to-end."
"Six components in sequence. (1) postgres-cluster is the OLTP source — PGO-managed PG 16 with Patroni HA, wal_level=logical, 5 replication slots per env. (2) Fivetran EC2 agent on Outpost runs a Docker controller and per-sync donkey workers; it reads WAL via pgoutput and pushes data to (3) Fivetran Cloud SaaS over outbound HTTPS through the corp proxy. Cloud writes the rows to (4) postgres-dwh, a Citus 13 cluster, into Bronze schemas like _raw_cfm_core_public. The naming is the load-bearing trick — Fivetran prefixes the source schema so 5 source DBs don't collide on shared public table names.

On sync complete, Fivetran fires a webhook through (5) HookDeck to a GitHub Actions repository_dispatch, which runs dbt on a self-hosted EC2 runner. dbt deps, seed, run, test — 107 SQL models that build Silver (analytics_base, analytics) and Gold (analytics_presentation) layers. Finally (6) the Power BI on-prem gateway on a Windows EC2 reads only Gold and tunnels results through to Power BI Cloud for dashboard rendering.

15 connectors total — 5 source DBs × 3 envs (CI/Demo/QA). End-to-end latency from source commit to Power BI dashboard is roughly the Fivetran sync interval (15 min) plus dbt run time (5-10 min)."
Q2: "What broke on April 26?"
"AWS Outposts host event rebooted the Fivetran agent EC2. All 15 connectors went broken for 21 hours.

Two faults stacked. The agent was installed with bare docker run and no --restart policy, so containers stayed in Exit(255) after the reboot. Datadog was silent because the API key on that instance had expired and was returning 403. Either fault alone we'd have caught — together they made it invisible.

Recovery: I started the controller manually via SSM, applied --restart unless-stopped as a band-aid, then mirrored the OLD agent's standard install — Fivetran's hdagent.sh wrapper plus a fivetran-agent.service systemd unit. Result: 3 layers of restart durability — container --restart on-failure:3, systemd Restart=on-failure, and systemd enable at boot. The Apr 26 mode is now Layer 3.

Slot lag: QA hit 46 GB (wal_status=extended), one day from lost. Drained to 176 MB within an hour of the controller restart.

Lesson: filed P1 ADO items for baking the systemd artifacts into the Golden AMI, moving the DD API key to AWS SM injection, and adding 4 monitors (heartbeat, slot wal_status, connector state, EC2 host check). The slot wal_status monitor would have paged me at minute 15, not hour 21."
Q3: "Why Fivetran when Postgres has native replication?"
"One specific reason: schema remapping. Native PG logical replication matches publisher to subscriber by fully qualified name — schema.table. The PG docs spell it out, unchanged from PG 10 through PG 18: replication to differently-named tables on the subscriber is not supported.

We have 5 source databases each with a public.orleansstorage table, and they all need to land in one destination database for dbt cross-source joins. With native replication, three of those writes collide on the same public.orleansstorage primary key. There's no MAP SCHEMA param, no extension hook in core PG.

Fivetran solves it by reading the WAL itself via pgoutput and writing with a configurable schema prefix — _raw_cfm_core_public, _raw_cfm_record_public, etc. Five collision-free Bronze schemas, dbt knows which source produced which row.

We considered the alternatives — pglogical isn't supported by our Crunchy operator, Debezium+Kafka is an entire CDC platform for 5 connectors, separate destination DBs break dbt's cross-DB join model. Fivetran at ~$1,500/mo plus the EC2 is the cheapest path. We're not paying for the 600+ connector ecosystem; we're paying for one PG limitation it solves."
Q4: "How do logical replication slots survive a Patroni failover?"
"Three layers, designed so any one layer keeps the connector alive.

Layer 1 is PG params — wal_level=logical, hot_standby_feedback=on so the standby's slot LSN flows back to the primary, max_slot_wal_keep_size=15GB as a safety cap before the slot gets dropped on disk pressure.

Layer 2 is Patroni useSlots: true, the load-bearing one most people miss. Without it, the slot only exists on the primary. When Patroni promotes a standby, the new primary has no slot — Fivetran's next read fails with replication slot does not exist. With useSlots: true, Patroni continuously mirrors slot LSN state to standbys, so on promotion the slot is already there at the right LSN and Fivetran resumes without gap.

Layer 3 is a Slot Manager CronJob every 5 minutes in each 207804-postgres-{env} namespace. It connects to the primary, queries pg_replication_slots, and if any slot is missing or has wal_status=lost, it recreates the slot AND calls Fivetran API POST /v1/connectors/{id}/resync. So even if both Layer 1 and Layer 2 fail us, we self-heal within 5 minutes.

The cost of slot loss is real — 4-6 hours of historical re-sync per connector. The 3 layers exist because that cost is high enough to justify the engineering."
Q5: "Why is the Fivetran agent on EC2 not Kubernetes? What's the plan?"
"Historical reason — Fivetran only officially supported K8s as of March 2025 (hybrid-deployment-agent Helm chart v0.19.0). The agent was deployed before that, and it works, so migration was deferred.

The case for migrating now: ~$730/mo wasted on r5.4xlarge for a 1.6 GiB workload (1.3% utilisation), no Datadog (Apr 26 silent outage), outside ArgoCD GitOps, and an unnecessary F5 LB → TGW → NLB hairpin to reach the DWH when a pod could use ClusterIP directly.

The plan: a Helm chart in charts/fivetran-agent/ deploying a Deployment + ExternalSecret for the token + EFS-backed PVC into 207804-datawarehouse-preprod. K8s mode replaces Docker-in-Docker with K8s Jobs spawned by the controller — cleaner isolation. Migration is three phases: validate EFS prerequisite, deploy in parallel to existing EC2 in CI for 1-2 weeks, cut over per-env in the Fivetran cloud UI, then decommission the EC2 plus the Lambda AMI rotation.

Real risks: RWX storage requirement (EBS gp2 won't work, EFS adds a service link dependency), pod IP not stable for any Fivetran cloud-side allowlist, K8s mode is only ~1 year old. Mitigation is the parallel run plus keeping the EC2 as a hot fallback during cutover.

Honest framing: $700/mo savings is the headline, but the bigger value is platform consistency — every other workload is in EKS managed by ArgoCD with Datadog DaemonSet observability. Eliminating the only EC2 outlier is what really matters."
Q6: "Talk me through the Medallion architecture in your dbt setup."
"Three layers in cfm_datalake_{env}, each with a different contract.

Bronze is what Fivetran writes — _raw_cfm_core_public, _raw_cfm_forms_public, etc. Raw, unmodified replica of source PG with Fivetran's metadata columns added (_fivetran_synced, _fivetran_deleted). Apps never read Bronze; dbt is the only consumer. The contract is simply 'this is the source-of-truth replica.'

Silver is dbt's working layer — analytics_base has 95 staging views (type cast, column rename, basic deduplication), analytics has intermediate tables that apply business logic like Orleans JSON unpacking into event records, analytics_references joins reference data. Mostly views with a few materialised tables where rebuilds are expensive. The contract is 'cleaned and joined data, but schema can change between dbt releases.'

Gold is the stable Power BI surface — analytics_presentation with 5-6 mart tables (facts + dimensions + report views) and analytics_system for data quality (e.g., fct_invalid_orleans_grains). Star-schema shaped, pre-aggregated for dashboard performance. Power BI reads only Gold. The contract is 'frozen column names, breaking changes go through a release process.'

Why the layering matters: Bronze is replication concern — Fivetran owns it. Silver is data engineering concern — dbt models can be refactored freely. Gold is product concern — Power BI dashboards depend on it, so changes are deliberate. Each layer has a different change cadence and a different blast radius. dbt orchestrates the whole pipeline, but the layer separation means a Silver refactor doesn't break Power BI and a Bronze re-sync doesn't break Silver business logic."
