Transit Gateway

Q What is a Transit Gateway and why would Westpac use it over VPC Peering?
YOUR ANSWER (strong)

VPC peering is not a router — it's a point-to-point private link. AWS explicitly blocks transitive routing — traffic arriving on one peering connection cannot exit via another. Transit Gateway is a central hub-and-spoke router. Instead of n(n-1)/2 VPC peering connections, every VPC connects once to TGW. For Westpac it enables centralised traffic inspection, connects on-prem via Direct Connect, and scales cleanly across multiple AWS accounts.

Insight: You also correctly calculated n(n-1)/2 = 45 connections for 10 VPCs (not 50 — caught your own mistake). This shows attention to detail interviewers notice.
Q Why can't VPC A talk to VPC C through VPC B in a peering setup?
YOUR ANSWER (strong)

VPC Peering is not a router. AWS explicitly blocks transitive routing — traffic arriving on one peering connection cannot exit via another peering connection. Even if B is peered with both A and C, the A-B peering connection cannot carry traffic destined for C. They're isolated tunnels, not a network fabric. TGW fixes this because it IS a real router with its own route tables.

Key word to use: "transitive" — that's the exact term interviewers use.

VPC & Subnets

Q What is a route table and what does it do?
YOUR ANSWER (strong)

A route table is a set of rules that tells traffic where to go. It is attached at the subnet level — each subnet has its own route table, which is why public and private subnets behave differently. For example, the public subnet has 0.0.0.0/0 → Internet Gateway so traffic can reach the internet. The private subnet has 0.0.0.0/0 → NAT Gateway so pods can go out but nothing comes in directly.

Insight: You connected this correctly — route tables at subnet level explain why subnets behave differently. Good architecture thinking.
Q Why does NAT Gateway sit in the PUBLIC subnet and not the private subnet?
YOUR ANSWER (strong)

Because NAT Gateway needs to reach the internet and it has to sit with the public subnet. It needs a public IP to talk to the internet. If it sat in the private subnet, it would have no path to the internet itself.

Flow to remember: EKS Pod → NAT Gateway (public subnet, has public IP) → Internet Gateway → Internet.
Q Why do we reference Security Group IDs in rules instead of IP addresses?
YOUR ANSWER (after prompt)

Because EKS nodes scale constantly — new node gets a new IP. If RDS Security Group allowed a specific IP, the new node would be blocked. By referencing the Security Group ID, any instance that belongs to that SG is automatically allowed regardless of its IP address.

Analogy: IP = allowing a specific person by name. SG = allowing anyone with a specific badge/role. Staff changes, badge rule stays.

VPC Endpoints

Q EKS pods must have zero internet egress but still need to pull images from ECR and send logs to CloudWatch. How?
YOUR ANSWER (needed prompting initially)

Use VPC Endpoints. Interface Endpoints create a private IP inside your subnet for each AWS service. ECR, CloudWatch, Secrets Manager, STS all get private endpoints. Traffic goes Pod → private IP → AWS service. Never leaves the AWS network. No NAT Gateway needed for AWS service calls.

Remember: Gateway Endpoint = route table entry (S3, DynamoDB, free). Interface Endpoint = private IP/ENI in your subnet (everything else, costs ~$0.01/hr per AZ).

Security Groups vs NACLs

Q What is the difference between stateful and stateless in networking?
YOUR ANSWER (after explanation)

Security Groups are stateful — when you allow inbound traffic, the return traffic is automatically allowed. You don't need a rule for the response. NACLs are stateless — they have no memory. Every packet is evaluated independently. You must explicitly allow both the inbound request AND the outbound response.

NACL gotcha: If you block port 443 inbound on a NACL, you must also block the outbound response — it's stateless so it won't do it automatically.
Q How do you set up Security Groups for ALB → EKS → RDS?
YOUR ANSWER (strong)

Three Security Groups, each only accepting from the layer above: ALB SG allows 443 from internet. EKS Node SG allows 8080 only from ALB SG — not from internet directly. RDS SG allows 5432 only from EKS Node SG. Defence in depth — each layer is a separate barrier. A compromised internet request cannot jump straight to RDS.

Key principle: Each layer only talks to the adjacent layer. This is defence in depth — same concept as your 7-layer security in Section 11.

Direct Connect & BGP

Q Why wouldn't a bank just use VPN to connect on-premises to AWS?
YOUR ANSWER (strong)

VPN is encrypted but goes over the public internet — unpredictable latency, no SLA, shared with everyone. Direct Connect is a dedicated private fibre connection — it never touches the public internet, consistent latency, up to 10Gbps, with an SLA. For a bank with sensitive financial data, DX is preferred. We run VPN in parallel as backup — DX is preferred via BGP, VPN activates automatically if DX fails.

Nuanced point that impresses: DX is NOT encrypted by default — but the private dedicated line is more secure than encrypted public internet for a bank. Add MACsec for L2 encryption on DX if needed.
Q What is BGP?
YOUR ANSWER (after explanation)

BGP is the routing protocol that exchanges network reachability information between on-premises and AWS. On-prem tells AWS which CIDRs it owns, AWS tells on-prem which VPC CIDRs it owns. In a DX + VPN setup, DX is configured with a shorter AS path so BGP prefers it. When DX goes down, BGP withdraws the DX routes and traffic automatically fails over to VPN within 30-90 seconds.

One liner: "BGP is the routing protocol that makes automatic failover between Direct Connect and VPN work."

EKS VPC CNI

Q What makes EKS pod networking special compared to other Kubernetes CNIs?
YOUR ANSWER (strong)

AWS VPC CNI gives every pod a real VPC IP address — not an overlay network IP. This means pods are first-class VPC citizens. Security Groups work on pods directly, VPC Endpoints work, route tables work. With overlay IPs, AWS infrastructure doesn't understand the pod IPs and traffic gets dropped.

The tradeoff: Real VPC IPs = IP exhaustion risk. 10 nodes × 30 pods = 300 IPs consumed. Always use /21 or larger for EKS subnets. Fix with secondary CIDR (100.64.0.0/16) if running out.
Q You have a /24 subnet for EKS. Is it enough?
YOUR ANSWER (strong)

No. /24 = 256 IPs, minus 5 AWS reserved = 251 usable. With autoscaling, 10 nodes at 30 pods each = 300 IPs. New pods will fail to schedule with "insufficient IPs" error. Use /21 minimum (2048 IPs). Fix existing exhaustion by adding secondary CIDR — zero downtime.

Real incident pattern: Cluster looks healthy, nodes are fine, but pods won't start. Always check subnet IP availability first in this scenario.

Route 53 & DNS

Q What is split-horizon DNS and why does Westpac need it?
YOUR ANSWER (strong — got it immediately)

Same domain resolves differently inside vs outside the VPC. api.westpac.com.au resolves to the internal ALB IP inside the VPC, and the public ALB IP externally. This avoids hairpin routing — without it, internal traffic would go out to the internet and come back in, which is slower, more expensive, and exposes internal traffic to the public internet unnecessarily.

This was your strongest answer of the session — you got it without any hints.
Q How do you solve DNS in a hybrid environment where on-prem needs to resolve AWS hostnames?
YOUR ANSWER (strong)

Route 53 Resolver endpoints in both directions. Inbound endpoint — on-prem DNS forwards *.westpac.internal queries to a private IP in the VPC, Route 53 resolves it. Outbound endpoint — AWS forwards *.corp.westpac queries to on-prem DNS. Pods can resolve mainframe hostnames, on-prem machines can resolve RDS endpoints.

Analogy: Inbound = reception accepting calls from outside. Outbound = staff calling the external phone book.

Network Firewall

Q We have Security Groups and NACLs. Why do we need Network Firewall?
YOUR ANSWER (strong)

Security Groups and NACLs can block ports and IPs but cannot block domain names. A compromised pod may try to reach an attacker's server on port 443 — SG allows port 443 for legitimate traffic, so it can't block this. Network Firewall operates at Layer 7 and understands domain names via TLS SNI. We use an allowlist — only *.amazonaws.com and *.datadog.com are allowed. Everything else is blocked by default. It's deployed in a centralised inspection VPC so all accounts' egress traffic passes through one policy.

Key insight you provided: Allowlist is better than blocklist — millions of malicious domains exist, you can't block them all. Easier to define the small set of approved destinations.

Terraform

Q Where does Terraform track what infrastructure it has deployed?
YOUR ANSWER (strong — reasoned independently)

Terraform manages its own state file (terraform.tfstate). In a team environment this must be stored remotely — in an S3 bucket so everyone can access it, with a DynamoDB table for locking to prevent two people running apply simultaneously.

You reasoned this independently — I asked "what AWS service stores files everyone can access?" and you said S3. Then "what provides locking?" and you said DynamoDB. That's the right way to think through architecture.

Q What are the steps in a Terraform CI/CD pipeline?
YOUR ANSWER (strong — got it immediately)

terraform fmt (format check), terraform validate (syntax check), terraform plan (show changes, post to PR as comment), human review and approval, terraform apply (only on merge to main).
