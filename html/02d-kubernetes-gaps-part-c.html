G11

Probes — Liveness, Readiness, Startup High Priority

Every interviewer asks this. You configure probes on every service in your platform — you need to be able to recite the three probe types, what each one controls, and when they fire from memory.

🏥 Analogy Three different health checks at a hospital: Startup probe = "is the patient checked in and stable?" (runs once after admission). Readiness probe = "are they ready to see visitors?" (runs continuously, blocks visitors when sick). Liveness probe = "are they still alive?" (runs continuously, triggers resuscitation if flatline). Getting these confused is the single most common mistake in pod stability.

The Three Probe Types

ProbeQuestion it answersFailure actionWhen it runs
startupProbe Has the app finished booting? Kill container, restart pod From container start until first success. Once it succeeds, it stops running forever. Liveness/Readiness are suspended until startup succeeds.
readinessProbe Can the app take traffic right now? Remove pod from Service endpoints (no restart) Continuously after startup. If it starts failing, kube-proxy removes the pod from Service rotation. Pod keeps running.
livenessProbe Is the app still alive (not deadlocked)? Kill container, kubelet restarts it Continuously after startup. If it fails failureThreshold times, kubelet kills the container and restarts it.

Your Actual Probe Config — pricing-be (CI)

This is the real pattern used across your 35+ .NET services. Grounded in environments/ci/pricing-be/values.yaml:

Real Probe Values From Your Helm Chart 
livenessProbe:
  httpGet:
    path: /api/confirmation/pricing/health/live    # shallow check — is the process responsive?
    port: 8080
    scheme: HTTP
  initialDelaySeconds: 0                           # startupProbe already gated this
  periodSeconds: 10                                # check every 10s
  timeoutSeconds: 1                                # fail if no response in 1s
  failureThreshold: 3                              # 3 consecutive fails → restart (30s total)

readinessProbe:
  httpGet:
    path: /api/confirmation/pricing/health/ready   # deep check — can I serve traffic? (DB, RabbitMQ up?)
    port: 8080
    scheme: HTTP
  initialDelaySeconds: 0
  periodSeconds: 10
  successThreshold: 3                              # need 3 in a row to be Ready (avoids flapping)

startupProbe:
  httpGet:
    path: /api/confirmation/pricing/health/ready
    port: 8080
    scheme: HTTP
  initialDelaySeconds: 60                          # give .NET 60s to boot before even starting to check
  periodSeconds: 10
  failureThreshold: 30                             # 60 + 10*30 = 360s max boot window
Key detail from your config: The startup probe gives a slow-starting .NET service up to 6 minutes to boot (initialDelay 60s + period 10s × threshold 30 = 360s). During that window, liveness and readiness probes don't run. This is why startupProbe exists — without it, a slow-booting app would be killed by liveness before it ever finished starting.

Two Endpoints — Why /live ≠ /ready

EndpointChecksWhy separate?
/health/live Process is running, HTTP server responds, no deadlock If this fails → restart the container. Shallow check, no dependency calls.
/health/ready Process + DB reachable + RabbitMQ reachable + cache populated If this fails → stop sending traffic, but do not restart. The pod is healthy; its dependency is down. Restarting won't help.
Critical anti-pattern — never do this: Pointing livenessProbe at /health/ready. If PostgreSQL has a brief hiccup, readiness fails (fine), but liveness also fails → kubelet restarts every pod → restart storm → now you have an incident. Readiness should check dependencies. Liveness should only check "am I deadlocked?" (always return 200 unless the process itself is broken).

Probe Types — httpGet vs tcpSocket vs exec vs grpc

httpGet
HTTP endpoint
Most common. Any 200-399 response = success. Your .NET services use this via ASP.NET Core health checks middleware.
tcpSocket
TCP port open
Just checks if the port accepts a TCP connection. Useful for services without HTTP endpoints (PostgreSQL, RabbitMQ). Doesn't verify the app actually works — just that it's listening.
exec
Run a command in container
Runs a shell command; exit 0 = success. Used for custom scripts, like pg_isready for Postgres or rabbitmq-diagnostics ping for RabbitMQ.
grpc
gRPC health check (1.24+)
Native gRPC health protocol. No need to expose HTTP on gRPC-only services. Requires the app to implement grpc.health.v1.Health.
Q: "A pod keeps restarting — CrashLoopBackOff. How do probes fit in the diagnosis?"
"Probes are usually step 3 or 4 in my checklist:
  1. kubectl describe pod → look at Events and Last State. If Reason: OOMKilled → memory limit issue, not a probe issue.
  2. If Reason: Error with exit code 137 → killed by kubelet. Could be liveness probe failing OR OOM. Check events for Liveness probe failed.
  3. Liveness probe failing: app might be deadlocked, OR the probe threshold is too aggressive, OR startup is too slow (should have startupProbe instead).
  4. Readiness failing but liveness passing: pod runs but never Ready. Usually a dependency issue — DB/RabbitMQ not reachable, or secret not mounted yet. Check kubectl logs for connection errors.
  5. Classic trap: same endpoint for liveness and readiness. A dependency hiccup cascades into pod restarts. Fix is to split them: liveness = shallow, readiness = deep.
In our platform, we had an incident where a service's liveness probe pointed at /health/ready. When PostgreSQL connection pool exhausted during peak, readiness failed → also liveness failed → kubelet restarted every replica → thundering herd on DB → made it worse. Fix was separating live/ready endpoints."

Pod Security Standards & securityContext High Priority Banking Sector

Banking and financial interviewers probe this hard. You already implement all of it — you just need the vocabulary. Pod Security Policy (PSP) was removed in Kubernetes 1.25; the replacement is Pod Security Admission (PSA) with three predefined levels.

The Three Pod Security Standards (PSS)

LevelWhat it enforcesTypical use
privileged No restrictions. Anything goes — hostNetwork, hostPID, privileged containers, root user. System-level workloads (CNI pods, kube-proxy, node agents like Datadog). Your DaemonSets run here.
baseline Prevents the most obvious escalations: no privileged containers, no hostNetwork, no hostPID, no hostPath (mostly). Middle ground. Most apps can run here without code changes.
restricted Hardened: must runAsNonRoot, drop ALL capabilities, no privilege escalation, readOnlyRootFilesystem strongly recommended, seccompProfile required. Where your Confirmation services run. Banking/finance default.

PSA Enforcement — Label the Namespace

PSA is configured at the namespace level via labels. Three modes per level:

PSA Namespace Labels 
apiVersion: v1
kind: Namespace
metadata:
  name: 207804-confirmation-ci
  labels:
    # Enforce (reject pods that violate):
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: v1.29

    # Audit (log violations to audit log, don't block):
    pod-security.kubernetes.io/audit: restricted

    # Warn (warn to kubectl output, don't block):
    pod-security.kubernetes.io/warn: restricted
Rollout strategy: Start with warn and audit on the new level before enforce. This lets you see violations without breaking workloads. Flip to enforce after the audit log is clean.

Your Actual securityContext — From rollout.yaml

This is the exact pattern in every one of your 35+ service charts. Straight from charts/pricing-be/templates/rollout.yaml:

Container-Level securityContext (Your Platform) 
securityContext:
  allowPrivilegeEscalation: false         # setuid/sudo blocked
  readOnlyRootFilesystem: true            # / is read-only; writes go to emptyDir mounts
  privileged: false                       # no --privileged flag
  capabilities:
    drop: [ALL]                           # drop every Linux capability
  runAsNonRoot: true                      # refuse to start if image user = root
  runAsUser: 14000                        # TR standard UID
  runAsGroup: 14001
  seccompProfile:
    type: RuntimeDefault                  # kernel syscall filter
Pod-Level securityContext 
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 14000
    runAsGroup: 14001
    fsGroup: 14000                        # files in volumes owned by GID 14000 (app can read them)

The 7 Controls You Already Enforce

#1
runAsNonRoot: true
Container refuses to start if the image USER is root (UID 0). First line of defense against container escapes.
#2
runAsUser: 14000
Force a specific non-root UID. TR standard. Must be set in the image or here — doesn't magically make root processes non-root.
#3
allowPrivilegeEscalation: false
Blocks setuid binaries and sudo. No process in the container can gain more privileges than its parent.
#4
readOnlyRootFilesystem: true
Root FS is read-only. Any write must go through an emptyDir/volume mount. Stops malware from modifying binaries.
#5
capabilities.drop: [ALL]
Strips every Linux capability (CAP_NET_RAW, CAP_SYS_ADMIN, etc.). Add back only what's needed via add:. Most apps need none.
#6
seccompProfile: RuntimeDefault
Enables kernel syscall filtering. RuntimeDefault = the runtime's default filter (containerd's blocks ~70 dangerous syscalls).
#7
privileged: false
No --privileged mode. Privileged containers effectively have root on the host — only for system agents, never apps.
readOnlyRootFilesystem gotcha: Many apps write temp files to /tmp or caches to /var/cache. With readOnlyRootFilesystem, these fail. Solution: mount an emptyDir volume at those paths (e.g., Datadog agent needs /var/run/datadog as emptyDir). Your rollout has datadogTemp emptyDir for exactly this reason.
Q: "Walk me through how you harden a pod at Thomson Reuters."
"Seven controls, all at the container and pod level, enforced by Pod Security Admission at the namespace level set to restricted:
  1. runAsNonRoot: true + runAsUser: 14000 — no root. UID 14000 is our TR standard.
  2. allowPrivilegeEscalation: false — block setuid/sudo escalation.
  3. readOnlyRootFilesystem: true — root FS is immutable. Writable paths go through emptyDir mounts.
  4. capabilities: drop [ALL] — no Linux capabilities unless explicitly required.
  5. privileged: false — not privileged. Only node agents get privileged.
  6. seccompProfile: RuntimeDefault — kernel syscall filter active.
  7. fsGroup: 14000 — volume files owned by GID 14000 so the app can read them.
On top of pod-level hardening we have Istio mTLS STRICT (only mTLS traffic allowed), AuthorizationPolicy (service-identity-based allowlists), and ExternalSecrets Operator (no secrets in Helm or Git). The base Docker image is our own a207804-ue1-op-python:3.11-slim from ECR, so we control the supply chain — never Docker Hub in CI."

PodDisruptionBudget — Protecting Availability High Priority

PDB controls how many pods of a service can be voluntarily disrupted at once. Guards against node drains, cluster upgrades, and eviction storms taking your service below a safe replica count.

🚑 Analogy Minimum ER doctors on duty. The hospital can send doctors home for training (voluntary disruption) but only if at least 2 remain at any time (minAvailable: 2). If someone tries to drain the last doctor, the drain waits. PDB is the contract that prevents K8s admin actions from accidentally taking your service to zero.

Voluntary vs Involuntary Disruption

TypeExamplePDB respected?
Voluntary kubectl drain, node upgrade, cluster autoscaler scale-down, manual pod deletion via Eviction API Yes — the Eviction API blocks if PDB would be violated.
Involuntary Node crash, kernel panic, hardware failure, OOM kill, network partition No — PDB cannot save you from a dead node. This is why you still need multiple replicas across nodes.

Your Actual PDB — audit-devops-outpost-storage

Straight from charts/audit-devops-outpost-storage/templates/pdb.yaml in your helm repo:

Your Platform PDB Template 
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: outpost-storage-pdb
  namespace: 207804-confirmation-ci
  annotations:
    argocd.argoproj.io/sync-wave: "3"     # after DestinationRule, before Rollout
spec:
  minAvailable: 1                          # at least 1 pod must stay Ready during drain
  selector:
    matchLabels:
      app: outpost-storage-rollout
      environment: ci

minAvailable vs maxUnavailable

FieldMeaningExample with 3 replicas
minAvailable: 1 Absolute number — at least this many must stay Ready Up to 2 can be disrupted at once
minAvailable: 50% Percentage — rounded UP ceil(50% of 3) = 2 must stay Ready → only 1 can be disrupted
maxUnavailable: 1 Absolute number — at most this many can be disrupted Equivalent to minAvailable: 2 for a 3-replica service
PDB deadlock trap: If you set minAvailable: 100% (or equal to replicas) and then try to drain a node, the drain hangs forever — PDB says "zero pods can be disrupted" and Kubernetes respects that. Common in single-replica services where someone sets minAvailable: 1 on a 1-replica Deployment — now the node can never be drained. Fix: either scale up to 2 replicas, or use maxUnavailable: 1 which is a safer default.
Q: "We're about to drain a node for upgrade. How do PDBs affect that?"
"Drain uses the Eviction API, which enforces PDBs. Here's what happens:
  1. kubectl drain node-X iterates through pods on the node.
  2. For each pod, Kubernetes calls the Eviction API. If evicting this pod would violate a PDB (drop below minAvailable), the eviction is refused.
  3. Drain retries — it waits for another pod of the same PDB group to come back up on a different node before evicting the next one.
  4. Services without a PDB get evicted with no guardrails — could go to zero momentarily during drain.
So on a platform without PDBs, a cluster upgrade can cause transient downtime as all replicas of a 2-replica service evict simultaneously. With minAvailable: 1, drain throttles itself to one-at-a-time.

Operational note: A drain can hang indefinitely if the evicted pods can't reschedule elsewhere (no capacity, PVC zone mismatch). Watch for Cannot evict pod as it would violate the pod's disruption budget in drain output — that's your signal to investigate why replacements aren't coming up."

Jobs & CronJobs Medium Priority

Deployments run forever. Jobs run to completion. Your platform uses both Jobs (Liquibase migrations) and CronJobs (OpenSearch snapshots, Fivetran slot manager). Interviewers ask about failure semantics, concurrency, and cleanup.

Job vs CronJob

ResourceRunsCompletion semanticsYour use case
Job Once (or N times) to completion Pod exits 0 → Job complete. Exit non-zero → restart until backoffLimit exceeded. Liquibase DB migration (PreSync hook, wave -1). Dynamodb-version-updater.
CronJob On a schedule (cron syntax) Creates a new Job per schedule tick. Tracks history of completed/failed Jobs. OpenSearch snapshot (daily), Fivetran replication slot manager, Outpost storage daily report.

Your Actual Liquibase Job — PreSync Hook

Every service chart has a liquibase.yaml that runs as an ArgoCD PreSync hook (wave -1) before pods start. This is the schema-migration-before-rollout pattern:

Liquibase Job Skeleton (Your Pattern) 
apiVersion: batch/v1
kind: Job
metadata:
  name: pricing-be-liquibase-{{ .Values.image.tag }}
  annotations:
    argocd.argoproj.io/sync-wave: "-1"             # BEFORE anything else
    argocd.argoproj.io/hook: PreSync               # ArgoCD sync hook
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation
spec:
  backoffLimit: 2                                   # retry twice on failure
  ttlSecondsAfterFinished: 3600                     # auto-delete after 1h
  template:
    metadata:
      annotations:
        sidecar.istio.io/inject: "false"            # no Istio sidecar in Jobs!
    spec:
      restartPolicy: OnFailure
      containers:
        - name: liquibase
          image: .../liquibase:latest
          command: ["liquibase", "update"]
Why sidecar.istio.io/inject: "false" on Jobs: Istio's Envoy sidecar runs forever. A Job's main container exits (migration done), but the sidecar keeps running → Job is stuck "Running" forever. Always disable Istio injection on Jobs and CronJobs. This is a classic interview trap for anyone running Istio + Jobs.

Your Actual CronJob — OpenSearch Snapshot

Real CronJob From charts/opensearch/templates/ 
apiVersion: batch/v1
kind: CronJob
metadata:
  name: opensearch-snapshot-ci
spec:
  schedule: "0 2 * * *"                    # daily at 02:00 UTC
  concurrencyPolicy: Forbid                # never run two at once
  successfulJobsHistoryLimit: 3            # keep last 3 successful
  failedJobsHistoryLimit: 3                # keep last 3 failed (for debugging)
  jobTemplate:
    spec:
      ttlSecondsAfterFinished: 3600        # clean up the Job 1h after completion
      template:
        metadata:
          annotations:
            sidecar.istio.io/inject: "false"
        spec:
          restartPolicy: OnFailure
          containers:
            - name: create-snapshot
              image: .../bitnami-os-shell:12-debian-12-r45

concurrencyPolicy — Critical to Understand

PolicyBehaviorWhen to use
Allow (default) New job starts even if previous still running Independent tasks that won't collide. Rarely appropriate.
Forbid Skip the new job if previous still running Your default. Snapshots, backups, replication slot management — don't want two running at once.
Replace Kill the old one, start the new one When the latest data matters more than finishing the in-flight run.

Fivetran Slot Manager — Real Incident Context (INC-012)

Why a CronJob for slot management? Per incident INC-012, Patroni failover creates new PostgreSQL replication slots on the new primary, but Fivetran still tries to read from the old slot (which no longer exists). The slot manager CronJob runs periodically, detects missing slots, and recreates them. It's an infrastructure reconciliation loop implemented as a CronJob — the Kubernetes-native operator pattern for work that doesn't warrant building a full CRD/controller.
Q: "Your Liquibase migration hangs — pods never start. What's happening?"
"PreSync hook failure is the most common cause of a stuck sync. My debug order:
  1. kubectl get jobs -n 207804-confirmation-ci | grep liquibase — is the Job in Failed or still Active state?
  2. kubectl logs job/pricing-be-liquibase-X — actual liquibase output. Lock conflict? Schema error? Missing DB credential?
  3. Classic Liquibase issue: lock held in databasechangeloglock from a previous killed run. Manually release: UPDATE databasechangeloglock SET LOCKED=FALSE;
  4. Istio sidecar trap: if someone forgot sidecar.istio.io/inject: false, the liquibase container exits but Envoy keeps running → Job stuck Active → ArgoCD stuck waiting for PreSync.
  5. ArgoCD hook: if hook-delete-policy is HookSucceeded, a failed Job stays around blocking the next sync. Pattern we use is BeforeHookCreation — deletes the previous hook Job before creating a new one.
I'd use the db-ops portal or kubectl exec into a postgres pod to check the lock table directly. If the migration fails legitimately (bad SQL), the dev team has to fix the changelog — ArgoCD won't proceed without the migration succeeding."

ResourceQuota & LimitRange Medium Priority

Two namespace-level objects that enforce resource fairness and prevent one workload from hogging the cluster. Especially relevant on Outposts where capacity is finite.

ResourceQuota — The Namespace Budget

💰 Analogy Monthly credit limit on a corporate card. The namespace gets a budget for total CPU, memory, and object counts. Once spent, new pods/PVCs/services in that namespace are rejected by the API server. Prevents a runaway HPA from consuming the entire cluster.
ResourceQuota Example — Non-Prod Namespace 
apiVersion: v1
kind: ResourceQuota
metadata:
  name: confirmation-ci-quota
  namespace: 207804-confirmation-ci
spec:
  hard:
    requests.cpu: "80"                     # total requested CPU across all pods
    requests.memory: 160Gi
    limits.cpu: "160"
    limits.memory: 320Gi
    persistentvolumeclaims: "50"
    services.loadbalancers: "2"            # cap LB services (cost control)
    count/deployments.apps: "100"
    count/rollouts.argoproj.io: "100"
Hard rule when ResourceQuota is set: Every pod in the namespace must specify resource requests and limits — otherwise API server rejects with must specify cpu/memory for Limits. This is why your Helm chart standard always includes resource requests/limits.

LimitRange — Default + Cap per Pod

LimitRange sets per-object defaults and min/max ranges. Complements ResourceQuota.

LimitRange — Defaults for Containers Missing Them 
apiVersion: v1
kind: LimitRange
metadata:
  name: confirmation-ci-defaults
  namespace: 207804-confirmation-ci
spec:
  limits:
    - type: Container
      default:                             # if limits NOT set, use these
        cpu: 500m
        memory: 512Mi
      defaultRequest:                      # if requests NOT set, use these
        cpu: 100m
        memory: 128Mi
      max:                                 # refuse pods exceeding this
        cpu: "4"
        memory: 8Gi
      min:                                 # refuse pods below this
        cpu: 50m
        memory: 64Mi

ResourceQuota vs LimitRange

ObjectScopePurpose
ResourceQuotaNamespace totalCap aggregate resources across ALL pods in the namespace
LimitRangePer pod/containerSet defaults + enforce min/max on individual pods
Outposts relevance: Outpost capacity is fixed by rack hardware. ResourceQuota per namespace prevents one app team from consuming the entire rack. In your 8 environments on 4 clusters, quotas would be how you reserve capacity for each namespace deterministically rather than hoping HPAs behave.

Topology Spread Constraints Medium Priority Modern Alt

Modern replacement for podAntiAffinity. Introduced GA in 1.19. More expressive, better at enforcing even spread across failure domains. Interviewers increasingly ask about this over antiAffinity.

Why Topology Spread Beats Pod Anti-Affinity

FeaturepodAntiAffinitytopologySpreadConstraints
Expressive Binary — same node or not Fine-grained — enforce "at most 1 pod skew across zones"
Scheduler performance O(pods × pods) — slow on large clusters O(pods) — scales better
Even distribution Best-effort, can clump Guarantees ≤ maxSkew imbalance
Intent "Don't be here" "Spread evenly"
Topology Spread — Spread Across Nodes Evenly 
spec:
  topologySpreadConstraints:
    - maxSkew: 1                                          # max imbalance between nodes
      topologyKey: kubernetes.io/hostname                 # spread across nodes
      whenUnsatisfiable: DoNotSchedule                    # enforce (hard)
      labelSelector:
        matchLabels:
          app: pricing-be
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone            # also spread across AZs
      whenUnsatisfiable: ScheduleAnyway                   # prefer (soft)
      labelSelector:
        matchLabels:
          app: pricing-be
whenUnsatisfiable:
  • DoNotSchedule — hard constraint. Pod stays Pending if spread would be violated.
  • ScheduleAnyway — soft. Prefer balanced spread but schedule somewhere if no ideal node.
Common pattern: hard constraint on hostname (must spread across nodes), soft on zone (prefer but don't block).

Outposts-Specific Usage

Outposts have a single logical AZ (e.g. us-east-1-op1). Zone-level topology spread won't help you there — there's only one Outpost zone. Spread across nodes (kubernetes.io/hostname) is what matters. If your Outpost rack has 6 worker nodes and you run 3 replicas, topology spread ensures each replica lands on a different node — so a single node reboot can't take out your service.

Finalizers — Why Things Get Stuck Terminating Medium Priority

A finalizer is a pre-delete hook. Controllers register finalizers on resources they manage. When you delete the resource, Kubernetes waits for every finalizer to be removed before the object actually disappears. This is why namespaces, PVCs, and CRDs can get stuck in Terminating for hours.

📄 Analogy Moving out of an apartment — you can't leave until the utilities are cancelled, keys returned, and final inspection done. Each item is a "finalizer." Only after every item is checked off does the landlord mark your lease as terminated. If a finalizer controller is dead or stuck, your object sits in Terminating forever.

How Finalizers Work

Deletion Flow With Finalizers

Common Stuck-Terminating Scenarios

ScenarioStuck finalizerFix
Namespace stuck Terminating Usually a CRD whose controller is gone, or a resource in the NS that can't be cleaned up kubectl get ns X -o json → find .spec.finalizers. If controller is truly gone, patch them out (last-resort, destructive).
PVC stuck Terminating kubernetes.io/pvc-protection — prevents deletion while a pod still uses it Delete pods referencing the PVC first. Finalizer auto-removes.
CRD stuck Terminating Instances of the CR still exist and their controller is down Restart the operator pod. If operator is gone permanently, remove the finalizer from remaining CRs first.
Pod stuck Terminating Usually NOT a finalizer — it's the terminationGracePeriodSeconds waiting for preStop or SIGTERM handling kubectl delete pod X --force --grace-period=0 (but understand this skips graceful shutdown)
The Nuclear Option — Manually Remove a Finalizer 
# Check current finalizers:
kubectl get ns stuck-ns -o jsonpath='{.spec.finalizers}'

# Remove all finalizers (DANGEROUS — skips real cleanup):
kubectl patch ns stuck-ns -p '{"metadata":{"finalizers":[]}}' --type=merge

# OR for CRs:
kubectl patch crontab/my-cron --type=json -p='[{"op":"remove","path":"/metadata/finalizers"}]'
Don't force-remove finalizers as your first move. They exist because a controller needs to clean up external resources (an AWS LB, an EBS volume, a database record). Skipping them leaves orphaned cloud resources costing you money. Only force-remove when you've confirmed the controller is permanently dead.
Q: "A namespace has been stuck Terminating for 2 hours. Walk me through how you'd investigate."
"Always investigate before force-patching.
  1. kubectl get ns stuck-ns -o yaml — look at .status.conditions. It often has NamespaceDeletionContentFailure with the exact resource that's blocking.
  2. kubectl api-resources --verbs=list --namespaced -o name | xargs -n1 kubectl get -n stuck-ns 2>/dev/null — find any lingering resources.
  3. For each lingering resource, inspect .metadata.finalizers — identify which controller is supposed to remove it.
  4. Check the controller pod is running (kubectl get pods -n operator-ns) and its logs.
  5. If the controller is fine but the cleanup is failing (maybe AWS API throttling), wait or fix the external dependency.
  6. If the controller is dead and can't be brought back, patch the finalizer out as a last resort — and manually clean the external resource it was supposed to clean.
I had a case with a stuck ExternalSecret namespace — ESO pod had crashed, finalizer never removed. Restarted ESO, finalizer cleared automatically in 10 seconds."

Ephemeral Containers & kubectl debug Medium Priority Readonly Root FS

Debugging your own platform is harder than generic K8s because every pod has readOnlyRootFilesystem: true, no shell in distroless-ish images, and dropped ALL capabilities. kubectl exec is often useless. kubectl debug with ephemeral containers is the answer.

🧰 Analogy Attaching a diagnostic computer to a running car engine without shutting it off. An ephemeral container is a temporary container injected into a running pod's namespace — it shares the network and (optionally) process namespace of the target container, but brings its own tools (curl, nslookup, strace). Target pod keeps running the whole time.

The Debug Patterns

CommandWhat it doesWhen to use
kubectl debug pod/X -it --image=busybox Adds an ephemeral busybox container to pod X. Shares network namespace. Network debugging — nslookup, curl, netstat from the pod's perspective.
kubectl debug pod/X --target=app --image=nicolaka/netshoot Attaches to the PID namespace of container app — can ps, strace that container's processes. Deep debugging — see actual processes, strace system calls on a running container that has no shell.
kubectl debug node/my-node -it --image=ubuntu Starts a privileged pod on a specific node with root FS mounted at /host. Node-level debugging — check kubelet logs, disk usage, containerd state without SSH.
kubectl debug pod/X --copy-to=X-debug --set-image=app=busybox Makes a copy of the pod with a different image for the target container. When the real image has a crash bug and you need a shell to inspect the environment/volumes.

Your Specific Pain Point — readOnlyRootFilesystem + distroless-style images

Why kubectl exec often fails on your pods:
  • readOnlyRootFilesystem: true — can't install tools temporarily.
  • Minimal base images (slim/alpine) often lack curl, dig, tcpdump.
  • Dropped ALL capabilities — even if you had tcpdump, it couldn't raw-socket.
  • UID 14000 non-root — can't install packages anyway.
kubectl debug --target=<container> --image=nicolaka/netshoot solves all four — the ephemeral container is a separate container with its own rootfs, its own capabilities (within PSA limits), but shares the network/PID namespace of your target.

The Proxy Problem — Why NO_PROXY Is A Nightmare

Per-service proxy config: On Outposts, pods are on-prem. To reach AWS regional services, traffic must traverse the corporate proxy → out TR's internet egress → back into AWS. But internal K8s traffic (service mesh, CoreDNS, pod-to-pod) must not go through the proxy. Every service's Helm values has:
Per-Service NO_PROXY Pattern 
env:
  HTTP_PROXY: "http://proxy.int.thomsonreuters.com:8080"
  HTTPS_PROXY: "http://proxy.int.thomsonreuters.com:8080"
  NO_PROXY: "10.0.0.0/8,.svc,.cluster.local,\
    dynamodb.us-east-1.amazonaws.com,\
    dynamodb.us-east-1.vpce.amazonaws.com,\
    secretsmanager.us-east-1.amazonaws.com,\
    *.s3.us-east-1.amazonaws.com,\
    localhost,127.0.0.1"
Each service needs a slightly different NO_PROXY because each one talks to different AWS services. Reports-be talks to Commercial Gateway hostnames — those must be in NO_PROXY or requests try to go via the proxy and fail. This was the root cause of INC-004 (Reports timing out 53-57 seconds). Getting the NO_PROXY wrong is one of the most common Outposts operational bugs.

Failure Modes — What Actually Happens

FailureObservable behaviorWhat still works
Service Link fully down EKS shows nodes as NotReady after node-monitor-grace-period (40s default). No new pods scheduled. ECR pulls fail. Existing pods keep running. CoreDNS cache serves intra-cluster DNS. Istio sidecars keep routing. Local traffic (via LGW) still works.
Service Link degraded (high latency) kubelet heartbeat intermittent. Node flips Ready/NotReady. HPA metrics stale. Image pulls slow. Running workloads mostly fine. Deployments slow or time out.
LGW down (customer-side) Pods can't reach on-prem PostgreSQL/RabbitMQ/legacy systems. Readiness probes fail. Services removed from Istio routing. Cluster control plane OK. ECR OK. Intra-cluster services still work. Just can't reach on-prem deps.
Outpost rack power/cooling Nodes go NotReady simultaneously. Involuntary pod termination. Only if you have a DR cluster, failover kicks in. Otherwise total platform outage.
Outpost capacity exhausted New pods stuck Pending with 0/N nodes have enough CPU/memory. PVCs stuck Pending if local EBS full. Existing workloads unaffected. Need AWS support to physically add capacity.

Capacity Planning — Different From Cloud

In the cloud, you don't plan capacity — AWS does. On Outposts, capacity is fixed by the rack hardware. If you need more, you file a capacity increase ticket with AWS and wait weeks for a physical rack upgrade. Planning matters:
  • CPU/memory headroom: Size your cluster for peak + 20% headroom. HPAs can't magically scale past the rack limit.
  • EBS local capacity: Pre-allocated at rack provision time. Enabling PVC expansion (allowVolumeExpansion: true) lets you grow individual PVCs up to the pool limit.
  • IP addresses: VPC CNI uses real IPs. Outposts CIDRs are TR-allocated on-prem ranges — much smaller than cloud VPCs. Enable prefix delegation (/28 prefixes per ENI) for ~16x more IPs per node.
  • Instance type diversity: Outposts only stocks specific EC2 types. No m7i/r7i. No Graviton/ARM. Plan for what the rack actually has.

IAM + IRSA on Outposts

IRSA still works. The EKS control plane is in the Region; IAM is a global service; the OIDC federation flow completes over the Service Link. Your pods get AWS credentials just like cloud EKS. The one caveat: if the Service Link is down, pods may not be able to refresh credentials when they expire (~1 hour) — eventual AWS API call failures even on pods that were running fine.

Why TR Chose Outposts — The Interview Answer

Three reasons:
  1. Data residency. TR audit/confirmation data must stay within TR's physical boundary for regulatory reasons. Regional EKS puts data in AWS us-east-1 data centers; Outposts keeps it in TR's own data center.
  2. Sub-millisecond latency to on-prem systems. Confirmation platform talks to on-prem legacy systems and some databases. Cloud EKS + DirectConnect adds 5-10ms per round trip, which compounds across microservice chains. Outposts + LGW is LAN-local.
  3. API compatibility. Same Kubernetes APIs, same Helm charts, same ArgoCD pipeline as cloud EKS. Migration from cloud → Outposts was mostly values.yaml changes (nodeAffinity, storage classes, proxy config) — no fundamental rearchitecture.

Q: "Walk me through what would happen if your Outpost's Service Link went down at 3 AM."
"I've thought about this scenario — it's a real operational risk.
  1. t+0: Link severs. No immediate symptom — worker nodes have cached kubelet auth tokens, pods are running.
  2. t+40s: kubelet can't reach API server. node-monitor-grace-period expires → nodes marked NotReady from controller's perspective.
  3. t+5min: tolerationSeconds on default pod tolerations expires → controller starts trying to evict pods from NotReady nodes. But the API request itself has to traverse the link, so evictions don't actually happen.
  4. Ongoing: Existing pods keep running — kubelet still manages them locally from cached specs. On-prem DB traffic via LGW still works. Istio sidecars keep routing. Requests from TR users to the platform still succeed if they hit the Outpost via LGW.
  5. What breaks: ImagePullBackOff on any new pod (ECR unreachable). ExternalSecrets refresh fails (pods with rotated secrets eventually get stale values). No scaling, no config changes, no deploys.
  6. Recovery: Link restores → heartbeats resume → nodes flip back to Ready → any in-flight ArgoCD syncs complete. Usually self-healing within minutes once connectivity restores.
What we monitor: We have a Datadog monitor on the Service Link latency metric (via CloudWatch). Alert fires at >100ms p95 sustained, pages at >500ms. For the actual mitigation on prolonged outage, we'd failover traffic to the DR cluster in a separate Outpost — but that's CHG-ticket-gated and not automatic."
Q: "Why not just run in the cloud? What justifies Outposts' cost and complexity?"
"Honest answer — for a pure greenfield app with no data locality constraints, cloud is simpler and cheaper. Outposts adds rack cost (6-figure monthly), capacity planning overhead, and operational complexity (proxy, LGW, Service Link, limited instance types).

The justification for us is specifically: (1) regulatory data residency (audit data must stay in TR's physical footprint), (2) sub-ms latency to legacy on-prem systems that can't move to cloud, and (3) a hybrid consistency story — we run the same EKS/Helm/ArgoCD pipeline that cloud EKS does, so our engineers develop against cloud and deploy to Outposts with mostly config-level changes.

If you removed any one of those three drivers, I'd probably recommend cloud EKS + DirectConnect instead. Outposts is the right tool for a specific compliance + latency combination — not a default."
