G1

RBAC — Role-Based Access Control Critical Gap

RBAC is how Kubernetes decides who can do what to which resources. Every request to the API Server is checked: Is this identity allowed to perform this verb on this resource in this namespace?

🏨 Analogy Hotel keycard system. A Role defines which doors exist and can be opened (verbs: get, list, create, delete — on resources: pods, secrets, deployments). A RoleBinding is handing that keycard to a specific person or team (a ServiceAccount, user, or group). ClusterRole is a master key that works on every floor (cluster-wide). ClusterRoleBinding hands that master key to someone.

The 4 RBAC Objects

namespace-scoped
Role
Defines permissions within one namespace. Lists which verbs (get, list, create, delete, patch, watch) are allowed on which API resources (pods, secrets, configmaps, deployments).
namespace-scoped
RoleBinding
Attaches a Role (or ClusterRole) to a subject (ServiceAccount, User, Group) within one namespace. The binding is what makes the permission active.
cluster-scoped
ClusterRole
Same as Role but cluster-wide. Also used for non-namespaced resources like Nodes, PersistentVolumes, and Namespaces themselves. System components (kube-system) use these.
cluster-scoped
ClusterRoleBinding
Grants a ClusterRole to a subject across the entire cluster. Use sparingly — this is broad access. Typically used for cluster-level operators (ArgoCD, ESO, Datadog agent).

Role vs ClusterRole — When to Use Which

Use CaseRight ObjectWhy
App pod reads its own ConfigMap in same namespace Role + RoleBinding Namespace-scoped — least privilege. Pod's ServiceAccount gets read access only in its namespace.
ExternalSecrets Operator syncing secrets across all namespaces ClusterRole + ClusterRoleBinding ESO needs to manage ExternalSecret CRDs cluster-wide — namespace Role can't span namespaces.
ArgoCD deploying to specific namespace only ClusterRole + RoleBinding (in target ns) Reuse a ClusterRole definition, but bind it only in the target namespace. Best practice for multi-tenant ArgoCD.
Datadog Agent reading node metrics ClusterRole + ClusterRoleBinding Nodes are cluster-scoped resources. Agent needs to list/watch nodes cluster-wide.
Developer read-only access to one namespace Role + RoleBinding Scoped to their namespace only. Cannot see other teams' resources.

ServiceAccounts — The Identity of a Pod

Every pod runs as a ServiceAccount. If you don't specify one, it uses the default ServiceAccount in the namespace — which has minimal permissions. Pods that need to call the K8s API (operators, custom controllers, ESO) need their own ServiceAccount with appropriate Role/ClusterRole.

Real Example — ExternalSecrets Operator RBAC 
# The ClusterRole defines what ESO is allowed to do
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: external-secrets-operator
rules:
  - apiGroups: ["external-secrets.io"]
    resources: ["externalsecrets", "secretstores"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: [""]
    resources: ["secrets"]           # native K8s secrets
    verbs: ["get", "create", "update", "patch", "delete"]
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["create", "patch"]
---
# The ClusterRoleBinding wires it to ESO's ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: external-secrets-operator
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: external-secrets-operator
subjects:
  - kind: ServiceAccount
    name: external-secrets
    namespace: external-secrets   # ESO's namespace
IRSA Connection: In EKS, RBAC controls access inside the cluster (K8s API). IRSA (IAM Roles for Service Accounts) controls access to AWS services (S3, Secrets Manager, DynamoDB). ESO uses both — RBAC to manage K8s Secrets, IRSA to read from AWS Secrets Manager. Two separate permission systems working together.

Verbs Cheat Sheet

VerbHTTP EquivalentWhat It Does
getGET /resource/nameRead a specific named resource
listGET /resourceList all resources of a type (needed for watch)
watchGET /resource?watch=trueStream real-time changes — operators use this
createPOST /resourceCreate a new resource
updatePUT /resource/nameReplace entire resource spec
patchPATCH /resource/namePartial update — safer than update
deleteDELETE /resource/nameDelete a specific resource
deletecollectionDELETE /resourceDelete all matching resources — rarely granted
Security Anti-patterns: Never use verbs: ["*"] or resources: ["*"] except for cluster-admin. Never bind cluster-admin to application ServiceAccounts. Treat RBAC like IAM — least privilege always. A misconfigured RBAC can let a compromised pod read all secrets cluster-wide.

Testing & Debugging RBAC

  --as=system:serviceaccount:207804-confirmation-ci:my-service-sa \
  --namespace=207804-confirmation-ci

# List all permissions a ServiceAccount has

kubectl get rolebindings,clusterrolebindings -A \
  -o wide | grep secrets
Q: "How do you ensure a pod in namespace A can't read secrets in namespace B?"
"By default in Kubernetes, a pod can only access resources via the API Server if its ServiceAccount has been granted permission via Role/RoleBinding. Namespaces are the isolation boundary.
  1. Give each microservice its own dedicated ServiceAccount — never share ServiceAccounts across teams/namespaces
  2. Bind a Role (not ClusterRole) to that ServiceAccount — the Role is scoped to its namespace only
  3. The Role grants only what the app needs — e.g., get its own ConfigMap, create Events
  4. With this setup, even if the pod is compromised, the attacker can only reach resources within that namespace via the K8s API
  5. For secrets specifically — combine RBAC with ExternalSecrets Operator. App pods don't need direct RBAC to secrets; ESO syncs them. Reducing blast radius further.
In our platform: each of the 30+ services has its own ServiceAccount. ESO's ClusterRole is tightly scoped to only the secret paths it needs in AWS Secrets Manager via IRSA conditions."
Q: "What's the difference between a Role and a ClusterRole?"
"Scope is the only difference.
  • Role lives in and applies to one namespace. A Role in 207804-confirmation-ci cannot grant access to resources in 207804-confirmation-prod.
  • ClusterRole is namespace-agnostic — it can be bound cluster-wide (ClusterRoleBinding) OR bound to a specific namespace (RoleBinding). The second pattern is useful: define the permissions once as a ClusterRole, reuse by binding it only in specific namespaces.
  • Some resources have no namespace (Nodes, PersistentVolumes, Namespaces, StorageClasses) — only ClusterRoles can grant access to these.
Rule of thumb: start with Role. Reach for ClusterRole only when the actor needs cross-namespace access or access to cluster-scoped resources."
G2

Network Policies — Traffic Firewall Inside the Cluster Critical Gap

By default, all pods in a Kubernetes cluster can reach all other pods, regardless of namespace. Network Policies are the mechanism to restrict this — defining which pods can send traffic to which other pods, and on which ports.

🏢 Analogy Office building firewall rules. Without Network Policies, every desk (pod) can call any other desk directly. With Network Policies, you define: "The frontend pod can call the backend pod on port 8080. The backend pod can only call the database pod on port 5432. Nothing else is allowed." Each pod gets its own firewall rulebook. A policy on the receiver side controls who can talk to it (ingress). A policy on the sender side controls where it can go (egress).
Critical Prerequisite: NetworkPolicy objects are only enforced if your CNI plugin supports it. The default AWS VPC CNI alone does NOT enforce NetworkPolicies. You need either: Calico (most common), Cilium (eBPF-based, modern), or enable the AWS VPC CNI Network Policy controller (GA in EKS since 2023). Without a supporting CNI, your NetworkPolicy YAML exists but does nothing.

NetworkPolicy Example — Restrict Backend Pod 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-netpol
  namespace: 207804-confirmation-ci
spec:
  podSelector:              # This policy applies to pods with this label
    matchLabels:
      app: backend

  policyTypes:
    - Ingress               # Control who can SEND traffic TO backend
    - Egress                # Control where backend can SEND traffic

  ingress:
    - from:
        - podSelector:      # Only pods with app=frontend can reach backend
            matchLabels:
              app: frontend
      ports:
        - protocol: TCP
          port: 8080

  egress:
    - to:
        - podSelector:      # Backend can only call postgres
            matchLabels:
              app: postgres
      ports:
        - protocol: TCP
          port: 5432
    - to:                   # Also allow DNS resolution (CoreDNS)
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
DNS Gotcha: If you add an Egress policy, you MUST explicitly allow port 53 UDP to CoreDNS. Otherwise your pod can't resolve DNS and nothing works — even if you've allowed all other outbound traffic. This trips up engineers who add egress rules without thinking about DNS. Always include the CoreDNS egress rule.

Three Selector Types for Policy Targets

SelectorWhat It MatchesExample Use Case
podSelector Pods with specific labels in the same namespace (unless combined with namespaceSelector) Allow only app=frontend pods to reach app=backend
namespaceSelector All pods in namespaces matching a label Allow monitoring namespace to scrape metrics from your namespace
ipBlock An IP CIDR range (for external traffic) Allow ingress only from your corporate VPN CIDR; block all others

Zero-Trust Pattern — Default Deny Everything

The recommended enterprise pattern: start with a default deny-all policy on every namespace, then add explicit allow policies for each allowed flow. Anything not explicitly allowed is blocked.

Default Deny-All Policy (apply to every namespace) 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: 207804-confirmation-ci
spec:
  podSelector: {}     # Matches ALL pods in the namespace
  policyTypes:
    - Ingress
    - Egress
  # No ingress or egress rules = deny everything
With Istio: In your setup, Istio mTLS already provides strong identity-based service-to-service authentication at the application layer. NetworkPolicies add a complementary network-layer firewall. Istio controls which services are authorized to talk (AuthorizationPolicy); NetworkPolicy controls which pod IPs can send packets. Defense in depth — both layers working together.

Istio AuthorizationPolicy vs NetworkPolicy

Use together?Yes — NetworkPolicy for network-layer isolation, Istio for application-layer authorization. Defense in depth.
Q: "Your cluster was audited and the finding says 'all pods can reach all pods'. How do you fix it?"
"This is a network segmentation finding. The fix is to implement NetworkPolicies with a zero-trust model.
  1. First: ensure your CNI enforces policies — confirm Calico, Cilium, or AWS VPC CNI network policy controller is running. Without enforcement, policies are decoration.
  2. Apply default-deny-all per namespace — podSelector: {} + no ingress/egress rules blocks all traffic in that namespace.
  3. Map traffic flows — use Istio's Kiali dashboard or network observability tools to visualize what actually talks to what. Don't guess.
  4. Add explicit allow policies — for each legitimate flow (frontend→backend, backend→postgres, monitoring→all), create a NetworkPolicy.
  5. Don't forget DNS — add egress allow to CoreDNS (port 53 UDP) or nothing will resolve.
  6. Test in CI first — apply in the CI namespace, run integration tests, confirm no regressions before rolling to prod.
In our platform, Istio AuthorizationPolicy already handles service-to-service authorization at L7. Adding NetworkPolicy at L3/L4 gives us defense in depth — even if Istio is misconfigured, the network layer still blocks unauthorized pod-to-pod traffic."
G3

Cluster Autoscaler & Karpenter — Node-Level Scaling Critical Gap

HPA scales pods. But what happens when there are no nodes with enough capacity to schedule those new pods? That's where node-level autoscaling comes in — automatically adding or removing EC2 instances based on cluster demand.

🏗️ Analogy HPA is a hiring manager, CA/Karpenter is a real estate developer. HPA says "we need 5 more workers." If there's no office space (node capacity), the workers just wait in the lobby (Pending pods). Cluster Autoscaler or Karpenter sees the lobby filling up and builds a new office floor (provisions a new EC2 node) to accommodate them. When demand drops, they demolish empty floors (terminate underutilized nodes) to save cost.

Cluster Autoscaler (CA) — The Traditional Approach

CA watches for Pending pods — pods the scheduler couldn't place due to insufficient node resources. When it sees Pending pods, it asks AWS to add a node to a pre-defined Auto Scaling Group (ASG). It only works with ASGs you've defined in advance.

Scale-Down — How CA Removes Nodes

CA checks every 10 seconds. A node is a candidate for removal if it has been underutilized for 10 minutes (utilization below 50% by default) AND all its pods can be safely moved elsewhere. CA simulates the eviction, checks PDBs, and drains the node gracefully before terminating the EC2 instance.

Pod not managed by a controller (standalone pod)CA can't guarantee rescheduling for unmanaged podsAlways use Deployments/StatefulSets, never bare pods
PodDisruptionBudget blocks evictionPDB would be violated if the pod is evictedEnsure minAvailable allows eviction (don't set minAvailable = replicas)
Node has local storage (emptyDir)Data would be lostUse PVCs instead of emptyDir for anything that matters

Karpenter — The Modern Alternative

Karpenter (open-source, AWS-native, now part of CNCF) takes a fundamentally different approach. Instead of working with pre-defined ASG node groups, Karpenter directly calls EC2 APIs to provision exactly the right instance type for pending pods — in seconds, not minutes.

🚗 Analogy CA is a taxi dispatch with a fixed fleet; Karpenter is Uber. CA can only use the cars (instance types) already in its pre-configured ASG fleet. Karpenter looks at what the passenger needs (pod requirements: CPU, memory, GPU, spot/on-demand) and dynamically summons exactly the right car from the entire EC2 catalog.

  name: default
spec:
  template:
    spec:
      requirements:
        - key: "karpenter.k8s.aws/instance-category"
          operator: In
          values: ["c", "m", "r"]         # compute, memory, or general purpose

AWS Outposts Context: On Outposts, Karpenter's dynamic provisioning is less useful because Outpost capacity is physically fixed — you can't dynamically add more bare-metal to an on-prem rack. This is a nuance worth mentioning in interviews when they ask about your Outposts migration.
Q: "Your pods keep going Pending during peak hours even though HPA is working. What do you do?"
"Pending pods after HPA scales = the cluster doesn't have enough node capacity. Here's my approach:
  1. Confirm the cause: kubectl describe pod <pending-pod> → Events will say 0/N nodes are available: insufficient cpu/memory
  2. Check CA/Karpenter logs: Is it seeing the pending pods? Is it blocked from scaling? Common blockers: max node group size reached, spot capacity unavailable, budget limit hit
  3. Short-term fix: Manually increase the max size of the ASG or Karpenter NodePool CPU limit
  4. Root cause analysis:
    • Are resource requests accurate? Inflated requests prevent bin-packing
    • Is node scale-up too slow? Karpenter is faster than CA — good candidate to switch
    • Is the peak predictable? Add scheduled scaling (KEDA or ASG scheduled actions) to pre-provision nodes before the peak hits
  5. KEDA consideration: For our RabbitMQ-driven services, KEDA scales pods based on queue depth. We combine it with Karpenter so nodes scale up proactively when queue depth grows, before pods even go Pending.
Q: "What's the difference between HPA, VPA, and Cluster Autoscaler?"
"Three different scaling dimensions — think of it as width, height, and floor count:
  • HPA (Horizontal Pod Autoscaler) — adds more pods (width). Scale out by replicating. Works for stateless services. My primary autoscaler for microservices.
  • VPA (Vertical Pod Autoscaler) — makes existing pods bigger (height). Increases CPU/memory of the same pod. Useful for batch jobs or apps that can't be easily replicated. Conflicts with HPA if both target the same CPU metric.
  • Cluster Autoscaler / Karpenter — adds more nodes (floor count). Responds to Pending pods by provisioning new EC2 instances. Works with HPA — they complement each other. CA/Karpenter is the infrastructure layer; HPA/VPA is the workload layer.
The scaling chain: traffic spike → HPA adds pods → no node capacity → Karpenter adds nodes → pods schedule → system stable."
