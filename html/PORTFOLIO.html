Key Design Decisions

  • Explicit stack dependencies: KMS → SecretManager ordering enforced in app.py to prevent deployment race conditions
  • Dual ECR repos: Separate CI and Demo ECR repos per service — image tags never cross-contaminate environments
  • TagConstruct: Reusable construct that applies cost allocation and environment tags to every resource
  • ECR naming convention: Standardized a207804/{env}-{service} pattern — enforced across all 40+ service repos
  • Secret naming: Renamed secrets (bill-ldr → bill-ldr-be, forms → forms-be, api-adapter → api-adapter-be) for consistency with K8s service names
  • JFrog secrets: Added JFrog Artifactory credentials to Secrets Manager for use in CI/CD pipelines (not hardcoded in GHA)

Health probe cachingCached validate_connectivity() result for 60s — prevents K8s liveness probe timeouts on proxy-routed AWS API callsMar 2, 2026

Why the health cache matters

Post-CHG0130049 all AWS calls route through a corporate HTTPS proxy. Each liveness probe call to /health was triggering a fresh AWS connectivity check (~3–5s via proxy). At 10s probe interval this caused intermittent probe timeouts and pod restarts. Caching the result for 60s reduces probe latency to ~0ms while still catching real connectivity failures within a 1-min window.

🎓 Interview Prep — Concepts & STAR Answers

GitOps & Kubernetes

Q: Explain GitOps and how you implemented it at scale.
Situation: 40+ .NET microservices needed consistent, auditable deployments across 8 environments.

Task: Eliminate manual kubectl applies and deployment drift.

Action: Used two Git repos as the source of truth (confirmation-helm-charts for non-prod, itops-helm-charts for prod). ArgoCD watches these repos and auto-syncs with pruning + self-healing. CI pipelines update image tags via GitHub Actions — the Git commit IS the deployment record. ArgoCD ApplicationSets templated all environments from a single chart.

Result: Zero configuration drift, full audit trail via Git history, rollback = git revert, every change reviewed via PR.
Q: How do you handle zero-downtime deployments in Kubernetes?
Answer: We use Argo Rollouts for canary deployments — traffic gradually shifts from stable to canary (e.g., 10% → 50% → 100%) with automated analysis steps. Rollback is automatic if error rate or latency breaches thresholds. Combined with Istio traffic splitting (VirtualService weights) for fine-grained control. For DB schema changes, Liquibase migrations run as K8s Jobs pre-deployment (separate ArgoCD sync wave) so app and DB are always compatible.
Q: Tell me about an incident you resolved and what you learned.
Situation: Three RabbitMQ production incidents in September 2025 — messages queuing, consumers unhealthy.

Root Cause: Persistence storage filling up (RabbitMQ default disk alarm at 50MB free), compounding with message TTL not set on error queues — causing unbounded accumulation.

Resolution: Increased PVC to 10Gi, added TTL policy on error queues via RabbitMQ Topology Operator, documented RabbitMQ incident runbook with prevention strategies.

Learning: Always set TTL policies on error queues from day one. Monitor disk_free_alarm, not just queue depth. Document everything so the next person can resolve in 15 minutes.

Security Engineering

Q: How do you secure microservices in Kubernetes?
Defense in depth:
1. Network layer: Istio mTLS STRICT — all pod-to-pod traffic encrypted, no clear-text service calls allowed.
2. Identity layer: JWT RS256 auth on all API endpoints. Validated against JWKS URL — signature verified, not just decoded. CIAM OAuth2 client_credentials for service-to-service (M2M).
3. Container layer: runAsUser 14000 (non-root), readOnlyRootFilesystem: true, allowPrivilegeEscalation: false, seccomp RuntimeDefault.
4. Secrets layer: ExternalSecrets Operator pulls from AWS Secrets Manager at runtime — secrets never baked into images or Helm values.
5. Code layer: Bandit + Snyk + SonarQube in CI — PR blocked if any HIGH finding.
Q: What is CIAM M2M and why is it better than a shared service token?
CIAM M2M = Machine-to-Machine authentication using OAuth2 RFC 6749 client_credentials grant.

Before (ESSO): Human SSO token (1h lifetime) stored in Secrets Manager, rotated daily by a cron job. If rotation failed → service down.

After (CIAM M2M): Service has its own client_id + client_secret. Fetches 24h JWT via client_credentials. CIAMTokenClient caches in memory with 5-min buffer pre-refresh. Zero human involvement, zero daily ops.

Security benefit: Human account = broad AD/SSO permissions. Service account = minimal scoped permissions for exactly what the service needs. Principle of least privilege.

AI / ML Engineering

Q: How did you design the AI assistant's routing without calling an LLM?
The problem: Routing user queries to the right agent (Datadog, K8s, or Cert) could itself call an LLM — adding 600ms+ latency and token cost on every request.

Solution: Rule-based keyword scoring. Each agent has a weighted keyword list. The orchestrator scores the query against each agent's keywords, applies weights (Cert=2.0, Datadog=1.5, K8s=1.0), and routes to the highest scorer. Sub-millisecond, deterministic, no cost.

Why prioritize Cert? TLS certificate expiry is operationally urgent and has a very distinct vocabulary (cert, tls, ssl, expir*). Datadog gets 1.5x because log/metric queries are more ambiguous with K8s.
Q: How does your agentic loop work and what prevents infinite loops?
Agentic loop in BaseAgent:
1. Send user message + tool definitions to Claude
2. Claude responds with text OR tool calls
3. Execute tool calls (async), add results to context
4. Loop back to step 1
5. Exit when: (a) Claude responds with no tool calls (final answer), or (b) iteration count reaches max 10

The 10-iteration hard cap is the safety net. In practice, most queries resolve in 2–3 iterations. OTEL spans track each iteration and tool call duration.

Infrastructure & PostgreSQL

Q: Explain how pgBackRest works and why you chose it over pg_dump.
pg_dump: Logical backup — exports SQL DDL + data. Good for small DBs, slow for large ones, no point-in-time recovery (PITR).

pgBackRest: Physical backup tool designed for PostgreSQL HA.
- WAL archiving: Ships Write-Ahead Log segments to S3 continuously — enables PITR to any point in time
- Incremental/Differential backups: Only changed blocks after the first full backup — much faster
- Parallel I/O: 8 workers configured — prevents WAL archive backlog during heavy write periods
- Integrated with Patroni: Leader-aware — always backs up from primary, restores to replicas

We use both: pgBackRest for PITR + operational recovery, pg_dump CronJob for portable logical exports to S3.
Q: How does Istio mTLS work and how did you configure it?
Istio mTLS: Each pod gets an Envoy sidecar proxy. Istio's CA (Citadel) issues X.509 certificates to each workload. When Pod A calls Pod B, Envoy on A presents its cert, Envoy on B verifies it against Istiod's trust bundle — mutual authentication.

Configuration: PeerAuthentication with mode: STRICT in each namespace — rejects any plaintext connection. We also configure VirtualService for traffic routing, ServiceEntry for external traffic (FTP servers, proxy endpoints), and DestinationRule for load balancing and circuit breaking.

Challenge we solved: EU FTP server had a fixed IP — needed to bypass Istio for outbound traffic. Used excludeOutboundIPRanges: 10.131.38.172/32 in pod annotations.
Q: What is OpenTelemetry and how did you integrate it with Datadog?
OpenTelemetry (OTEL): Vendor-neutral observability framework — defines APIs for traces, metrics, and logs. We use the Python SDK.

How it works in our assistant:
1. FastAPI startup: TracerProvider configured with OTLP exporter pointing to Datadog Agent sidecar (port 4317/gRPC)
2. Every chat request creates a root span: chat_stream_complete with request_id, model, duration_ms
3. Each agent invocation creates a child span: agent_request_start
4. Each tool call creates a span: tool_execution_complete with tool name, iteration, duration

Result: Datadog shows full trace flame graph per request. Instantly find slow tool calls, expensive LLM iterations, or failed requests — with @request_id correlation across all log lines.

🏭 Lessons Learned & Engineering Maturity

Docker Hub rate limits in CI are a real problem. Hit 100/6h anonymous pull limit — ECR image failed mid-pipeline. Lesson: always default to TR ECR base images in all Dockerfiles. Set a team-wide rule: "no docker.io/library/* in CI".
Pydantic v2 List[str] from env vars. The env_list_delimiter approach from Pydantic v1 doesn't work in v2.12. env vars are passed through json.loads() which fails on plain comma-separated strings. Fix: Annotated[List[str], NoDecode] + field_validator(mode='before') to split manually.
Bandit false positives waste CI time. Learned to distinguish real security issues from false positives early — random.uniform() for jitter is not a crypto issue, K8s dict access is not an HTTP timeout. Document every suppression with a reason comment.
Never commit .env files — but verify first. Learned to always run git ls-files | grep .env before raising a security alarm — .env is .gitignored but it's easy to assume it's tracked.
Cross-agent workflows look easy on paper but are complex in practice. The k8s-ops-assistant has handle_complex() — it was never wired into the call path. Lesson: don't document a feature as working until it's end-to-end tested and callable.
pgBackRest WAL archive backlog. During high write periods, WAL segments were queuing faster than archiving. Fix: increase parallel workers from 4 to 8, increase archive timeout to 1800s. Monitor pg_stat_archiver.last_failed_wal, not just backup status.
RabbitMQ error queues must have TTL from day one. Without TTL, error queues grow unbounded. Disk alarm fires at 50MB free. Add RabbitMQ topology policy at cluster setup time, not after incidents.
Human SSO tokens as service identities is an anti-pattern. ESSO token expires → service down. Human account = too broad permissions. Always use service accounts (OAuth2 client_credentials / CIAM M2M) for automated workloads.
ArgoCD PreSync hooks are the right place for prerequisite work. RabbitMQ CA bundle had to exist before the operator started. ArgoCD sync wave -1 with PreSync hook runs before the main sync — perfect for cert injection, DB migrations, and other ordering dependencies.
CDK bootstrap bootstrap-less pattern. Standard CDK bootstrap requires admin IAM permissions most orgs don't allow. CliCredentialsStackSynthesizer + tr-permission-boundary via CDK Aspect is the right pattern for enterprise CDK deployments with strict IAM guardrails.

🌐 Kubernetes Networking — Concepts Reference

Every pod can talk to every other pod — no NAT, no port mapping. K8s networking is flat by design. Here is the full mental model from pod-to-pod all the way to external traffic.

The K8s Networking Model — 4 Rules

#RuleWhat It Means
1Every pod gets its own IPNo two pods share an IP — even on the same node
2Pods on any node can reach any other podWithout NAT — flat routable address space
3Agents on a node (kubelet) can communicate with all pods on that nodeLocal loopback access
4A pod's IP is the same inside and outside the podNo port translation — what you see is what you get

What CNI Does

CNI is the plugin spec that sets up pod networking when a pod starts. It creates the veth pair, assigns the IP from the pod CIDR, adds the route to the kernel routing table, and configures the bridge.

NetworkPolicy — Pod-Level Firewall

Default Behavior

Without any NetworkPolicy — all pod-to-pod traffic is allowed (ingress and egress). A NetworkPolicy is a whitelist — it denies everything not explicitly allowed for pods that match its selector.

Ingress Policy Example

podSelector:
  matchLabels:
    app: backend
ingress:
- from:
  - podSelector:
      matchLabels:
        app: frontend
  ports:
  - port: 8080

Default Deny All

podSelector: {}   # all pods
policyTypes:
- Ingress
- Egress
# No ingress/egress rules = deny all

Important Gotcha

NetworkPolicy is enforced by the CNI plugin, not kube-proxy. If your CNI doesn't support it (e.g., Flannel), the policy is created but silently ignored. Calico, Cilium, and VPC CNI all support it.

1. Pod makes DNS query: my-svc.default.svc.cluster.local
2. Query hits /etc/resolv.conf nameserver → CoreDNS ClusterIP (e.g., 10.96.0.10)

4. Returns ClusterIP (e.g., 10.96.10.5)
5. Pod opens TCP to 10.96.10.5:8080
6. iptables DNAT → actual pod IP (e.g., 10.0.1.7:8080)
7. Response comes back, iptables SNAT rewrites src to ClusterIP

AuthorizationPolicy — L7 RBAC Deep Dive

AuthorizationPolicy lets you say: "only service X in namespace Y, making GET requests to /api/prices, is allowed to talk to pricing-be". This runs inside Envoy — not in the app code.

Principal = mTLS Identity

The principal in an AuthorizationPolicy is the SPIFFE identity extracted from the mTLS cert:

cluster.local/ns/<namespace>/sa/<serviceaccount>

istiod mints this cert when the pod starts. Envoy reads the cert from the established mTLS handshake to verify identity — no tokens, no passwords.

Evaluation Order

Istio evaluates in this order:
1. CUSTOM (external auth — e.g., OPA)
2. DENY — if any deny policy matches, reject
3. ALLOW — if any allow policy exists for the workload, at least one must match
4. If no allow policy at all → default ALLOW

This means: adding your first ALLOW policy implicitly blocks everything not listed.

iptables Rules Istio Injects (Init Container)

istio-init container runs before app starts and sets these iptables rules:

# Intercept ALL outbound traffic → send to Envoy port 15001
-A OUTPUT -p tcp -j REDIRECT --to-port 15001
# EXCEPT: traffic from Envoy itself (UID 1337) — avoids infinite loop
-A OUTPUT -m owner --uid-owner 1337 -j RETURN
# EXCEPT: localhost traffic (pod internal)
-A OUTPUT -d 127.0.0.1/32 -j RETURN

# Intercept ALL inbound traffic → send to Envoy port 15006
-A PREROUTING -p tcp -j REDIRECT --to-port 15006
# EXCEPT: Envoy's own traffic
-A PREROUTING -m owner --uid-owner 1337 -j RETURN

This is why you see istio-init as an init container — it needs root/NET_ADMIN capability to write iptables rules. The app container and Envoy sidecar run as non-root after this.

VPC — Virtual Private Cloud

VPC Basics

A VPC is a logically isolated network in AWS. You define the CIDR block (e.g., 10.0.0.0/16), then subnet it. Resources inside share routing, security groups, and NACLs.

cidrsubnetroute-table

Subnets

Public subnet — has a route to the Internet Gateway (0.0.0.0/0 → IGW). Resources can get public IPs.
Private subnet — route to NAT Gateway for outbound only. No direct inbound from internet.

public-subnetprivate-subnetnat-gateway

Route Tables

Each subnet is associated with a route table. Routes are evaluated most-specific-first. Local route (10.0.0.0/16 → local) is always implicit and cannot be deleted.

Internet Gateway vs NAT Gateway

IGW: Bidirectional — internet can initiate connections to public-subnet resources.
NAT GW: Outbound only — private resources can reach internet (e.g., pull images), but internet cannot reach them.

AWS Networking — Key Gotchas & Interview Tips

Security Groups are stateful, NACLs are not. If you allow port 443 inbound in an SG, the response is automatically allowed. In a NACL, you must explicitly allow the ephemeral port range (1024–65535) outbound for responses.
NAT Gateway is not free. You pay per hour and per GB of data transferred. Use VPC Gateway Endpoints for S3 and DynamoDB to avoid routing that traffic through NAT GW — significant cost saving.
VPC CIDR planning matters — do it before you build. Once a VPC has a CIDR, you can add secondary CIDRs but cannot remove the primary. Overlapping CIDRs block VPC Peering. Plan for /16 VPCs with /24 subnets per AZ.
DNS resolution flags on VPCs. Two VPC settings must be enabled for Route 53 private hosted zones and AWS service DNS to work: enableDnsHostnames and enableDnsSupport. Easy to forget when creating VPCs via CDK.
EKS max pods per node is limited by ENI capacity. Formula: (max_ENIs × (max_IPs_per_ENI - 1)) + 2. For m5.xlarge: 4 ENIs × 15 IPs = 58 max pods. Plan node types based on pod density, not just CPU/RAM.
NO_PROXY must include 169.254.169.254 on EC2. This is the EC2 metadata endpoint — if it goes through the proxy, IMDS (Instance Metadata Service) calls fail, breaking IRSA and instance profile credential refresh.

Resource Requests vs Limits

Requests

What the scheduler uses to find a node. The node must have at least this much allocatable resource free. This is a guaranteed reservation — the pod will always get at least this much.

resources:
  requests:
    cpu: "250m"    # 0.25 cores
    memory: "256Mi"

Limits

The hard ceiling. CPU: throttled (not killed) if exceeded. Memory: OOMKilled if exceeded — Linux kernel kills the process. Set limits to protect neighbours, not to guarantee resources.

resources:
  limits:
    cpu: "500m"
    memory: "512Mi"

CPU Units

1 CPU = 1000m (millicores) = 1 vCPU core.
250m = quarter of a core.
CPU is compressible — exceeding limit throttles the container (no kill).

Memory Units

Mi = mebibytes (1024²). MB = megabytes (1000²).
Memory is incompressible — exceeding limit = OOMKill.
Always set memory limit = memory request for predictability.

QoS Classes — Critical for Production

Probes — Liveness, Readiness, Startup

Liveness Probe

Question it answers: "Is this container alive or should I restart it?"
Failing liveness → container killed + restarted.
Use for: deadlock detection, hung threads.

livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3

Readiness Probe

Question it answers: "Is this container ready to receive traffic?"
Failing readiness → pod removed from Service endpoints (no traffic). Container NOT restarted.
Use for: warm-up time, dependency not ready.

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
  failureThreshold: 3

Startup Probe

Question it answers: "Has the app finished starting?"
Disables liveness + readiness until it passes. For slow-starting apps (.NET warm-up, JVM startup).
Once it passes once — startup probe stops running.

startupProbe:
  httpGet:
    path: /health/live
    port: 8080
  failureThreshold: 30   # 30 × 10s = 5min
  periodSeconds: 10

Probe Types

httpGet — HTTP GET, success = 200-399.
tcpSocket — TCP connection success.
exec — run command in container, success = exit 0.
grpc — gRPC health check protocol.

exec:
  command:
  - cat
  - /tmp/healthy
Key interview point: Liveness and readiness solve DIFFERENT problems. Liveness = restart a stuck process. Readiness = don't send traffic to a pod that isn't ready. Never set liveness too aggressive — it causes restart loops during load spikes.

Pod Termination — Graceful Shutdown

Pod delete request received:
  1. Pod set to Terminating state — removed from Service endpoints IMMEDIATELY
  2. preStop hook runs (if defined) — use for draining
  3. SIGTERM sent to PID 1 in each container
  4. App should: stop accepting new connections, finish in-flight requests, exit cleanly
  5. After terminationGracePeriodSeconds (default: 30s) — SIGKILL sent regardless
  6. Pod deleted from etcd

Best practice for our FastAPI services:
  lifecycle:
    preStop:
      exec:
        command: ["/bin/sh", "-c", "sleep 5"]  # let LB drain
  terminationGracePeriodSeconds: 60

💾 K8s Storage — PV, PVC, StorageClass & CSI

Storage in K8s is decoupled: developers request storage (PVC) without knowing the underlying infrastructure. Admins define how storage is provisioned (StorageClass). CSI drivers bridge K8s to cloud storage.

Storage Hierarchy

PersistentVolume (PV)

Access Modes

ModeAbbrevMeaning
ReadWriteOnceRWOOne node mounts R/W (EBS)
ReadOnlyManyROXMany nodes mount Read-Only
ReadWriteManyRWXMany nodes mount R/W (EFS, NFS)
ReadWriteOncePodRWOPOne POD mounts R/W (K8s 1.22+)

Reclaim Policies

Delete (default for dynamic): PV and backing disk deleted when PVC deleted. Use for ephemeral data.

Retain: PV stays after PVC deleted — manual cleanup required. Use for databases, backups. We use this for PostgreSQL PVCs.

Recycle: Deprecated. Don't use.

PVC Binding

K8s matches PVC to PV by: storage size (PV must be >= PVC request), access mode, StorageClass name, label selectors. Once bound, it's 1:1 — no sharing between PVCs.

Volume Expansion

With allowVolumeExpansion: true on StorageClass, you can increase PVC size online (EBS supports this). We did this repeatedly — 200Gi → 350Gi → 450Gi → 600Gi for pgBackRest repo. Decrease is NOT supported.

StorageClass — Dynamic Provisioning

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-encrypted
provisioner: ebs.csi.aws.com       # CSI driver
parameters:
  type: gp3
  encrypted: "true"
  kmsKeyId: arn:aws:kms:...        # encrypt with KMS
reclaimPolicy: Retain              # don't delete disk on PVC delete
allowVolumeExpansion: true         # allow online resize
volumeBindingMode: WaitForFirstConsumer  # don't create EBS until pod scheduled (AZ matters!)
mountOptions:
  - debug
volumeBindingMode: WaitForFirstConsumer — critical on multi-AZ clusters. Without it, EBS is created in a random AZ. Pod schedules in a different AZ. Pod stays Pending forever because EBS can't cross AZs. Always use WaitForFirstConsumer for EBS.

📈 Autoscaling — HPA, VPA & KEDA

Three complementary autoscalers — horizontal (more pods), vertical (bigger pods), and event-driven (queue depth, custom metrics).

HPA — Horizontal Pod Autoscaler

How HPA Works

HPA controller polls metrics every 15s (default). Compares current metric to target. Calculates desired replicas:

desired = ceil(current × (current_metric / target_metric))

Scale-up: immediate. Scale-down: waits stabilizationWindowSeconds (default 300s) to prevent flapping.

HPA on CPU

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef:
    kind: Deployment
    name: pricing-be
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

HPA on Custom Metric

metrics:
- type: Pods
  pods:
    metric:
      name: http_requests_per_second
    target:
      type: AverageValue
      averageValue: 100
# Needs: metrics-server +
# custom metrics API adapter
# (Datadog Cluster Agent,
#  Prometheus Adapter)

HPA Gotchas

1. Requires resource requests — HPA can't calculate % utilization without them.
2. Metrics server must be running.
3. HPA and manual replica changes conflict — HPA wins.
4. Don't set maxReplicas higher than your node capacity allows.

VPA — Vertical Pod Autoscaler

VPA ModeWhat It DoesDownside
OffOnly recommends — check with kubectl describe vpaNone — safe to start here
InitialSets requests/limits only at pod creation — no live updatesNeed to delete pods to apply new sizing
AutoUpdates running pods by evicting and recreating themPod restarts during scaling — disruptive
RecreateSame as Auto but always evictsSame
HPA + VPA conflict: Don't use both on CPU/memory simultaneously. VPA changes requests → HPA recalculates replicas → instability loop. Use VPA for right-sizing in staging, HPA for production scaling. Or use VPA in Off mode for recommendations only.

👀 Observability — SLI, SLO, SLA & Alert Design

Observability is knowing what your system is doing from the outside. Three pillars: Logs, Metrics, Traces. SLIs/SLOs give you objective reliability targets to engineer and alert against.

The Three Pillars

Logs

Timestamped records of discrete events. Best for debugging specific incidents. We use structured JSON logs (structlog) — every log line has: timestamp, level, event, request_id, service, namespace, pod.

structlogjsoncorrelation-iddatadog-logs

Metrics

Numeric time-series data aggregated over time. Best for alerting and dashboards. Examples: request rate (RPS), error rate (%), p99 latency (ms), CPU/memory %, queue depth, pod count.

prometheusdatadog-metricstime-seriescardinality

Traces

Distributed request flows across multiple services — shows exactly where time is spent. We use OTEL + Datadog APM. Each trace = root span + child spans. Spans have: service, operation, duration, status, tags.

oteldatadog-apmspantrace-id

SLI → SLO → SLA — The Chain

TermDefinitionExample for pricing-be
SLI — Service Level IndicatorThe metric you measure. A ratio: good events / total events.% of HTTP requests completing in <200ms with status 2xx
SLO — Service Level ObjectiveThe internal target for your SLI. Engineering commits to meeting this.99.5% of requests complete in <200ms over a 30-day window
SLA — Service Level AgreementExternal contract with customers. Stricter penalties if breached. Always less strict than SLO.99% uptime per month (SLA) vs 99.5% internal SLO
Error BudgetHow much SLO you can "spend" before breaching. 100% - SLO. Budget consumed by incidents.0.5% of 30 days = 3.6 hours/month allowed downtime

Error Budget — How to Use It

Error Budget Calculation

SLO = 99.5%
Error budget = 0.5% per 30 days
= 0.005 × 30 × 24 × 60
= 216 minutes/month

If an incident takes 2 hours:
Budget remaining = 216 - 120 = 96 min
= 44% of budget consumed

What to Do With Budget

Budget healthy (>50%): Ship features, take risks, deploy more often.

Budget <25%: Slow down deployments, focus on reliability work (reduce toil, add tests, improve monitoring).

Budget exhausted: Feature freeze. Only reliability work until next window.

Alert Fatigue — How to Avoid It

Alert on Symptoms, Not Causes

Alert: "Error rate > 1% for 5 min" (symptom — user is impacted).
Don't alert: "CPU > 80%" (cause — may not affect users at all).
CPU alerts = noise. Error rate alerts = signal.

Use Multi-Window Alerting

Short window catches fast burns. Long window catches slow burns:

Alert if: (error rate > 14.4× SLO budget/hour for 1min) OR (error rate > 6× for 5min). This is the Google SRE burn rate model.

Alert Severity Levels

P1/Critical: Page on-call immediately — user impact now.
P2/High: Page during business hours — degraded but not down.
P3/Low: Ticket — potential future issue, no user impact yet.

Runbook Link in Every Alert

Every alert should have: what it means, how to triage it, likely causes, remediation steps. If you can't write a runbook — the alert is probably too vague.

Docker Image Layers — How They Work

graph BT BASE["Layer 0: Base OS\n(python:3.11-slim)\n~120MB"] --> DEP["Layer 1: OS deps\n(apt-get install)\n~50MB"] DEP --> PY["Layer 2: Python deps\n(pip install)\n~80MB"] PY --> APP["Layer 3: App code\n(COPY . .)\n~5MB"] APP --> CMD["Layer 4: CMD\n(entrypoint)\n~0MB"]

Each layer is a content-addressable tarball. Unchanged layers are cached — only changed layers and all layers above them are rebuilt. This is why COPY requirements.txt before COPY . . — dependencies change less often than code.

Multi-Stage Dockerfile — Best Practice

WORKDIR /build

ARG BASE_IMAGE=833618288067.dkr.ecr.us-east-1.amazonaws.com/a207804/a207804-ue1-op-python:3.11-slim

    useradd -u 14000 -g appgroup -s /sbin/nologin appuser

WORKDIR /app

# Copy app source
COPY --chown=appuser:appgroup audit_devops_assistant/ ./audit_devops_assistant/

     "--host", "0.0.0.0", "--port", "8080", \

🐧 Linux Internals for DevOps

A strong Linux foundation separates senior DevOps engineers. You debug containers at the kernel level — namespaces, cgroups, file descriptors, signals, and process management are not optional knowledge.

Process Management

PID 1 — Why It Matters

PID 1 is the init process. It's responsible for:
• Reaping zombie processes (wait() for children)
• Forwarding signals to children

If your app IS PID 1 and doesn't handle SIGTERM — it's killed with SIGKILL after gracePeriodSeconds, dropping in-flight requests.

Solution: use a proper init (tini, dumb-init) or handle SIGTERM in your app.

Toil — What It Is and How to Reduce It

What is Toil?

Manual, repetitive, automatable work that scales with service growth. It provides no enduring value. Google SRE target: <50% of time on toil. Above 50% = unsustainable.

Toil We Eliminated

• ESSO token rotation (daily, manual) → CIAM M2M (automated)
• Pod restarts via SSH → service-ops Restart API
• Manual Helm deploys → ArgoCD GitOps + GHA
• Manual Windows runner setup → AWS CDK automation

Toil We Still Have

• Investigating image pull failures (Docker Hub rate limits before ECR migration)
• Manually bumping Liquibase image versions
• NO_PROXY config updates for new services

Runbook vs Playbook

Runbook: Step-by-step guide for a specific known failure. "If RabbitMQ disk alarm fires, do X, Y, Z."
Playbook: Broader tactical guide for a category of incidents. "How we handle database failures."

Policy Evaluation Order — How AWS Decides

flowchart LR REQ["API Request"] --> DENY1["Explicit Deny?\n(any policy)"] DENY1 -->|"Yes"| DENIED["DENIED"] DENY1 -->|"No"| SCP["SCP allows?\n(Org policy)"] SCP -->|"No"| DENIED SCP -->|"Yes"| BOUND["Permission Boundary\nallows?"] BOUND -->|"No"| DENIED BOUND -->|"Yes"| RBAC["Session Policy\nallows?"] RBAC -->|"No"| DENIED RBAC -->|"Yes"| IDENTITY["Identity Policy\nallows?"] IDENTITY -->|"No"| RESOURCE["Resource Policy\nallows?"] RESOURCE -->|"Yes"| ALLOWED["ALLOWED"] RESOURCE -->|"No"| DENIED IDENTITY -->|"Yes"| ALLOWED

IRSA — IAM Roles for Service Accounts (Full Flow)

StepWhat Happens
1. OIDC setupEKS cluster has an OIDC provider URL. AWS IAM trusts this URL as an identity federation source.
2. ServiceAccount annotationK8s SA annotated with eks.amazonaws.com/role-arn. EKS mutating webhook injects env vars + volume mount into the pod.
3. Token projectionK8s projects a short-lived SA token (1h) at /var/run/secrets/eks.amazonaws.com/serviceaccount/token
4. AssumeRoleWithWebIdentityAWS SDK reads the token. Calls STS AssumeRoleWithWebIdentity with the token. STS validates token with EKS OIDC provider.
5. Temp credentialsSTS returns temporary credentials scoped to the IAM role. SDK caches and auto-renews them.
6. API callsPod calls S3/DynamoDB/Secrets Manager using the temp credentials. Resource policy + identity policy both checked.

Common IAM Mistakes & Fixes

Mistake: Using IAM Users for services. Users have long-term access keys — if leaked, they're valid until manually rotated. Fix: always use IAM Roles for services (assumed via IRSA, EC2 instance profile, Lambda execution role).
Mistake: Wildcard resource + wildcard action. Action: * on Resource: * = root-equivalent. Even if attached to a role, permission boundary doesn't help if it allows *. Fix: always scope to specific actions and specific resource ARNs.
Mistake: Not understanding permission boundaries. A boundary of AdministratorAccess + identity policy of ReadOnlyAccess = ReadOnly. The boundary is the ceiling, identity policy is the floor. Boundary can never grant — only restrict.
Mistake: Forgetting the Trust Policy. A role's identity policy controls what the role can DO. The Trust Policy controls WHO can ASSUME the role. Both must be correct for cross-account or IRSA to work.
