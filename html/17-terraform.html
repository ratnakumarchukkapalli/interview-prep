HCL Fundamentals

Terraform uses HCL (HashiCorp Configuration Language) — a declarative language where you describe the desired state of infrastructure. Unlike CDK where you write Python/TypeScript imperatively, HCL is declarative: you say what you want, Terraform figures out how to get there.

terraform block
Specifies required providers, Terraform version constraints, and backend configuration. Always the first block in main.tf
provider block
Configures the AWS provider — region, assume_role for cross-account, default tags. One per AWS account/region combination
resource block
Declares infrastructure. resource "aws_vpc" "main" — type then local name. Referenced as aws_vpc.main.id
data block
Read-only. Fetches existing resources or values. data "aws_ami" "latest_eks" — doesn't create, just queries
# main.tf — complete minimal example
terraform {
  required_version = ">= 1.6.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # allows 5.x, blocks 6.x
    }
  }
  backend "s3" {
    bucket         = "westpac-terraform-state"
    key            = "prod/eks/terraform.tfstate"
    region         = "ap-southeast-2"
    dynamodb_table = "terraform-state-lock"
    encrypt        = true
    kms_key_id     = "arn:aws:kms:ap-southeast-2:ACCOUNT:key/KEY-ID"
  }
}

provider "aws" {
  region = var.aws_region
  assume_role {
    role_arn = "arn:aws:iam::${var.account_id}:role/TerraformDeployRole"
  }
  default_tags {
    tags = {
      Environment = var.environment
      ManagedBy   = "terraform"
      CostCentre  = var.cost_centre
      Owner       = var.team
    }
  }
}

# Variable declaration
variable "aws_region" {
  type    = string
  default = "ap-southeast-2"
}

# Resource
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true
  tags = { Name = "${var.environment}-vpc" }
}

# Output
output "vpc_id" {
  description = "The VPC ID"
  value       = aws_vpc.main.id
}

Core Workflow

terraform init      # download providers, initialise backend
terraform fmt       # auto-format HCL (always run before commit)
terraform validate  # syntax + schema check (no AWS API calls)
terraform plan      # show what WILL change — review carefully
terraform apply     # make it so (prompts for confirmation)
terraform destroy   # tear everything down — never run in prod without review

State Management

Terraform state is the source of truth that maps your HCL configuration to real infrastructure. Without state, Terraform cannot know what already exists. State is what makes plan work — it compares desired (HCL) vs actual (state) vs real (AWS API).

Never store state locally in a team or production environment. Local state means only your machine knows what Terraform manages. Another engineer running apply will try to create duplicate resources, causing conflicts or failures. 
# Remote state backend — S3 + DynamoDB locking (banking standard)
# S3 bucket for state storage
resource "aws_s3_bucket" "terraform_state" {
  bucket = "westpac-terraform-state-${var.account_id}"
}

resource "aws_s3_bucket_versioning" "state" {
  bucket = aws_s3_bucket.terraform_state.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "state" {
  bucket = aws_s3_bucket.terraform_state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.terraform_state.arn
    }
  }
}

# DynamoDB table for state locking
resource "aws_dynamodb_table" "terraform_lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  attribute {
    name = "LockID"
    type = "S"
  }
}
State Locking
DynamoDB prevents two engineers from running apply simultaneously. One gets the lock, the other waits. Critical in team environments
terraform state mv
Rename a resource in state without destroying it. Essential when refactoring module structure without downtime
terraform import
Bring an existing AWS resource under Terraform management. Must write the HCL first, then import the real resource ID into state
terraform state rm
Remove a resource from state without destroying it. Use when handing off a resource to another team's Terraform workspace
# Common state operations
terraform state list                        # list all resources in state
terraform state show aws_vpc.main           # show details of one resource
terraform state mv aws_subnet.old aws_subnet.new   # rename in state
terraform state rm aws_instance.legacy      # remove from state (keeps real resource)
terraform import aws_vpc.main vpc-abc123    # import existing VPC into state

# Force-unlock a stuck lock (use with caution — verify no one else is running apply)
terraform force-unlock LOCK-ID

Module Architecture

Modules are the primary way to organise and reuse Terraform code. Think of them like CDK Constructs — reusable building blocks with inputs (variables) and outputs.

# Standard module file structure
modules/
└── eks-cluster/
    ├── main.tf        ← resources
    ├── variables.tf   ← input variables (the module's "API")
    ├── outputs.tf     ← what the module exposes to callers
    └── versions.tf    ← provider requirements

# Root module calls child modules
module "eks" {
  source  = "./modules/eks-cluster"
  # or from Terraform Registry:
  # source  = "terraform-aws-modules/eks/aws"
  # version = "~> 20.0"

  cluster_name    = "westpac-prod-eks"
  cluster_version = "1.29"
  vpc_id          = module.vpc.vpc_id
  subnet_ids      = module.vpc.private_subnet_ids
}

# Reference module output
resource "aws_route53_record" "eks_api" {
  zone_id = data.aws_route53_zone.internal.id
  name    = "eks-api.prod.westpac.internal"
  type    = "CNAME"
  records = [module.eks.cluster_endpoint]
}
CDK analogy: A Terraform module is like a CDK Construct. The root module (where you call all child modules) is like your CDK Stack. variables.tf is like the props interface. outputs.tf is like expose properties on a Construct. The key difference: modules don't have inheritance — you compose by passing outputs as inputs.

Workspaces & Environments

StrategyHowBest ForRisks
Terraform Workspacesterraform workspace new prodSimple differences between envs (just var values)Shared state backend, easy to apply to wrong workspace
Directory-per-envenvironments/prod/, environments/staging/Completely separate state files per envCode duplication unless modules are well-structured

For banking (Westpac): Do NOT use workspaces for prod vs non-prod separation. A mistake like terraform workspace select prod followed by terraform destroy is catastrophic. Use separate AWS accounts with separate state files. The extra complexity is worth the safety.

Terraform in CI/CD

graph LR PR["Pull Request\nOpened"] --> FMT["terraform fmt\n--check"] FMT --> VAL["terraform validate"] VAL --> PLAN["terraform plan\n(save to file)"] PLAN --> COMMENT["Post plan output\nto PR comment"] COMMENT --> REVIEW["Human Review\n+ Approval"] REVIEW --> APPLY["terraform apply\n(on merge to main)"] APPLY --> NOTIFY["Notify Slack\n+ CloudTrail log"] 
# .github/workflows/terraform.yml — OIDC (no static AWS credentials)
name: Terraform

on:
  pull_request:
    paths: ['terraform/**']
  push:
    branches: [main]
    paths: ['terraform/**']

permissions:
  id-token: write    # required for OIDC
  contents: read
  pull-requests: write

jobs:
  terraform:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: terraform/environments/prod

    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials (OIDC — no static keys)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::ACCOUNT:role/GitHubActionsRole
          aws-region: ap-southeast-2

      - uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "1.6.6"

      - name: Terraform Init
        run: terraform init

      - name: Terraform Format Check
        run: terraform fmt -check -recursive

      - name: Terraform Validate
        run: terraform validate

      - name: Terraform Plan
        id: plan
        run: terraform plan -out=tfplan -no-color
        continue-on-error: true

      - name: Post plan to PR
        uses: actions/github-script@v7
        if: github.event_name == 'pull_request'
        with:
          script: |
            const output = `#### Terraform Plan 📋
            \`\`\`\n${{ steps.plan.outputs.stdout }}\n\`\`\``;
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            });

      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply tfplan
OIDC instead of static AWS keys: GitHub Actions can assume an IAM role using OIDC — no AWS_ACCESS_KEY_ID or AWS_SECRET_ACCESS_KEY stored in GitHub Secrets. The IAM role trust policy restricts which repo and branch can assume it. This is the secure, modern approach for CI/CD.

CDK vs Terraform — Bridging Your Experience

You have deep CDK (Python) experience from Thomson Reuters. Here's how to map that to Terraform and articulate both clearly in an interview.

ConceptAWS CDK (Python)Terraform
LanguagePython, TypeScript, Java (imperative)HCL (declarative)
StateCloudFormation manages stateTerraform manages its own .tfstate
OutputGenerates CloudFormation templatesCalls AWS APIs directly via provider
Reuse unitConstruct (L1/L2/L3)Module
Stack = ?CDK Stack → CloudFormation StackRoot module + backend config
Logic/loopsFull Python — for loops, classes, inheritanceHCL for_each, count, dynamic blocks
Multi-cloudAWS onlyAWS, Azure, GCP, Kubernetes, anything
Testingcdk-assertions, pytestTerratest (Go), checkov, OPA/Conftest
Drift detectionCloudFormation drift detectionterraform plan shows drift
Destroy protectionRemovalPolicy.RETAINlifecycle { prevent_destroy = true }
Interview framing: "I've built 11 CDK stacks at Thomson Reuters covering EKS, RDS, networking, and observability. CDK gives me the power of Python for complex logic. Terraform's strength is its ecosystem, multi-cloud support, and the fact that most enterprises — including most Australian banks — standardise on it. I'm comfortable in both and choose based on the requirement: CDK for AWS-native greenfield, Terraform when the organisation already has Terraform patterns or needs multi-cloud."

Advanced HCL

# for_each — create multiple resources from a map
variable "subnets" {
  default = {
    "app-a"  = { cidr = "10.0.10.0/24", az = "ap-southeast-2a" }
    "app-b"  = { cidr = "10.0.11.0/24", az = "ap-southeast-2b" }
    "data-a" = { cidr = "10.0.20.0/24", az = "ap-southeast-2a" }
  }
}

resource "aws_subnet" "this" {
  for_each          = var.subnets
  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value.cidr
  availability_zone = each.value.az
  tags = { Name = each.key }
}

# Reference: aws_subnet.this["app-a"].id

# dynamic block — conditionally create nested blocks
resource "aws_security_group" "eks_nodes" {
  name   = "eks-nodes"
  vpc_id = aws_vpc.main.id

  dynamic "ingress" {
    for_each = var.extra_ingress_rules
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = "tcp"
      cidr_blocks = ingress.value.cidrs
    }
  }
}

# lifecycle — protect critical resources
resource "aws_rds_cluster" "postgresql" {
  # ...
  lifecycle {
    prevent_destroy       = true   # block terraform destroy
    ignore_changes        = [master_password]  # don't revert manual rotation
    create_before_destroy = true   # zero-downtime replacement
  }
}

# locals — computed values, avoid repetition
locals {
  cluster_name = "${var.environment}-${var.project}-eks"
  common_tags = merge(var.default_tags, {
    ClusterName = local.cluster_name
  })
}

Terraform for EKS

# Using the official terraform-aws-modules/eks module
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 20.0"

  cluster_name    = local.cluster_name
  cluster_version = "1.29"

  cluster_endpoint_public_access  = false   # private cluster for banking
  cluster_endpoint_private_access = true

  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_subnets

  # IRSA — enable OIDC provider for service accounts
  enable_irsa = true

  # Managed node groups
  eks_managed_node_groups = {
    general = {
      instance_types = ["m5.2xlarge"]
      min_size       = 3
      max_size       = 10
      desired_size   = 3

      labels = {
        role = "general"
      }

      taints = []
    }
  }

  # aws-auth ConfigMap — who can access the cluster
  manage_aws_auth_configmap = true
  aws_auth_roles = [
    {
      rolearn  = "arn:aws:iam::ACCOUNT:role/AdminRole"
      username = "admin"
      groups   = ["system:masters"]
    },
    {
      rolearn  = "arn:aws:iam::ACCOUNT:role/DevRole"
      username = "developer"
      groups   = ["developers"]
    }
  ]
}

# IRSA for a service — give a pod Secrets Manager access
module "confirmation_app_irsa" {
  source  = "terraform-aws-modules/iam/aws//modules/iam-role-for-service-accounts-eks"
  version = "~> 5.0"

  role_name = "confirmation-app"
  oidc_providers = {
    main = {
      provider_arn               = module.eks.oidc_provider_arn
      namespace_service_accounts = ["confirmation:confirmation-app"]
    }
  }
}

resource "aws_iam_role_policy" "confirmation_secrets" {
  name = "secrets-access"
  role = module.confirmation_app_irsa.iam_role_name

  policy = jsonencode({
    Statement = [{
      Effect   = "Allow"
      Action   = ["secretsmanager:GetSecretValue"]
      Resource = "arn:aws:secretsmanager:ap-southeast-2:ACCOUNT:secret:prod/confirmation/*"
    }]
  })
}

Banking-Specific Patterns

Mandatory Tagging
APRA requires cost attribution. Use provider default_tags + policy-as-code (Sentinel/OPA) to block deploys missing required tags
Naming Convention
{env}-{team}-{service}-{resource} e.g. prod-platform-confirmation-eks. Enforced via naming module or variable validation
Drift Detection
Run terraform plan in CI on a schedule (not just on PRs). Alert if plan shows unexpected changes — someone made a manual change
prevent_destroy
Set on all stateful resources: RDS, ElastiCache, S3, DynamoDB. Blocks terraform destroy from taking out production databases
# variable validation — enforce naming convention
variable "environment" {
  type = string
  validation {
    condition     = contains(["prod", "staging", "dev", "sandbox"], var.environment)
    error_message = "environment must be one of: prod, staging, dev, sandbox"
  }
}

# Mandatory tags via provider default_tags (apply to all resources)
provider "aws" {
  region = var.aws_region
  default_tags {
    tags = {
      Environment  = var.environment
      CostCentre   = var.cost_centre
      Owner        = var.team_email
      ManagedBy    = "terraform"
      DataClass    = var.data_classification   # "Public", "Internal", "Confidential"
      APRAScope    = var.apra_scope            # "in-scope" or "out-of-scope"
    }
  }
}

# OPA/Conftest policy — block resources without required tags
# (Run in CI via: conftest test plan.json --policy policies/)
package terraform.analysis

deny[msg] {
  resource := input.resource_changes[_]
  resource.type == "aws_instance"
  not resource.change.after.tags.CostCentre
  msg := sprintf("Resource %v missing required tag: CostCentre", [resource.address])
}

Interview Scenarios

Q: You have CDK experience but we use Terraform. How quickly can you be productive?

Very quickly — the concepts transfer directly. I understand infrastructure as code deeply: resource graphs, dependency management, state, modules, and how IaC interacts with CI/CD pipelines. Those mental models are language-agnostic.

Specifically, CDK Constructs map to Terraform modules, CDK Stacks map to root modules with backend config, CDK outputs map to Terraform outputs, and the imperative vs declarative difference is something I've already worked with since CDK generates CloudFormation and I've debugged both layers.

I'd be writing meaningful Terraform within days, not weeks. I'd start with the terraform-aws-modules registry for common patterns (EKS, VPC, IAM), focus on understanding your team's module structure, and run a few plans against non-prod before touching anything production-critical.

Q: How do you manage Terraform state across 10 AWS accounts for different environments?

Each account gets its own state file, stored in S3 in a centralised "Terraform state" account. The bucket key structure encodes account and workload: accounts/{account-id}/{workload}/terraform.tfstate.

The CI/CD pipeline authenticates to the target account via OIDC (GitHub Actions assumes an account-specific IAM role). The backend is configured with partial config — the key is provided at runtime so the same code can target different accounts: terraform init -backend-config="key=prod/eks/terraform.tfstate".

State access is locked down with S3 bucket policies: only the Terraform execution role for that account can read/write that account's state. Cross-account state reading is allowed for shared outputs (e.g., the VPC ID from the networking account) using a separate read-only role.

Q: A developer ran terraform apply manually on production without going through CI/CD. How do you prevent this and what do you do now?

Prevention: The Terraform IAM role in production should only be assumable by the CI/CD service account (GitHub Actions OIDC), not by individual developer IAM users. Use the role's trust policy condition to restrict to the specific GitHub repository and branch: "token.actions.githubusercontent.com:ref": "refs/heads/main". Developers get read-only access — they can plan but not apply.

Right now: Run terraform plan to see what changed vs what state shows. Check CloudTrail for the full API call log — what resources were created/modified/destroyed, and when. If production is in an unexpected state, the priority is to get it back to the known-good state from the last successful CI/CD apply, not to manually fix it further.

Process: Raise it in the team. Not to blame, but to understand why they bypassed CI/CD — maybe there was a legitimate emergency. Document the incident, update the runbook, and tighten the IAM trust policy so it cannot happen again.

Q: What is terraform state drift and how do you detect and handle it?

Drift is when the real infrastructure diverges from what Terraform's state believes it to be. This happens when someone makes a manual change in the AWS console, or an AWS service itself changes a resource attribute (e.g., auto-scaling adjusts instance count).

Detection: Run terraform plan — it calls AWS APIs to get the current real state and compares against the .tfstate file. Any difference is shown as a change. For proactive detection, run plan on a schedule in CI and alert on any non-empty plan output.

Handling: Depends on the drift. If someone manually added a security group rule that is legitimate, add it to the HCL and run apply. If it's an unwanted change, apply restores to the desired state. For attributes that legitimately drift (e.g., managed passwords), use lifecycle { ignore_changes = [password] }.

In banking, any drift should trigger an investigation — who made the manual change and why? It's a process failure, not just a technical one.

Q: How do you import existing AWS infrastructure that wasn't created with Terraform?

This is a common scenario when joining a team that managed infra manually or with CloudFormation. The process:

1. Write the HCL first. Write the resource block in Terraform that matches the existing resource's configuration. Get it as close as possible to the real resource.

2. Import. terraform import aws_vpc.main vpc-abc123. This creates a state entry linking the HCL resource to the real AWS resource ID.

3. Plan. Run terraform plan immediately. It will show differences between your HCL and the real resource. These must be zeroed out — either update your HCL to match reality, or use ignore_changes for attributes you don't want to manage.

4. Apply. Only apply when plan shows "No changes". Otherwise you might accidentally modify the production resource.

Terraform 1.5+ introduced terraform import blocks in HCL, and terraform plan -generate-config-out which auto-generates HCL from the real resource — a massive time saver for bulk imports.
