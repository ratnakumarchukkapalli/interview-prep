VPC Architecture for Banking

Key Points to Remember:
  • Public subnet = faces internet. ALB and NAT Gateway only. EKS nodes NEVER go here.
  • Private subnet = EKS nodes live here. Can go OUT via NAT, nothing comes IN directly.
  • Data subnet = no internet route at all. RDS, ElastiCache only reachable from private subnet.
  • Route table = rules that tell traffic where to go. Each subnet has its own. That's why subnets behave differently.
  • Analogy: Public = building lobby, Private = inner offices, Data = vault. NAT Gateway is the receptionist who can go outside on staff behalf.

Banks don't use simple 2-tier VPCs. Westpac's AWS architecture follows a 3-tier subnet model across 3 Availability Zones with strict isolation between public-facing, application, and data layers.

APRA consideration: Data subnets must have no internet route. This is not optional in a banking environment — APRA CPS 234 requires that customer data stores are isolated from internet-facing systems. Auditors will check route tables.

Transit Gateway

Key Points to Remember:
  • VPC Peering = point-to-point, non-transitive. A cannot reach C via B. Scale is n(n-1)/2 connections.
  • TGW = central hub router. Every VPC connects once. Transitive routing works.
  • TGW Route Tables = control which VPCs talk to which. Prod and non-prod on separate route tables = isolation.
  • Connects on-premises via Direct Connect or VPN — VPC Peering cannot do this.
  • Analogy: Peering = direct roads between every building. TGW = central bus station. Adding new building = one connection, done.

Transit Gateway (TGW) is a regional hub-and-spoke router. Every VPC and on-premises network connects to it once, and TGW handles routing between all of them. It replaces a mesh of VPC peering connections.

TGW Route Tables — Segmentation

TGW has its own route tables separate from VPC route tables. This lets you control which VPCs can talk to which — critical for separating prod from non-prod.

# TGW Route Table: "prod-rt"
# Only Prod VPC and Shared Services can communicate — NonProd is excluded

│  Destination     │  Target                               │
├──────────────────┼───────────────────────────────────────┤
│  10.0.0.0/16     │  prod-vpc-attachment                  │

Common mistake: Attaching all VPCs to one TGW route table. That means prod and non-prod can talk — a security and compliance violation. Always use separate route tables per security domain.

VPC Endpoints & PrivateLink

Key Points to Remember:
  • Gateway Endpoint = adds an entry to your route table. Free. Only S3 and DynamoDB. No ENI, no IP.
  • Interface Endpoint = creates a private IP (ENI) inside your subnet. For all other AWS services (ECR, Secrets Manager, STS, CloudWatch).
  • Without endpoints: Pod → NAT → Internet → ECR. With endpoints: Pod → private IP → ECR. Never leaves AWS.
  • For no-internet EKS cluster you need: ecr.api, ecr.dkr, s3, sts, secretsmanager, logs, monitoring endpoints.
  • Why banking needs this: APRA requires sensitive data never traverses public internet. Endpoints enforce this at network level.

Banking workloads must not send sensitive data over the internet — even to AWS services. VPC Endpoints keep all traffic on the AWS private network.

Gateway Endpoints
Free. Only for S3 and DynamoDB. Adds an entry to your VPC route table. Traffic stays on AWS backbone, no ENI in your subnet
Interface Endpoints
For all other AWS services (Secrets Manager, ECR, STS, CloudWatch, etc.). Creates an ENI in your subnet with a private IP. Costs ~$0.01/hr per AZ
PrivateLink
The underlying technology for Interface Endpoints. Also lets you expose your own services privately across accounts or to customers
Endpoint Policies
Resource-based policies on the endpoint. Restrict which principals and resources can be accessed — e.g., only allow access to your org's S3 buckets

Security Groups vs Network ACLs

Key Points to Remember:
  • Security Group = instance level, stateful. Return traffic is automatic. Allow rules only. Primary control.
  • NACL = subnet level, stateless. Must explicitly allow both directions. Has Deny rules. Backup control.
  • Always reference Security Group IDs, not IPs — EKS nodes scale constantly, IPs change. SG ID stays the same.
  • 3-tier SG pattern: ALB SG → EKS Node SG → RDS SG. Each layer only accepts from the layer above it.
  • Use NACLs to Deny specific IPs during incidents — Security Groups can't deny, only allow.
FeatureSecurity GroupsNetwork ACLs
LevelInstance/ENI levelSubnet level
StatefulnessStateful — return traffic auto-allowedStateless — must explicitly allow both directions
RulesAllow only (no deny)Allow and Deny, processed in order
DefaultDeny all inbound, allow all outboundAllow all (default NACL)
Use casePrimary security control per resourceSubnet-level blast radius containment
PerformanceTracked connections (slight overhead)Packet-level, no state tracking
Banking pattern: Use Security Groups as primary control (granular, per-service). Use NACLs as a coarse backup — e.g., explicitly deny the RFC 1918 ranges you should never receive from the internet, or block a specific IP range during an incident without touching Security Groups.

AWS Direct Connect & VPN

Key Points to Remember:
  • Direct Connect = dedicated private fibre. Never touches public internet. Consistent latency. 1/10/100 Gbps.
  • VPN = encrypted but goes over internet. Unpredictable latency. Max 1.25 Gbps. Cheaper and faster to set up.
  • Always run both: DX as primary, VPN as backup. BGP prefers DX (shorter AS path). VPN activates automatically on DX failure.
  • BGP = routing protocol that exchanges network routes between on-prem and AWS. Handles automatic failover.
  • DX Gateway connects one DX to multiple VPCs across multiple regions — critical for DR.

Westpac has on-premises data centres (legacy banking systems, mainframes, Swift connectivity). Direct Connect provides a dedicated private connection from on-prem to AWS — not over the internet.

Direct Connect Gateway
Connects one DX connection to multiple VPCs across multiple regions. Essential for DR — one physical circuit, multi-region reach
BGP Routing
DX uses BGP to exchange routes. On-prem advertises its CIDRs, AWS advertises VPC CIDRs. More specific routes win over less specific
Redundancy
Always provision two DX connections from different providers/facilities. If one fibre cut takes down your only DX, you're in trouble
Site-to-Site VPN as Backup
Configure VPN as failover. DX preferred via BGP (more specific route). VPN activates automatically if DX goes down
DX vs VPN: VPN is encrypted but goes over the internet. DX is private (not encrypted by default) but never touches the public internet. For banking, DX + MACsec encryption at Layer 2 is the gold standard. VPN alone is acceptable for non-production.

EKS Networking — VPC CNI

Key Points to Remember:
  • VPC CNI = every pod gets a real VPC IP. Not an overlay IP. Pods are first-class VPC citizens.
  • Why it matters: Security Groups, VPC Endpoints, route tables all work on pods directly. Overlay IPs break all of this.
  • IP exhaustion is real: 10 nodes × 30 pods = 300 IPs. A /24 only has 251 usable. Use /21 or larger for EKS subnets.
  • Fix for exhaustion: Add secondary CIDR (100.64.0.0/16) — 65,536 IPs for pods. Zero downtime.
  • Instance type matters: m5.xlarge = max 59 pods. m5.4xlarge = max 234 pods. Check ENI limits per instance type.

EKS uses the AWS VPC CNI plugin by default. Unlike other CNIs, every pod gets a real VPC IP address — not an overlay network IP. This means pods are first-class VPC citizens but it also means IP address exhaustion is a real risk.

Route 53 & Hybrid DNS

Key Points to Remember:
  • Private Hosted Zone = internal DNS names (*.westpac.internal) visible only inside the VPC.
  • Inbound Resolver Endpoint = on-prem DNS forwards queries here → Route 53 resolves AWS names for on-prem machines.
  • Outbound Resolver Endpoint = AWS forwards queries to on-prem DNS → pods can resolve mainframe hostnames.
  • Split-horizon DNS = same domain resolves differently inside vs outside VPC. Avoids hairpin routing — internal traffic stays internal.
  • Analogy: Inbound = reception desk accepting calls from outside. Outbound = staff calling the external phone book.

In a hybrid banking environment, DNS resolution must work seamlessly between AWS and on-premises. Route 53 Resolver endpoints bridge the two worlds.

AWS Network Firewall & Centralised Egress

Key Points to Remember:
  • Security Groups/NACLs block ports and IPs — not domain names. Network Firewall adds Layer 7 (domain-level) control.
  • Allowlist over blocklist: Allow only *.amazonaws.com, *.datadog.com. Block everything else by default. More secure than trying to block all bad domains.
  • Centralised egress pattern: All VPC internet traffic → TGW → Inspection VPC → Network Firewall → NAT → Internet. One policy for all accounts.
  • Why banks need it: Compromised pod may try to reach attacker C2 server on port 443. SG allows 443 (legitimate). NFW blocks unknown domain.
  • Suricata rules = the rule engine inside Network Firewall. Same technology used by enterprise IDS/IPS systems.

A bank cannot allow unrestricted internet egress from workloads. AWS Network Firewall provides stateful Layer 7 inspection of all traffic passing through it.

# Only allow specific domains for outbound HTTPS
[domain_allowlist]

  tls.sni; content:"amazonaws.com"; endswith;

  msg:"Block unallowed HTTPS egress"; sid:9999999;
)
Centralised egress pattern: Route all VPC internet-bound traffic through TGW to a dedicated Inspection VPC that hosts Network Firewall and NAT Gateways. This gives you one place to enforce egress policy across all accounts — critical for APRA compliance and cost control.

Interview Scenarios

Q: Why would Westpac use Transit Gateway instead of VPC Peering?

VPC Peering is point-to-point and non-transitive. For Westpac with production, non-production, shared-services, security, and sandbox accounts — that's already 10 peering connections for 5 VPCs (n(n-1)/2). Add Direct Connect for on-premises, and peering simply cannot model that topology.

Transit Gateway solves this with a hub-and-spoke model. Each VPC connects once. TGW handles all routing. Most importantly, TGW lets you enforce isolation via route tables — prod and non-prod VPCs attach to different TGW route tables so they can't accidentally communicate. And you can route all traffic through a centralised Network Firewall in an inspection VPC — something that's architecturally very difficult with peering.

Q: An EKS cluster is running out of IP addresses. How do you fix it without downtime?

First, confirm the root cause: kubectl describe nodes shows allocatable IPs. Check if subnets are exhausted via the AWS console.

Short-term fix: Add new subnets in different AZs and register them with the EKS node group. New nodes will launch into the new subnets with fresh IP pools.

Proper fix: Enable secondary CIDR on the VPC using the 100.64.0.0/10 RFC 6598 range (carrier-grade NAT — commonly used for this pattern). Create subnets in this range. Set AWS_VPC_K8S_CNI_CUSTOM_NETWORK_CFG=true on the VPC CNI daemonset and configure ENIConfig CRDs to assign pods to the secondary CIDR subnets while nodes stay in the primary CIDR.

This is zero-downtime — existing pods keep their IPs, new pods use the secondary range. I've seen teams avoid this and end up with a 2 AM crisis where nodes can't schedule new pods. Size your subnets generously from day one.

Q: How do you ensure EKS pods in a banking environment never send traffic to the internet?

Three layers of control:

1. Subnet routing: Place EKS nodes in private subnets with no NAT Gateway route. The route table has no 0.0.0.0/0 entry. Physically impossible to reach the internet.

2. VPC Endpoints: All AWS service calls (ECR, Secrets Manager, STS, CloudWatch) go via Interface Endpoints — private IPs in the VPC. No internet required for AWS APIs.

3. Network Firewall: Even if a subnet had a NAT route, all egress goes through Network Firewall first with a domain allowlist. Only pre-approved domains (Datadog, AWS) can exit. Unknown domains are dropped and alerted on.

I'd also add Kubernetes NetworkPolicies and Istio egress controls as application-level enforcement, but the network layer is the primary control for compliance.

Q: Westpac's Direct Connect goes down. What happens to production?

If designed correctly, nothing catastrophic happens immediately. DX failure should be a planned-for event, not a disaster.

Failover path: We run a Site-to-Site VPN in parallel with DX. The DX route is preferred via BGP (more specific AS path). When DX fails, BGP withdraws the DX routes, and traffic automatically fails over to VPN within the BGP convergence time (typically 30-90 seconds).

Degradation: VPN is limited to 1.25 Gbps vs 10 Gbps DX. High-bandwidth workloads (batch jobs, large file transfers) should be throttled or paused during failover.

On-prem dependent services: Any services that strictly require on-premises (mainframe calls, SWIFT) will be degraded until DX recovers. These should have circuit-breaker patterns to fail fast rather than timeout.

Recovery: AWS provides status at status.aws.amazon.com. DX issues are usually at the colocation facility level. Engage AWS Premium Support immediately for P1.

Q: How do you design DNS for a hybrid banking environment where on-prem needs to resolve AWS service endpoints?

This requires Route 53 Resolver endpoints in both directions.

For on-prem to resolve AWS names: Deploy a Route 53 Resolver Inbound Endpoint — this creates ENIs in the VPC with private IPs (e.g. 10.0.10.50). Configure the corporate DNS server to forward queries for *.westpac.internal and *.ap-southeast-2.amazonaws.com to these IPs. Now on-prem machines can resolve RDS endpoints, private ALBs, and EKS service DNS.

For AWS to resolve on-prem names: Deploy a Route 53 Resolver Outbound Endpoint. Create a Resolver Rule forwarding *.corp.westpac to the on-premises DNS server IP. Now pods in EKS can resolve mainframe hostnames and legacy application names.

Split-horizon: api.westpac.com.au resolves to the internal ALB IP within the VPC (via Private Hosted Zone) and the public ALB IP externally. This avoids hairpin routing and keeps internal traffic private.
